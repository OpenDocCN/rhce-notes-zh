# Ansible 角色：第8章：Ansible 角色详解 🧩

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_1.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_3.png)

在本节课中，我们将要学习 Ansible 角色的概念、结构、创建方法以及如何使用它们来组织和复用 Playbook 代码。角色是 Ansible 中用于打包和复用任务、变量、文件、模板和处理程序的强大工具，能极大简化复杂自动化任务的管理。

## 概述

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_5.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_7.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_9.png)

上一节我们介绍了 Ansible 的模板和变量控制。本节中我们来看看如何将多个任务、变量和文件通过既定的结构进行打包封装，这就是 Ansible 角色。角色可以理解为“打包工具箱”，它允许我们将功能模块化，在调用时只需引用角色名称即可，使 Playbook 更加简洁和可维护。

```
├── handlers
```

每个目录都有其特定用途，我们无需手动创建这些目录，可以使用 `ansible-galaxy` 命令自动初始化生成。

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_56.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_58.png)

以下是各个子目录的用途说明：

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_64.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_66.png)

*   **`defaults/main.yml`**：定义角色的**默认变量**，优先级较低，可在 Playbook 中被覆盖。
*   **`files/`**：存放需要复制到受管主机的**静态文件**。在任务中引用时只需写文件名。
*   **`handlers/main.yml`**：定义角色的**处理程序**，由任务中的 `notify` 触发。
*   **`meta/main.yml`**：包含角色的**元数据**，如作者、许可证、支持的平台以及所依赖的其他角色。
*   **`tasks/main.yml`**：这是角色的核心，包含所有**任务定义**。可以将任务分片写在多个文件中，然后在 `main.yml` 中编排顺序。
*   **`templates/`**：存放 **Jinja2 模板文件**。
*   **`tests/`**：包含用于**测试**角色的清单和 Playbook 文件。
*   **`vars/main.yml`**：定义角色的**变量**，优先级高于 `defaults/` 中定义的变量，通常用于内部用途。
*   **`README.md`**：可编写角色的说明文档，如使用要求、作者信息和示例。

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_89.png)

在 Playbook 中，只需使用 `roles` 关键字调用角色即可：

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_90.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_91.png)


现在，我们将之前编写过的部署 Apache HTTPD 服务的 Playbook 重构为一个角色。这个过程就是将一个大任务拆分成多个小部分，并放入角色对应的目录中。

以下是构建 `httpd` 角色的步骤分解：

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_197.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_198.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_200.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_207.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_209.png)

1.  **编写分任务文件**：在 `tasks/` 目录下创建多个 YAML 文件，每个文件负责一个独立任务（如安装软件包、复制文件、配置防火墙、启动服务）。
2.  **整合主任务文件**：在 `tasks/main.yml` 中使用 `import_tasks` 或 `include_tasks` 模块按顺序导入所有分任务文件，就像组装零件一样。
3.  **定义变量**：在 `vars/main.yml` 或 `defaults/main.yml` 中定义角色所需的变量，如软件包名 `package_name: httpd` 和监听端口 `http_port: 80`。
4.  **准备静态文件**：将网站首页文件（如 `index.html`）放入 `files/` 目录。
5.  **准备模板文件**：将 HTTPD 的配置文件模板（如 `httpd.conf.j2`）放入 `templates/` 目录，并在模板中使用变量（如 `{{ http_port }}`）。
6.  **定义处理程序**：在 `handlers/main.yml` 中定义重启服务的处理程序。
7.  **编写调用角色的 Playbook**：最后，创建一个简单的 Playbook，其内容主要就是调用这个 `httpd` 角色。

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_210.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_212.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_219.png)

完成以上步骤后，运行这个 Playbook，Ansible 会自动执行角色中定义的所有任务。这种方式使得代码结构清晰，易于维护和复用。

## 使用系统角色和 Ansible Galaxy

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_230.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_232.png)

除了自定义角色，我们还可以使用现成的角色。

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_234.png)

*   **系统角色**：从 RHEL 7.4 开始，系统提供了一些预装的角色包（如 `rhel-system-roles`），用于管理时间同步、内核参数调优等常见系统任务。可以通过系统包管理器安装后直接使用。
*   **Ansible Galaxy**：这是一个由社区维护的公共角色库（网址：galaxy.ansible.com）。我们可以使用 `ansible-galaxy` 命令搜索、下载和使用他人编写好的高质量角色。

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_236.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_238.png)

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_240.png)

例如，搜索 Redis 相关的角色：
```bash
ansible-galaxy search 'redis' --platforms EL
```

![](img/d23aa5cb9e2ee9a5f1f7bf0d6456b4b1_243.png)

下载一个特定的角色：
```bash
ansible-galaxy install geerlingguy.redis
```

## 总结

本节课中我们一起学习了 Ansible 角色的核心知识。我们了解了角色的标准化目录结构及其每个部分的作用，掌握了如何创建角色框架并将复杂的 Playbook 任务模块化到角色中。我们还学习了如何在 Playbook 中简洁地调用角色，以及如何利用系统角色和 Ansible Galaxy 社区资源来复用现有成果，从而极大地提升自动化脚本的可维护性、可读性和复用效率。
