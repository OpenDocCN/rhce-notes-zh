## 6.1：Linux存储结构概述 🌳

上一节我们介绍了课程安排，本节中我们来看看Linux的存储结构。与Windows系统使用盘符（如C盘、D盘）不同，Linux系统没有盘符的概念。它的一切都从根目录（`/`）开始，采用一种倒树状结构。不同的硬件设备或存储分区会被“挂载”到树形结构的不同目录下，用户通过访问这些目录来使用设备。

Linux系统遵循一个名为**FHS（Filesystem Hierarchy Standard，文件系统层次结构标准）**的协议来规范每个顶级目录的作用。这有助于用户和管理员快速定位文件。

以下是Linux系统中一些核心目录及其作用的介绍：

*   **`/root`**：系统管理员（root用户）的**家目录**。用于存放管理员个人的配置文件和数据。
*   **`/bin`**：存放**所有用户都可以执行的命令文件**（二进制可执行文件）。例如 `ls`, `cp`, `cat` 等基础命令。
*   **`/sbin`**：存放**只有系统管理员（root）可以执行的命令文件**。例如 `reboot`（重启）、`fdisk`（分区）等系统管理命令。
*   **`/boot`**：存放**与系统启动相关的文件**，如Linux内核、引导加载程序（GRUB）的配置文件。此目录非常重要，一般不建议用户修改。
*   **`/dev`**：存放**设备文件**。在Linux中，“一切皆文件”，硬件设备（如硬盘、鼠标、键盘）也被映射成文件存放在此目录下供系统访问。例如，第一块SATA硬盘可能被识别为 `/dev/sda`。
*   **`/etc`**：存放**系统和应用程序的配置文件**。绝大多数服务的配置文件都位于此目录或其子目录下。例如，网络配置、用户账户信息等。
*   **`/home`**：存放**普通用户的家目录**。每个用户在此目录下都有一个以自己用户名命名的子文件夹，用于存放个人数据。
*   **`/var`**：存放**经常发生变动的数据**，如系统日志（`/var/log`）、数据库文件、网站内容等。
*   **`/lib`**：存放**系统开机时所需的函数库文件**，以及`/bin`和`/sbin`目录下命令所需的共享库。普通用户通常无需直接操作此目录。
*   **`/usr`**：存放**用户安装的应用程序和文件**，类似于Windows的“Program Files”目录。其子目录如`/usr/local`常用于存放手动编译安装的软件。
*   **`/media`**：系统提供的**可移动设备挂载点**。当插入U盘、光盘时，系统常会自动将其挂载到此目录下的子目录中。
*   **`/mnt`**：**临时挂载目录**。管理员常手动将设备（如额外的硬盘分区）临时挂载到此目录下。
*   **`/tmp`**：存放**临时文件**。所有用户都可在此目录创建文件，但默认设置了特殊权限（Sticky Bit），用户只能删除自己的文件。
*   **`/proc`**：一个**虚拟文件系统**。它不占用磁盘空间，里面的文件映射了系统的运行状态（如CPU信息、内存使用情况）。直接读取这些文件可以获取实时系统信息。

> **注意**：FHS是一种规范而非强制标准。在实际工作中，管理员可以根据需要灵活调整，但遵循此规范能使系统管理更清晰、高效。

上一节我们了解了目录结构，本节中我们来看看如何解读设备文件的名称。在 `/dev` 目录下，设备文件的命名并非随意，而是由 **udev 服务**根据一套规则生成的，目的是让用户能通过文件名直观判断设备类型。

![](img/7d72eda5e6cf031bb98541e5ef2a5993_25.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_27.png)

对于最常见的硬盘设备，其文件名遵循以下模式：
`/dev/sd[a-z][1-9]`

![](img/7d72eda5e6cf031bb98541e5ef2a5993_29.png)

让我们用公式来分解这个模式：

![](img/7d72eda5e6cf031bb98541e5ef2a5993_31.png)

**`/dev/sd[a-z][1-9]`**

![](img/7d72eda5e6cf031bb98541e5ef2a5993_33.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_35.png)

*   **`/dev/`**：表示该文件位于设备目录下。
*   **`sd`**：表示这是一个 **SCSI、SATA、USB 或 NVMe** 接口的存储设备。这是目前最常见的硬盘接口类型。
    *   历史上还有 `hd`（IDE接口）等，现已少见。
*   **`[a-z]`**：**设备识别号**，代表系统识别到的第几块同类设备。例如：
    *   `a` 表示第一块被识别的SATA/SCSI硬盘 (`sda`)。
    *   `b` 表示第二块 (`sdb`)。
    *   以此类推。**注意**：这个顺序由内核识别顺序决定，可能与物理插槽顺序有关，但非直接对应。
*   **`[1-9]`**：**分区编号**，代表这块硬盘上的第几个分区。
    *   **1-4**：**主分区或扩展分区**的编号。一块硬盘最多只能有4个主分区，或3个主分区+1个扩展分区。
    *   **5 及以上**：**逻辑分区**的编号。逻辑分区存在于扩展分区之内。**因此，编号为5的分区 (`sda5`) 一定是一个逻辑分区。**

![](img/7d72eda5e6cf031bb98541e5ef2a5993_37.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_39.png)

**示例解析**：
*   `/dev/sda`：第一块SATA硬盘的整体设备。
*   `/dev/sda1`：第一块SATA硬盘上的第一个**主分区或扩展分区**。
*   `/dev/sdb5`：第二块SATA硬盘上的第一个**逻辑分区**。

## 6.3：文件系统与虚拟文件系统（VFS） 💾

上一节我们学会了识别设备，本节中我们来看看数据是如何在设备上被组织和访问的。一块硬盘或分区必须经过**格式化**，创建**文件系统**后才能存储文件。常见的Linux文件系统有 **ext3, ext4, XFS** 等。红帽RHEL 7/8 默认使用 **XFS** 文件系统，它支持超大容量（可达18EB）且性能良好。

在Linux中，一个文件在磁盘上由两部分组成：

1.  **inode（索引节点）**：保存文件的**元数据**，如权限、所有者、大小、时间戳以及指向数据块的指针。**不包含文件名**。
2.  **block（数据块）**：实际保存文件**内容**的数据块。大文件会占用多个block，并通过指针链接。

那么，用户发出的简单命令（如 `touch file.txt`）是如何适配底层不同的文件系统（ext4, XFS, FAT32等）的呢？这要归功于 **VFS（Virtual File System，虚拟文件系统）**。

**VFS** 是内核中的一个软件层，它为用户程序提供了一个统一的、抽象的文件操作接口。其核心目的是：**让用户和应用程序无需关心底层具体的文件系统类型**。

**工作流程简化**：
1.  用户执行文件操作命令（如 `cat /home/user/file.txt`）。
2.  VFS 接收请求，并解析路径。
3.  VFS 调用底层具体文件系统（如XFS）的驱动程序来执行实际的读写操作。
4.  底层文件系统操作硬件，获取数据。
5.  数据通过VFS层返回给用户程序。

## 6.4：硬盘管理三部曲：分区、格式化、挂载 ⚙️

上一节我们理解了文件系统的抽象层，本节中我们来看看使用一块新硬盘的具体步骤。要让一块新硬盘在Linux中可用，需要经过三个核心步骤：

1.  **分区**
2.  **格式化**
3.  **挂载**

### 分区

分区就像给一个大房子划分房间。使用 `fdisk` 或 `gdisk` 命令对硬盘进行分区。

**常用 `fdisk` 命令交互操作**：
```bash
fdisk /dev/sdb # 对第二块硬盘进行操作
```
进入交互界面后，常用键：
*   `n`：新建分区
*   `p`：打印分区表
*   `d`：删除分区
*   `w`：保存并退出
*   `q`：不保存退出

### 格式化

![](img/7d72eda5e6cf031bb98541e5ef2a5993_45.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_47.png)

格式化就是给每个“房间”（分区）安装一套管理规则（文件系统），以便有序地存放“家具”（文件）。使用 `mkfs` 命令。

**格式化分区为 XFS 文件系统**：
```bash
mkfs.xfs /dev/sdb1
```
**格式化分区为 ext4 文件系统**：
```bash
mkfs.ext4 /dev/sdb1
```

### 挂载

挂载是将一个设备文件（如 `/dev/sdb1`）与一个目录（称为**挂载点**）进行关联的动作。关联后，用户访问该目录就等于访问设备内的数据。

![](img/7d72eda5e6cf031bb98541e5ef2a5993_49.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_51.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_53.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_55.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_57.png)

**临时挂载**（重启后失效）：
```bash
mount /dev/sdb1 /mnt/mydata
```

![](img/7d72eda5e6cf031bb98541e5ef2a5993_59.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_61.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_63.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_65.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_67.png)

**卸载设备**：
```bash
umount /dev/sdb1
# 或
umount /mnt/mydata
```

![](img/7d72eda5e6cf031bb98541e5ef2a5993_69.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_71.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_73.png)

**永久挂载**：需要编辑 `/etc/fstab` 配置文件，使挂载在系统重启后依然有效。
文件格式如下：
```
设备文件 挂载点 文件系统类型 挂载参数 备份标记 自检顺序
```
**示例**：
```
/dev/sdb1 /mnt/mydata xfs defaults 0 0
```
编辑保存后，可使用 `mount -a` 命令重新加载所有 `fstab` 中的配置。

理论是基础，实践出真知。现在，让我们通过一个完整的实战演练来巩固所学知识。我们将模拟为虚拟机添加一块新硬盘，并完成分区、格式化、挂载的全过程。

![](img/7d72eda5e6cf031bb98541e5ef2a5993_87.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_89.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_91.png)

**实验目标**：将一块20GB的新硬盘分区，格式化并挂载到 `/mnt/mydata` 目录。

![](img/7d72eda5e6cf031bb98541e5ef2a5993_93.png)

![](img/7d72eda5e6cf031bb98541e5ef2a5993_95.png)

**操作步骤**：

1.  **添加虚拟硬盘**：关闭虚拟机，在VMware设置中添加一块20GB的SCSI新硬盘，然后启动系统。
2.  **识别新硬盘**：以root用户登录，使用 `lsblk` 或 `fdisk -l` 命令查看新硬盘（例如 `/dev/sdb`）。
3.  **分区**：
    ```bash
    fdisk /dev/sdb
    ```
    *   输入 `n` 创建新分区。
    *   选择分区类型 `p`（主分区）。
    *   分区号默认为 `1`。
    *   起始扇区直接回车（默认）。
    *   结束扇区输入 `+10G`，创建一个10GB的分区。
    *   输入 `w` 保存并退出。
4.  **格式化**：将新分区格式化为XFS文件系统。
    ```bash
    mkfs.xfs /dev/sdb1
    ```
5.  **临时挂载**：
    ```bash
    mkdir /mnt/mydata # 创建挂载点目录
    mount /dev/sdb1 /mnt/mydata
    ```
6.  **验证挂载**：使用 `df -h` 命令查看，应能看到 `/dev/sdb1` 已挂载到 `/mnt/mydata`。
7.  **永久挂载**：
    ```bash
    vim /etc/fstab
    ```
    在文件末尾添加一行：
    ```
    /dev/sdb1 /mnt/mydata xfs defaults 0 0
    ```
    保存退出。执行 `mount -a` 测试配置是否正确（无报错即成功）。
8.  **最终测试**：重启系统，再次使用 `df -h` 命令验证挂载是否依然有效。
