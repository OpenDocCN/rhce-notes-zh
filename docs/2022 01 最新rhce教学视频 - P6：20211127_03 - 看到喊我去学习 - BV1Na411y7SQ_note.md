在本节课中，我们将要学习 Ansible 中的两个核心概念：**变量** 和 **事实**。变量允许我们定义动态值，而事实则是 Ansible 从被管理主机自动收集的系统信息。掌握它们对于编写灵活、可复用的 Playbook 至关重要。



![](img/234ab68eaa62f841b00d14fcc4995167_3.png)

### 变量：定义与使用

![](img/234ab68eaa62f841b00d14fcc4995167_5.png)

变量用于存储可在 Playbook 中重复使用的值。它们让 Playbook 变得动态，能够适应不同的主机或环境。



![](img/234ab68eaa62f841b00d14fcc4995167_6.png)

![](img/234ab68eaa62f841b00d14fcc4995167_7.png)

上一节我们介绍了 Playbook 的基本结构，本节中我们来看看如何定义和使用变量。



![](img/234ab68eaa62f841b00d14fcc4995167_8.png)

![](img/234ab68eaa62f841b00d14fcc4995167_9.png)

变量可以在多个地方定义，例如在 Playbook 内部、独立的变量文件中，或通过命令行传入。



![](img/234ab68eaa62f841b00d14fcc4995167_10.png)

以下是定义变量的几种常见方式：

![](img/234ab68eaa62f841b00d14fcc4995167_11.png)

![](img/234ab68eaa62f841b00d14fcc4995167_13.png)

*   **在 Playbook 的 `vars` 部分定义：**
    ```yaml
    - hosts: webservers
      vars:
        http_port: 80
        max_clients: 200
      tasks:
        - name: Ensure Apache is at the latest version
          yum:
            name: httpd
            state: latest
    ```

![](img/234ab68eaa62f841b00d14fcc4995167_14.png)

*   **在独立的变量文件中定义（如 `vars.yml`），并通过 `vars_files` 引入：**
    ```yaml
    # playbook.yml
    - hosts: all
      vars_files:
        - vars.yml
      tasks:
        - name: Print variable
          debug:
            msg: "The value is {{ my_variable }}"
    ```
    ```yaml
    # vars.yml
    my_variable: "Hello from vars file"
    ```

![](img/234ab68eaa62f841b00d14fcc4995167_15.png)

![](img/234ab68eaa62f841b00d14fcc4995167_16.png)

*   **通过命令行传递变量：**
    ```bash
    ansible-playbook playbook.yml --extra-vars "http_port=8080"
    ```



![](img/234ab68eaa62f841b00d14fcc4995167_17.png)

![](img/234ab68eaa62f841b00d14fcc4995167_18.png)

要使用变量，你需要将其用双花括号 `{{ }}` 括起来。Ansible 会在执行任务前将变量替换为其实际值。



![](img/234ab68eaa62f841b00d14fcc4995167_19.png)

### 事实：自动收集的系统信息

![](img/234ab68eaa62f841b00d14fcc4995167_20.png)

![](img/234ab68eaa62f841b00d14fcc4995167_21.png)

事实是 Ansible 在 Playbook 执行开始时，自动从目标主机收集的系统和环境信息。这些信息以变量的形式提供，可以直接在 Playbook 中使用。



![](img/234ab68eaa62f841b00d14fcc4995167_23.png)

了解了用户自定义的变量后，我们来看看 Ansible 自动为我们准备的变量——事实。



![](img/234ab68eaa62f841b00d14fcc4995167_24.png)

默认情况下，Ansible 会为每个被管理主机收集事实。你可以使用 `setup` 模块来查看所有收集到的事实。



![](img/234ab68eaa62f841b00d14fcc4995167_26.png)

以下是如何查看和使用事实：

![](img/234ab68eaa62f841b00d14fcc4995167_28.png)

*   **手动收集并查看所有事实：**
    ```bash
    ansible all -m setup
    ```
    这条命令会输出一个包含大量系统信息（如 IP 地址、主机名、磁盘空间、操作系统类型等）的 JSON 结构。

![](img/234ab68eaa62f841b00d14fcc4995167_30.png)

*   **在 Playbook 中使用事实：**
    ```yaml
    - hosts: all
      tasks:
        - name: Print OS family
          debug:
            msg: "This host's OS family is {{ ansible_os_family }}"
        - name: Print default IPv4 address
          debug:
            msg: "The IP address is {{ ansible_default_ipv4.address }}"
    ```



![](img/234ab68eaa62f841b00d14fcc4995167_32.png)

![](img/234ab68eaa62f841b00d14fcc4995167_34.png)

有时为了提升执行速度，你可能希望禁用事实收集。这可以通过在 Playbook 中设置 `gather_facts: no` 来实现。



![](img/234ab68eaa62f841b00d14fcc4995167_35.png)

### 变量的优先级

![](img/234ab68eaa62f841b00d14fcc4995167_37.png)

当同一个变量名在多个位置被定义时，Ansible 会按照既定的优先级顺序来决定使用哪个值。了解这个顺序对于调试和设计 Playbook 非常重要。



![](img/234ab68eaa62f841b00d14fcc4995167_38.png)

现在我们已经知道变量可以从多个来源定义，那么当变量名冲突时，哪个值会生效呢？这就涉及到变量的优先级。



![](img/234ab68eaa62f841b00d14fcc4995167_40.png)

![](img/234ab68eaa62f841b00d14fcc4995167_41.png)

以下是 Ansible 变量优先级从低到高的部分关键顺序（越靠后优先级越高）：

![](img/234ab68eaa62f841b00d14fcc4995167_43.png)

1.  角色默认值（在 `roles/xxx/defaults/main.yml` 中定义）
2.  Playbook 中 `vars` 部分定义的变量
3.  通过 `vars_files` 引入的变量文件
4.  主机或主机组清单变量（在 `inventory` 文件中定义）
5.  通过 `--extra-vars` 命令行传递的变量



![](img/234ab68eaa62f841b00d14fcc4995167_44.png)

这意味着，通过命令行传递的变量拥有最高的优先级，会覆盖其他来源的定义。



![](img/234ab68eaa62f841b00d14fcc4995167_46.png)

![](img/234ab68eaa62f841b00d14fcc4995167_47.png)

### 注册变量：捕获任务输出

![](img/234ab68eaa62f841b00d14fcc4995167_48.png)

![](img/234ab68eaa62f841b00d14fcc4995167_49.png)

除了预定义和收集的变量，你还可以将某个任务的执行结果（如命令的输出、模块的返回值）保存到一个自定义变量中，这个变量称为“注册变量”。这在后续的条件判断、循环或调试中非常有用。



![](img/234ab68eaa62f841b00d14fcc4995167_50.png)

![](img/234ab68eaa62f841b00d14fcc4995167_51.png)

让我们学习如何捕获一个任务的输出，并将其用于后续的任务。



![](img/234ab68eaa62f841b00d14fcc4995167_52.png)

你可以使用 `register` 关键字来保存任务的输出。



![](img/234ab68eaa62f841b00d14fcc4995167_54.png)

![](img/234ab68eaa62f841b00d14fcc4995167_56.png)

以下是一个使用注册变量的例子：

![](img/234ab68eaa62f841b00d14fcc4995167_57.png)

![](img/234ab68eaa62f841b00d14fcc4995167_58.png)

```yaml
- hosts: all
  tasks:
    - name: Get the current working directory
      command: pwd
      register: pwd_result

    - name: Debug the registered variable
      debug:
        var: pwd_result

    - name: Use the stdout from the registered variable
      debug:
        msg: "The working directory is {{ pwd_result.stdout }}"
```

![](img/234ab68eaa62f841b00d14fcc4995167_59.png)

在这个例子中，`pwd` 命令的输出被保存在 `pwd_result` 变量中。这个变量是一个字典，其中 `stdout` 键就包含了命令的标准输出内容。



![](img/234ab68eaa62f841b00d14fcc4995167_60.png)

![](img/234ab68eaa62f841b00d14fcc4995167_62.png)

### 总结

![](img/234ab68eaa62f841b00d14fcc4995167_64.png)

本节课中我们一起学习了 Ansible 中变量与事实的核心知识。我们了解了如何定义和使用自定义变量，包括在 Playbook 内部、外部文件以及命令行中定义。我们探讨了 Ansible 自动收集的主机事实及其用法。我们还明确了当变量来源冲突时的优先级规则。最后，我们学会了如何使用 `register` 关键字来捕获任务的输出，并将其注册为变量以供后续使用。这些概念是构建复杂、自适应自动化流程的基石。
