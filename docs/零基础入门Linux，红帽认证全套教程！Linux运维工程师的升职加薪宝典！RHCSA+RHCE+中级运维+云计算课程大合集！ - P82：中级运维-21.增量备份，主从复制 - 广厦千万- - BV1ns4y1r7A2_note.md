# Linux运维课程：P82：中级运维-21. 增量备份，主从复制

## 概述 📖

在本节课中，我们将学习MySQL数据库的“一主多从”架构及其优化方案。我们将重点探讨当从库数量增多时，主库读写压力过大的问题，并引入“中继库”的概念来分担压力。核心在于理解如何使用“黑洞（Blackhole）”存储引擎构建一个不存储数据、仅用于日志传递的中继节点，从而实现更稳定、可扩展的主从复制架构。

---

## 主从复制架构的压力问题 ⚠️

上一节我们介绍了基础的主从复制原理。本节中我们来看看当从库数量增多时可能遇到的问题。

![](img/81542a522b226d25b3b2a6d20835b20b_1.png)

![](img/81542a522b226d25b3b2a6d20835b20b_3.png)

“一主多从”架构存在一个问题：从库越多，主库的读写压力就越大。因为主库每次修改数据，都需要同步到每一个从库。每增加一个从库，主库就需要多向一个目标发送数据。发送的数据越多，主库压力越大，服务器就越容易出故障。

![](img/81542a522b226d25b3b2a6d20835b20b_5.png)

![](img/81542a522b226d25b3b2a6d20835b20b_6.png)

服务器宕机的原因，除了断电等极端情况，最常见的就是资源（如CPU、内存、硬盘）使用率过高。任何一个硬件资源满载都可能导致问题。因此，不能让单台服务器的压力过大。

![](img/81542a522b226d25b3b2a6d20835b20b_8.png)

---

## 解决方案：引入中继库 🔄

![](img/81542a522b226d25b3b2a6d20835b20b_10.png)

![](img/81542a522b226d25b3b2a6d20835b20b_12.png)

![](img/81542a522b226d25b3b2a6d20835b20b_14.png)

为了解决主库压力过大的问题，我们引入一个中继库（Relay Server）作为中间层。

![](img/81542a522b226d25b3b2a6d20835b20b_16.png)

![](img/81542a522b226d25b3b2a6d20835b20b_18.png)

中继库的作用是过渡和传递数据。在新的架构中，主库只连接中继库这一台“从库”。对于主库而言，这仍然是“一主一从”的结构，读写压力自然减小，服务器就不容易宕机。

![](img/81542a522b226d25b3b2a6d20835b20b_20.png)

以下是这种架构的核心思路：
*   **主库** -> **中继库** -> **多个从库**

中继库的核心特点是使用**黑洞（Blackhole）**存储引擎。黑洞引擎的特点是**不存储任何数据**，它只接收并传递二进制日志，自身不执行实际的写入操作。这能显著降低中继库自身的硬盘I/O和CPU压力，避免其成为新的瓶颈。

**强调**：在配置好主从复制后，从库（包括中继库）**不应该再直接写入数据**。因为从库写入的数据主库无法感知，且容易与主库同步下来的数据产生冲突。

## 中继库的配置步骤 ⚙️

接下来，我们具体讲解如何配置带有中继库的主从复制架构。假设我们有三台服务器：主库（Master）、中继库（Relay）、从库（Slave）。

### 1. 修改配置文件

首先，我们需要修改三台服务器的MySQL配置文件（通常是 `my.cnf` 或 `my.ini`），并重启数据库服务。

以下是各服务器需要修改的核心参数：

**主库配置**：
只需确保已开启二进制日志。
```ini
[mysqld]
server-id = 1
log-bin = mysql-bin
```

**中继库配置**：
这是配置的关键，需要开启二进制日志、设置级联复制，并将默认存储引擎改为黑洞。
```ini
[mysqld]
server-id = 2
log-bin = mysql-bin
# 启用级联复制，允许中继库将其接收到的更新写入自己的二进制日志
log-slave-updates = ON
# 将默认存储引擎设置为黑洞
default-storage-engine = BLACKHOLE
```

**从库配置**：
只需确保server-id唯一。
```ini
[mysqld]
server-id = 3
```

### 2. 创建复制用户并授权

![](img/81542a522b226d25b3b2a6d20835b20b_22.png)

在主库上，创建用于复制的用户，并授予复制权限。这里我们使用一个简化的授权命令，授予所有权限（包含复制权限）给来自任意IP的root用户（生产环境建议使用更严格的授权）。
```sql
GRANT ALL PRIVILEGES ON *.* TO 'root'@'%' IDENTIFIED BY 'your_password';
FLUSH PRIVILEGES;
```
在中继库上，也需要执行类似的授权操作，以便从库能够连接它。

![](img/81542a522b226d25b3b2a6d20835b20b_24.png)

![](img/81542a522b226d25b3b2a6d20835b20b_26.png)

### 3. 配置复制链路

![](img/81542a522b226d25b3b2a6d20835b20b_28.png)

![](img/81542a522b226d25b3b2a6d20835b20b_30.png)

![](img/81542a522b226d25b3b2a6d20835b20b_32.png)

![](img/81542a522b226d25b3b2a6d20835b20b_34.png)

![](img/81542a522b226d25b3b2a6d20835b20b_36.png)

**第一步：配置中继库作为主库的从库**
在中继库上执行，将其设置为指向主库的从库。
```sql
CHANGE MASTER TO
MASTER_HOST='主库IP',
MASTER_USER='root',
MASTER_PASSWORD='主库密码',
MASTER_LOG_FILE='主库的二进制日志文件名',
MASTER_LOG_POS=主库的二进制日志位置;
START SLAVE;
```
`MASTER_LOG_FILE` 和 `MASTER_LOG_POS` 的值需要在主库上执行 `SHOW MASTER STATUS;` 命令获取。

![](img/81542a522b226d25b3b2a6d20835b20b_38.png)

![](img/81542a522b226d25b3b2a6d20835b20b_40.png)

**第二步：配置从库作为中继库的从库**
在从库上执行，将其设置为指向中继库的从库。
```sql
CHANGE MASTER TO
MASTER_HOST='中继库IP',
MASTER_USER='root',
MASTER_PASSWORD='中继库密码',
MASTER_LOG_FILE='中继库的二进制日志文件名',
MASTER_LOG_POS=中继库的二进制日志位置;
START SLAVE;
```
`MASTER_LOG_FILE` 和 `MASTER_LOG_POS` 的值需要在中继库上执行 `SHOW MASTER STATUS;` 命令获取。

![](img/81542a522b226d25b3b2a6d20835b20b_42.png)

### 4. 验证与测试

![](img/81542a522b226d25b3b2a6d20835b20b_44.png)

![](img/81542a522b226d25b3b2a6d20835b20b_46.png)

配置完成后，分别在**中继库**和**从库**上执行 `SHOW SLAVE STATUS\G`，查看 `Slave_IO_Running` 和 `Slave_SQL_Running` 是否都为 `Yes`。

![](img/81542a522b226d25b3b2a6d20835b20b_48.png)

在主库上插入测试数据，然后在**从库**上查询，确认数据已成功同步。在**中继库**上查询黑洞引擎的表，数据应为空，这验证了黑洞引擎只传递日志而不存储数据。

![](img/81542a522b226d25b3b2a6d20835b20b_50.png)

## 关于双主复制 💡

“双主复制”是指两个数据库互为主从，任何一方写入的数据都会同步到另一方。其配置本质上是为两个数据库互相做一次主从复制。

![](img/81542a522b226d25b3b2a6d20835b20b_56.png)

虽然双主架构允许双向读写，但其缺点也很明显：
1.  **写入冲突风险**：如果双方同时修改同一行数据，容易导致同步失败或数据不一致。
2.  **性能压力**：双方都需要承担读写和同步的双重压力。

![](img/81542a522b226d25b3b2a6d20835b20b_58.png)

因此，在实际生产环境中，**“主从复制”配合“读写分离”**是更常用、更稳定的方案。主库负责写，多个从库负责读，既能分摊压力，又能避免数据冲突。

![](img/81542a522b226d25b3b2a6d20835b20b_60.png)

## 总结 🎯

本节课我们一起学习了MySQL高级主从复制架构。我们首先分析了传统“一主多从”模式可能给主库带来的巨大压力，然后引入了**中继库**作为解决方案。通过将中继库的存储引擎设置为**黑洞（BLACKHOLE）**，我们构建了一个只负责传递二进制日志、自身不存储数据的中间层，有效降低了主库和中间节点的负载。最后，我们简要对比了“双主复制”，并指出“主从+读写分离”是更优的实践选择。

![](img/81542a522b226d25b3b2a6d20835b20b_62.png)

理解并掌握这种带中继的架构，对于构建高性能、高可用的数据库集群至关重要。
