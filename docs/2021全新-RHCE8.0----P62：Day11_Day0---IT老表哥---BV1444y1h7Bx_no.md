## 第一部分：Ansible 安装与临时命令 ⚙️

上一节我们介绍了 Ansible 的安装与初始化配置。本节中，我们来看看 Ansible 的临时命令。

Ansible 临时命令主要用于快速执行单个任务，其基本结构如下：

**`ansible <host-pattern> -m <module_name> -a <module_arguments>`**

以下是临时命令的组成部分解析：
*   **`<host-pattern>`**：指定在资产清单中定义的主机或主机组。
*   **`-m <module_name>`**：指定要使用的模块。
*   **`-a <module_arguments>`**：指定模块对应的参数。

![](img/60dda1e6faacf7440aa9fcee05727275_13.png)

如果需要指定不同的资产清单文件，可以使用 `-i` 参数，但通常已在配置文件中定义，故不常用。

![](img/60dda1e6faacf7440aa9fcee05727275_15.png)

### 幂等性

![](img/60dda1e6faacf7440aa9fcee05727275_17.png)

Ansible 命令具有幂等性。这意味着控制节点向受管主机发出指令后，目标是使受管主机达到一个特定的状态。如果目标状态已经存在，则任务不会重复执行，直接返回 `ok` 而非 `changed`。

## 第二部分：常用管理模块回顾 🧰

![](img/60dda1e6faacf7440aa9fcee05727275_34.png)

上一节我们介绍了临时命令的结构。本节中，我们来回顾几个核心的管理模块。

![](img/60dda1e6faacf7440aa9fcee05727275_36.png)

![](img/60dda1e6faacf7440aa9fcee05727275_37.png)

![](img/60dda1e6faacf7440aa9fcee05727275_38.png)

![](img/60dda1e6faacf7440aa9fcee05727275_39.png)

Ansible 管理模块主要分为文件模块、软件包模块、服务模块和系统模块。

![](img/60dda1e6faacf7440aa9fcee05727275_41.png)

![](img/60dda1e6faacf7440aa9fcee05727275_42.png)

### 文件模块

![](img/60dda1e6faacf7440aa9fcee05727275_44.png)

![](img/60dda1e6faacf7440aa9fcee05727275_45.png)

以下是常用的文件操作模块：

![](img/60dda1e6faacf7440aa9fcee05727275_47.png)

![](img/60dda1e6faacf7440aa9fcee05727275_49.png)

**1. copy 模块**
用于将文件从控制节点复制到受管主机，或直接在受管主机上生成文件。
*   **`src`**：源文件路径（从控制节点复制时使用）。
*   **`content`**：文件内容（直接生成文件时使用）。
*   **`dest`**：目标文件路径（**必需参数**）。
*   其他常用参数：`mode`（权限）、`owner`（所有者）、`backup`（是否备份）等。

![](img/60dda1e6faacf7440aa9fcee05727275_50.png)

![](img/60dda1e6faacf7440aa9fcee05727275_51.png)

![](img/60dda1e6faacf7440aa9fcee05727275_52.png)

**2. file 模块**
用于在受管主机上创建目录、文件或链接。
*   **`path`**：文件或目录的路径。
*   **`state`**：状态，决定创建的类型。例如 `directory`（目录）、`touch`（文件）、`link`（软链接）。

![](img/60dda1e6faacf7440aa9fcee05727275_54.png)

![](img/60dda1e6faacf7440aa9fcee05727275_56.png)

**3. fetch 模块**
用于从受管主机拉取文件到控制节点。
*   **`src`**：受管主机上的源文件路径。
*   **`dest`**：控制节点上的目标保存路径。

![](img/60dda1e6faacf7440aa9fcee05727275_58.png)

![](img/60dda1e6faacf7440aa9fcee05727275_60.png)

![](img/60dda1e6faacf7440aa9fcee05727275_62.png)

**4. lineinfile 模块**
以行为单位，流式地修改文件内容。
*   **`path`**：要修改的远程文件路径。
*   **`regexp`**：用于定位行的正则表达式。
*   **`line`**：要替换或插入的整行内容。
*   **`state`**：状态，默认为 `present`（存在/修改），`absent` 为移除匹配行。
*   其他定位方式：`insertafter`（在匹配行后插入）、`insertbefore`（在匹配行前插入）。

### 软件包模块

在综合练习中，软件包管理是必备技能。

**1. yum_repository 模块**
用于配置 YUM 软件仓库。在红帽 8 系统中，通常需要配置两个仓库：`BaseOS`（基础系统包）和 `AppStream`（应用流包）。
其参数与 repo 文件类似，包括仓库地址（`baseurl`）、描述（`description`）、是否启用（`enabled`）、GPG 检查（`gpgcheck`）和 GPG 密钥地址（`gpgkey`）等。

**2. yum 模块**
用于管理软件包。
*   **`name`**：指定软件包或软件包组的名称（组名前加 `@`）。
*   **`state`**：指定操作状态，如 `present`（安装）、`absent`（移除）、`latest`（升级）。

### 系统模块

![](img/60dda1e6faacf7440aa9fcee05727275_64.png)

![](img/60dda1e6faacf7440aa9fcee05727275_66.png)

以下是常用的系统管理模块：

**1. user 模块**
管理用户，相当于 `useradd`、`userdel`、`usermod` 命令的集合。
常用参数：`name`（用户名）、`comment`（注释）、`group`（主组）、`groups`（附加组）、`state`、`shell`、`uid` 等。

**2. group 模块**
管理用户组，参数与 `user` 模块类似。

**3. service 模块**
管理系统服务的状态。
*   **`name`**：服务名。
*   **`state`**：服务运行状态，如 `started`、`stopped`、`restarted`。
*   **`enabled`**：是否开机自启。

**4. firewalld 模块**
管理防火墙规则，类似于 `firewall-cmd` 命令。
配置规则时，除了 `permanent`（永久生效）参数，务必加上 `immediate: yes` 参数以使规则立即生效，这相当于执行了 `firewall-cmd --reload`。

**5. get_url 模块**
用于从网络下载文件。

## 第三部分：Playbook 剧本编写 📜

![](img/60dda1e6faacf7440aa9fcee05727275_70.png)

![](img/60dda1e6faacf7440aa9fcee05727275_72.png)

上一节我们回顾了各种临时命令模块。本节中，我们来看看如何将这些任务组织成可重复执行的剧本（Playbook）。

剧本使用 YAML 语言编写，结构清晰。一个基本的剧本结构如下：

```yaml
---
- name: Playbook 的描述信息
  hosts: target_hosts
  tasks:
    - name: 第一个任务的描述
      module_name:
        parameter1: value1
        parameter2: value2

    - name: 第二个任务的描述
      another_module:
        parameter: value
```

![](img/60dda1e6faacf7440aa9fcee05727275_74.png)

以下是编写剧本的注意事项：
*   **起始**：三个连字符 `---` 表示一个 YAML 文档的开始。
*   **列表**：使用短横线加空格 `- ` 开头表示列表项（如 `tasks` 下的每个任务）。
*   **键值对**：使用冒号加空格 `: ` 表示键值对。
*   **缩进**：使用两个空格进行缩进，统一结构，不要使用 Tab 键。
*   **可读性**：任务之间可以用空行隔开，便于阅读和调试。

![](img/60dda1e6faacf7440aa9fcee05727275_76.png)

### 剧本的执行与检查

编写完剧本后，建议按以下步骤操作：
1.  **语法检查**：使用 `ansible-playbook --syntax-check <playbook.yml>` 命令检查 YAML 语法。
2.  **空运行（模拟）**：使用 `ansible-playbook -C <playbook.yml>` 命令模拟执行，查看可能发生的变化，但不实际执行。
3.  **正式执行**：使用 `ansible-playbook <playbook.yml>` 命令运行剧本。可以添加 `-v`、`-vv`、`-vvv` 等参数来增加输出信息的详细程度。

## 第四部分：变量与事实 🔑

![](img/60dda1e6faacf7440aa9fcee05727275_86.png)

上一节我们学习了剧本的基本结构。本节中，我们来看看如何在剧本中使用变量和事实来增加灵活性。

![](img/60dda1e6faacf7440aa9fcee05727275_88.png)

### 变量

![](img/60dda1e6faacf7440aa9fcee05727275_90.png)

![](img/60dda1e6faacf7440aa9fcee05727275_91.png)

变量用于存储可变化的值。

![](img/60dda1e6faacf7440aa9fcee05727275_93.png)

![](img/60dda1e6faacf7440aa9fcee05727275_95.png)

**变量命名规则**：可以包含字母、数字和下划线，但**不能以数字开头**，也**不建议以下划线开头**。避免使用点号`.`，因为它有特殊含义。

![](img/60dda1e6faacf7440aa9fcee05727275_97.png)

**变量定义位置**：
*   **在 Playbook 中定义**：使用 `vars:` 关键字。
*   **在资产清单中定义**：可以为特定主机或主机组定义变量。
*   **通过命令行传入**：使用 `-e "var_name=value"` 参数。

**变量引用**：使用双花括号 `{{ variable_name }}` 的形式引用变量，注意花括号内需保留空格。

**变量类型**：
*   **简单变量**：`variable_name: value`
*   **列表/数组**：`package_list: [‘httpd‘, ‘git‘, ‘vim‘]`
*   **字典**：`user_info: {name: ‘alice‘, uid: 1001, shell: ‘/bin/bash‘}`
*   **注册变量**：使用 `register` 关键字将一个任务的输出结果保存到变量中，供后续任务使用。这在配合条件判断时非常有用。

![](img/60dda1e6faacf7440aa9fcee05727275_99.png)

### 事实

事实是 Ansible 在任务执行前从受管主机自动收集的系统信息。

**启用事实收集**：在 Playbook 中，默认会收集事实。也可通过 `gather_facts: yes` 显式启用。

**使用事实**：可以通过 `ansible_facts[‘key‘]` 或 `ansible_facts.key` 的形式引用事实。例如，主机名是 `ansible_facts[‘hostname‘]`。
> 注意：如果变量名本身包含点，建议使用第一种括号形式，避免歧义。

**魔法变量**：这是一类特殊变量，不是通过 `gather_facts` 收集的，而是 Ansible 内部提供的。例如：
*   `groups`：包含所有主机组信息的字典。
*   `group_names`：当前主机所属的所有组名列表。
*   `inventory_hostname`：当前主机在资产清单中定义的主机名。
