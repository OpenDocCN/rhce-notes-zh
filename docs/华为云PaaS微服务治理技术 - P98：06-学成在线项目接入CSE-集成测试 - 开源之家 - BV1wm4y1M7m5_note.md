## 初始访问测试与问题定位

服务启动后，我们尝试通过前端界面访问系统功能。由于我们接入了网关，初始访问很可能会遇到错误，我们可以根据错误信息来定位和解决问题。

首先访问搜索功能，确认其是否可以正常工作。如果搜索功能正常，则说明搜索服务的基础连通性没有问题。

接着，我们尝试访问在线学习相关的功能。此时，通过浏览器开发者工具检查网络请求，会发现一个关键问题：学习服务的请求并未通过我们配置的网关，而是直接访问了后端服务的原始端口（例如 `40600`）。

![](img/427c454487f0c0f45b1ef61bc3b06e64_21.png)

**问题核心**：根据微服务架构的设计，所有外部请求都应通过统一的API网关 (`gateway-service`) 进入系统，由网关负责路由、认证和流量管理。当前的配置导致请求绕过了网关。

## 配置网关路由与公共服务

为了让请求正确通过网关，我们需要修改Nginx的代理配置。同时，由于当前测试的功能（如查看课程视频）属于无需登录即可访问的公共服务，我们必须在网关的过滤器中将这些服务配置为白名单，避免被登录校验拦截。

以下是具体的配置步骤：

1.  **配置网关公共服务白名单**：在网关服务的配置文件中，将 `learning-service`、`search-service`、`port view` 等服务名添加到无需认证的公共服务列表中。
    ```yaml
    # 示例配置 (具体格式取决于您的框架)
    public-services:
      - learning-service
      - search-service
      - port-view-service
    ```
    如果不配置，网关会要求用户登录，导致未登录用户无法获取数据。

2.  **修改Nginx配置**：编辑Nginx配置文件，将原本直接代理到具体微服务端口的规则，改为代理到网关服务。
    *   **找到原始配置**：定位到代理 `learning-service`、`search-service` 等服务的 `location` 块。
    *   **修改代理目标**：将 `proxy_pass` 指令的地址改为网关服务的地址（例如 `http://gateway-service:50201`）。
    *   **设置路由路径**：在代理路径中，使用网关的默认路由规则，通常格式为 `/{微服务名}/**`。例如，将学习服务的请求路径改为 `/learning-service`。

    ```nginx
    # 修改前示例：直接代理到学习服务
    location /api/learning/ {
        proxy_pass http://learning-service:40600;
    }

    # 修改后示例：通过网关代理
    location /api/learning/ {
        proxy_pass http://gateway-service:50201/learning-service;
    }
    ```
    按照此方式，依次修改搜索服务、门户服务等其他服务的Nginx配置。

3.  **重载Nginx配置**：修改完成后，执行命令使新配置生效。
    ```bash
    nginx -s reload
    ```

## 验证网关路由与可视化监控

完成配置修改并重载Nginx后，再次通过前端界面访问各项功能。此时页面应能正常显示数据。

![](img/427c454487f0c0f45b1ef61bc3b06e64_33.png)

![](img/427c454487f0c0f45b1ef61bc3b06e64_34.png)

为了确认请求确实通过了网关，我们可以通过CSE平台提供的监控功能进行验证：

1.  **网关调用列表**：在CSE控制台的网关管理页面，查看“调用服务列表”。只要请求通过了网关，被调用的微服务名称就会出现在这个列表中。
2.  **服务依赖拓扑图**：CSE平台提供了强大的服务治理可视化界面。在相关面板中，可以查看实时的服务间调用拓扑图。这张图会清晰地展示：用户请求首先到达 **网关**，然后由网关路由到 **学习服务**、**搜索服务** 等，并且服务之间也可能存在相互调用（例如学习服务调用门户服务）。这直观地呈现了微服务技术栈的动态调用关系。

![](img/427c454487f0c0f45b1ef61bc3b06e64_36.png)

这个图形化界面不仅展示了调用链路，还集成了流量监控、调用链追踪和多种服务治理策略（如熔断、限流）的配置入口，体现了云平台微服务治理的强大能力。
