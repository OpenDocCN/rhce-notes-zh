## 特殊权限概述

上一节我们介绍了Linux的基本权限。本节中，我们来看看四种特殊权限。前两种（SUID、SGID）主要作为了解，后两种（SBIT、ACL）则是后期常用的、必须掌握的内容。

![](img/d79ce77d44b4c45f97679c0c4692476d_23.png)

![](img/d79ce77d44b4c45f97679c0c4692476d_25.png)

### 1. SUID (Set User ID) 👑

SUID全称为Set User ID。它主要针对**可执行文件**。

**功能**：当一个可执行文件被设置了SUID权限后，任何普通用户在执行该文件时，会**临时拥有文件所有者（通常是root）的身份**。该权限仅在程序执行过程中有效，执行完毕后用户恢复原有身份。

![](img/d79ce77d44b4c45f97679c0c4692476d_27.png)

**为什么需要SUID？**
在系统中，许多关键命令（如`passwd`）的所有者是root。普通用户需要修改自己的密码，但密码最终保存在`/etc/shadow`文件中，而普通用户对此文件**没有写权限**。SUID机制使得普通用户在执行`passwd`命令时，临时获得root身份，从而能够修改`/etc/shadow`文件。

**查看示例**：
我们可以查看`/usr/bin/passwd`文件的权限。
```bash
ls -l /usr/bin/passwd
```
输出中，所有者的执行位是 `s`（而不是 `x`），这表示设置了SUID权限。文件背景色通常为红色作为提示。

![](img/d79ce77d44b4c45f97679c0c4692476d_29.png)

**设置与取消SUID**：
*   设置：`chmod u+s 文件名`
*   取消：`chmod u-s 文件名`

**重要提示**：SUID权限需谨慎设置。如果给`vim`命令设置SUID，普通用户将能以root身份编辑任何系统文件，可能导致严重安全问题。

![](img/d79ce77d44b4c45f97679c0c4692476d_31.png)

**小结**：SUID就像一道“圣旨”，让普通用户临时“变成”文件所有者（如root）去执行特定命令，完成后恢复原本身份。

### 2. SGID (Set Group ID) 👥

SGID全称为Set Group ID。它同样主要针对**可执行文件**。

![](img/d79ce77d44b4c45f97679c0c4692476d_45.png)

![](img/d79ce77d44b4c45f97679c0c4692476d_47.png)

**功能**：当一个可执行文件被设置了SGID权限后，任何普通用户在执行该文件时，会**临时拥有文件所属组的权限**。

**为什么需要SGID？**
以`locate`命令为例。它需要读取`/var/lib/mlocate/mlocate.db`数据库文件，但该文件对其他用户没有读(`r`)权限。`locate`命令本身被设置了SGID，其所属组为`slocate`。当普通用户执行`locate`时，便临时拥有了`slocate`组的身份，从而继承了该组对数据库文件的读权限。

![](img/d79ce77d44b4c45f97679c0c4692476d_49.png)

**查看示例**：
查看`locate`命令的文件权限。
```bash
ls -l /usr/bin/locate
```
输出中，所属组的执行位是 `s`，表示设置了SGID权限。文件背景色通常为黄色。

![](img/d79ce77d44b4c45f97679c0c4692476d_51.png)

**设置与取消SGID**：
*   设置：`chmod g+s 文件名`
*   取消：`chmod g-s 文件名`

![](img/d79ce77d44b4c45f97679c0c4692476d_53.png)

**小结**：SGID让执行文件的用户临时“加入”文件所属组，以继承该组的权限。此权限通常由系统预设，我们一般无需手动设置。

![](img/d79ce77d44b4c45f97679c0c4692476d_55.png)

### 3. SBIT (Sticky Bit) 🚫

![](img/d79ce77d44b4c45f97679c0c4692476d_59.png)

SBIT权限**只针对目录有效**。

![](img/d79ce77d44b4c45f97679c0c4692476d_61.png)

**功能**：当一个目录被设置了SBIT（粘滞位）权限后，**除了root用户和文件所有者本人**，即使其他用户对该目录有完整的写(`w`)权限，也只能删除或重命名**自己创建的文件**，而不能删除或重命名其他用户的文件。

![](img/d79ce77d44b4c45f97679c0c4692476d_63.png)

**为什么需要SBIT？**
常见的例子是系统的`/tmp`临时目录。所有用户都需要在其中创建临时文件。如果没有SBIT权限，用户A可以删除用户B的文件，造成混乱。设置了SBIT后，保证了用户文件的安全。

**查看示例**：
查看`/tmp`目录的权限。
```bash
ls -ld /tmp
```
输出中，其他用户的执行位是 `t`，表示设置了SBIT权限。目录背景色通常为绿色。

![](img/d79ce77d44b4c45f97679c0c4692476d_65.png)

![](img/d79ce77d44b4c45f97679c0c4692476d_67.png)

![](img/d79ce77d44b4c45f97679c0c4692476d_69.png)

**设置与取消SBIT**：
*   设置：`chmod o+t 目录名`
*   取消：`chmod o-t 目录名`

![](img/d79ce77d44b4c45f97679c0c4692476d_71.png)

**小结**：SBIT常用于需要多人协作写入、但又需防止互相干扰的共享目录，它确保了“各人自扫门前雪”。

### 4. ACL (Access Control List) 访问控制列表 📋

ACL的全称是Filesystem Access Control List（文件系统访问控制列表）。它提供了比传统“所有者-所属组-其他人”更精细的权限控制。

**功能**：可以为**单个用户**或**单个用户组**单独设置对某个文件或目录的访问权限，而不影响原有的权限模型。

![](img/d79ce77d44b4c45f97679c0c4692476d_73.png)

![](img/d79ce77d44b4c45f97679c0c4692476d_75.png)

![](img/d79ce77d44b4c45f97679c0c4692476d_77.png)

**为什么需要ACL？**
假设有一个`/devops`目录，属于`devops`组。你希望组员`tom`和`jerry`有完整权限，新来的实习生`xiaowei`只有查看和进入的权限，而其他部门的人`zhang`没有任何权限。传统的权限设置无法满足这种复杂需求，而ACL可以轻松实现。

**ACL相关命令**：
*   **设置ACL权限**：`setfacl -m u:用户名:权限 文件/目录名` 或 `setfacl -m g:组名:权限 文件/目录名`
    *   例如，给用户`xiaowei`对`/devops`目录添加读和执行权限：`setfacl -m u:xiaowei:rx /devops`
*   **查看ACL权限**：`getfacl 文件/目录名`
    *   输出中会显示额外的`user:`和`group:`条目。
*   **删除指定用户的ACL权限**：`setfacl -x u:用户名 文件/目录名`
*   **删除所有ACL权限**：`setfacl -b 文件/目录名`

**小结**：ACL实现了“量身定制”的权限分配，极大地增强了权限管理的灵活性，是实际工作中非常实用的功能。

---

## 总结

本节课中我们一起学习了Linux的四种特殊权限：
1.  **SUID**：针对可执行文件，让执行者临时拥有文件所有者的身份。
2.  **SGID**：针对可执行文件，让执行者临时拥有文件所属组的身份；也可针对目录，使在该目录下新建的文件自动继承目录的所属组。
3.  **SBIT**：针对目录，保护目录内的文件只能被其所有者和root删除。
4.  **ACL**：提供更精细的权限控制，可以为特定用户或组单独设置权限。

![](img/d79ce77d44b4c45f97679c0c4692476d_79.png)

![](img/d79ce77d44b4c45f97679c0c4692476d_81.png)

其中，**SBIT**和**ACL**在实际运维工作中应用广泛，需要重点掌握。理解并合理运用这些权限，是构建安全、可控的Linux系统环境的重要基础。
