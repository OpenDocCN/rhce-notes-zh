在本节课中，我们将要学习Linux系统中的四种特殊权限：SUID、SGID、SBIT和ACL。这些权限扩展了基本权限（rwx）的功能，用于满足更精细和特殊的安全控制需求。我们将逐一了解它们的作用、应用场景及设置方法。

![](img/63cf84b211bc769ae48687a7fd2d733f_9.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_11.png)

特殊权限主要分为两类：一类是针对可执行文件的（SUID、SGID），用于临时改变执行者的身份；另一类是针对目录的（SBIT、ACL），用于实现更灵活的访问控制。

## SUID（Set User ID）权限 🛡️

SUID权限主要应用于**可执行文件**。其核心功能是：当普通用户执行一个被设置了SUID权限的程序时，该程序会**临时拥有文件所有者（通常是root）的身份**执行。程序执行完毕后，用户恢复原有身份。

### 核心概念与公式

SUID权限通过在所有者的**执行位（x）**上体现。如果该文件**原本有执行权限**，则显示为小写 `s`；如果**原本没有执行权限**，则显示为大写 `S`。

**权限位示例：**
```
-rwsr-xr-x. 1 root root ... /usr/bin/passwd
```
（所有者权限位的 `x` 被 `s` 替代）

### 工作原理与应用

一个典型的例子是 `/usr/bin/passwd` 命令。普通用户可以使用它修改自己的密码，但密码最终存储在 `/etc/shadow` 文件中，而该文件对普通用户是**不可写**的。这是因为 `passwd` 命令被设置了SUID权限（属于root），当普通用户执行它时，实际上是以root身份在修改 `/etc/shadow` 文件。

**设置与取消SUID的命令：**
```bash
# 设置SUID
chmod u+s 文件名

# 取消SUID
chmod u-s 文件名
```

**注意：** 不当设置SUID（例如给 `vim` 设置）会带来严重的安全风险，可能导致普通用户获得root权限。因此，除非有明确且安全的需求，否则不应随意设置。

## SGID（Set Group ID）权限 👥

上一节我们介绍了SUID是针对用户的提权，本节中我们来看看针对组的SGID权限。SGID同样主要应用于**可执行文件**。

![](img/63cf84b211bc769ae48687a7fd2d733f_21.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_23.png)

### 核心概念

当普通用户执行一个设置了SGID权限的程序时，该程序会**临时拥有文件所属组的权限**。这通常用于让用户临时成为某个组的成员，以继承该组对某些资源的访问权限。

### 工作原理

SGID权限体现在所属组的**执行位（x）**上。同样，有执行权限时显示为小写 `s`，无则显示为大写 `S`。

![](img/63cf84b211bc769ae48687a7fd2d733f_25.png)

**权限位示例（黄色背景常见）：**
```
-rwxr-sr-x. 1 root slocate ... /usr/bin/locate
```
（所属组权限位的 `x` 被 `s` 替代）

例如，`locate` 命令需要读取 `/var/lib/mlocate/mlocate.db` 数据库文件，而该文件对其他用户无读权限。但 `locate` 命令本身设置了SGID（属于 `slocate` 组），当普通用户执行 `locate` 时，便临时拥有了 `slocate` 组的身份，从而可以读取数据库文件。

![](img/63cf84b211bc769ae48687a7fd2d733f_27.png)

**设置与取消SGID的命令：**
```bash
# 设置SGID
chmod g+s 文件名

# 取消SGID
chmod g-s 文件名
```

与SUID类似，SGID通常由系统预设，管理员一般无需手动设置。

## SBIT（Sticky Bit）权限 📌

前面两种权限针对可执行文件，而SBIT权限则**只针对目录**有效。这是一个非常实用且常用的权限。

### 核心概念

![](img/63cf84b211bc769ae48687a7fd2d733f_43.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_45.png)

当一个目录被设置了SBIT（粘滞位）权限后，**即便该目录对所有人开放了写权限（如777）**，用户也只能删除或重命名**自己创建的文件和目录**，而无法操作其他用户的文件。Root用户不受此限制。

### 工作原理与应用

![](img/63cf84b211bc769ae48687a7fd2d733f_47.png)

SBIT权限体现在其他人的**执行位（x）**上。如果该目录对其他人有执行权限，则显示为小写 `t`；如果没有，则显示为大写 `T`。

![](img/63cf84b211bc769ae48687a7fd2d733f_49.png)

**权限位示例（常见于 `/tmp` 目录）：**
```
drwxrwxrwt. 15 root root ... /tmp
```
（其他人权限位的 `x` 被 `t` 替代）

系统临时目录 `/tmp` 和 `/var/tmp` 默认就设置了SBIT权限。这保证了多个用户都可以使用这个公共空间存放临时文件，但不会互相干扰或误删。

![](img/63cf84b211bc769ae48687a7fd2d733f_51.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_53.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_55.png)

**设置与取消SBIT的命令：**
```bash
# 设置SBIT
chmod o+t 目录名

# 取消SBIT
chmod o-t 目录名
```

## ACL（Access Control List）访问控制列表 🗂️

之前介绍的权限模型（ugo）有时不够灵活。例如，我们想给一个既不是所有者，也不在所属组内的特定用户分配某个目录的权限。这时就需要用到ACL（文件系统访问控制列表）。

![](img/63cf84b211bc769ae48687a7fd2d733f_61.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_63.png)

### 核心概念

ACL可以为单个用户或用户组**单独设置**超出基本权限模型的、更精细的文件/目录访问权限。它是对传统权限模型的扩展。

![](img/63cf84b211bc769ae48687a7fd2d733f_65.png)

### 常用命令

![](img/63cf84b211bc769ae48687a7fd2d733f_67.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_69.png)

以下是使用ACL的核心命令：

![](img/63cf84b211bc769ae48687a7fd2d733f_71.png)

**1. 查看ACL权限：**
```bash
getfacl 文件或目录名
```

**2. 设置ACL权限：**
```bash
# 为用户设置权限
setfacl -m u:用户名:权限 文件或目录名
# 例如，给用户xiaowei分配读和执行权限
setfacl -m u:xiaowei:rx /ops

# 为用户组设置权限
setfacl -m g:组名:权限 文件或目录名
```

**3. 删除特定ACL条目：**
```bash
# 删除指定用户的ACL权限
setfacl -x u:用户名 文件或目录名
# 删除指定用户组的ACL权限
setfacl -x g:组名 文件或目录名
```

**4. 清除所有ACL权限：**
```bash
setfacl -b 文件或目录名
```

### 应用示例

![](img/63cf84b211bc769ae48687a7fd2d733f_73.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_75.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_77.png)

假设有一个 `/ops` 目录，属于 `ops` 组。我们想让实习生 `xiaowei` 可以查看和进入该目录，但不能修改或删除内容（即 `rx` 权限），而其他非组成员则无任何权限。

```bash
# 1. 先确保其他人无权限
chmod o=--- /ops

# 2. 为xiaowei单独设置ACL权限
setfacl -m u:xiaowei:rx /ops

# 3. 验证权限
getfacl /ops
```
此时，`xiaowei` 用户即可访问 `/ops` 目录，而其他非组成员则不能。目录的权限列表后会出现一个 `+` 号，表示存在扩展的ACL权限。

## 总结

本节课中我们一起学习了Linux的四种特殊权限：
*   **SUID**：针对可执行文件，使执行者临时获得文件所有者的身份。
*   **SGID**：针对可执行文件或目录（目录时作用不同），使执行者临时获得文件所属组的身份。
*   **SBIT**：仅针对目录，保护目录中的文件不被其他用户删除（root除外）。
*   **ACL**：提供更精细的权限控制，可以为特定用户或组单独设置权限。

![](img/63cf84b211bc769ae48687a7fd2d733f_79.png)

![](img/63cf84b211bc769ae48687a7fd2d733f_81.png)

其中，**SBIT**和**ACL**在实际运维工作中应用非常广泛，需要重点掌握其概念和设置方法。而SUID和SGID更多由系统自身管理，了解其原理即可。合理运用这些权限，可以极大地增强系统的安全性和灵活性。
