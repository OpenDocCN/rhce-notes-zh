在本节课中，我们将要学习SSH（Secure Shell）服务的基本概念、工作原理以及两种主要的身份验证方式。SSH是Linux系统远程管理的核心工具，理解其配置对于系统安全和自动化运维至关重要。

## SSH基本概念

![](img/f327e59fd8f406304e640e40d5755093_5.png)

上一节我们介绍了系统服务管理，本节中我们来看看SSH服务。默认情况下，Linux操作系统的SSH服务是开启的。我们可以通过 `systemctl status` 命令来查看SSHD服务的状态。

![](img/f327e59fd8f406304e640e40d5755093_7.png)

```bash
systemctl status sshd.service
```

命令输出显示服务默认处于运行（running）状态。

![](img/f327e59fd8f406304e640e40d5755093_9.png)

接下来，我们总结SSH的一些核心概念。

![](img/f327e59fd8f406304e640e40d5755093_11.png)

以下是SSH的核心要点列表：

![](img/f327e59fd8f406304e640e40d5755093_13.png)

![](img/f327e59fd8f406304e640e40d5755093_15.png)

1.  **OpenSSH**：在红帽Linux上实现SSH协议的开源软件。Linux系统使用OpenSSH来提供SSH服务。
2.  **功能**：SSH用于实现远程登录和管理，并且所有传输的数据都是加密的。
3.  **加密算法**：SSH使用加密算法保证安全，主要采用非对称加密（如RSA算法）。非对称加密会生成一对密钥：公钥和私钥。
4.  **客户端保存公钥**：SSH客户端需要保存服务端的公钥才能成功建立连接。客户端首次连接服务端时，服务端会发送其公钥。
5.  **公钥存储位置**：客户端将接收到的服务端公钥保存在用户家目录下的 `.ssh/known_hosts` 文件中。

![](img/f327e59fd8f406304e640e40d5755093_17.png)

例如，从客户端（如foundation）SSH到服务端（servera）后，可以在客户端查看该文件：

![](img/f327e59fd8f406304e640e40d5755093_19.png)

```bash
cat ~/.ssh/known_hosts
```

![](img/f327e59fd8f406304e640e40d5755093_21.png)

文件内容包含servera的公钥信息。如果删除此文件中的对应条目，再次连接时，客户端会重新接收并确认服务端的公钥。

## SSH配置文件与密钥位置

![](img/f327e59fd8f406304e640e40d5755093_23.png)

了解了基本连接原理后，我们来看看SSH相关的配置文件和各组件的位置。

以下是SSH相关的重要文件路径：

![](img/f327e59fd8f406304e640e40d5755093_25.png)

![](img/f327e59fd8f406304e640e40d5755093_27.png)

*   **服务端配置文件**：主要的SSH服务端配置文件位于 `/etc/ssh/sshd_config`。用户级别的配置文件可能在 `~/.ssh/config`（此文件可能需要手动创建）。
*   **服务端公钥存储**：SSH服务端的公钥通常存放在 `/etc/ssh/` 目录下，文件名以 `.pub` 结尾（例如 `ssh_host_rsa_key.pub`）。私钥文件没有 `.pub` 后缀，且权限严格，不应泄露。
*   **客户端配置文件**：SSH客户端的配置文件位于 `~/.ssh/config`，可用于定义连接特定主机的参数（如使用的密钥、端口等）。

![](img/f327e59fd8f406304e640e40d5755093_29.png)

## SSH身份验证的两种方式

![](img/f327e59fd8f406304e640e40d5755093_31.png)

SSH提供了两种身份验证机制，以确保连接的安全性和便利性。

### 1. 基于密码的身份验证

这是最常见和默认的方式。连接过程如下：
1.  客户端发起连接，服务端发送其公钥。
2.  客户端确认并保存公钥（`known_hosts`文件）。
3.  服务端提示输入用户名和密码。
4.  客户端提供密码，验证通过后建立加密连接。

![](img/f327e59fd8f406304e640e40d5755093_33.png)

此方式简单，但每次连接都需要交互式输入密码，且密码可能被暴力破解。

### 2. 基于密钥的身份验证

![](img/f327e59fd8f406304e640e40d5755093_35.png)

这种方式更安全，常用于自动化脚本或免密登录。其核心是使用两对密钥：
*   **连接加密密钥对**：用于加密SSH通信流量，在首次连接时建立。
*   **身份验证密钥对**：由用户生成，用于证明客户端身份，替代密码验证。

![](img/f327e59fd8f406304e640e40d5755093_37.png)

![](img/f327e59fd8f406304e640e40d5755093_39.png)

实现基于密钥验证的步骤如下：

![](img/f327e59fd8f406304e640e40d5755093_41.png)

![](img/f327e59fd8f406304e640e40d5755093_43.png)

![](img/f327e59fd8f406304e640e40d5755093_45.png)

1.  **在客户端生成密钥对**：使用 `ssh-keygen` 命令。
    ```bash
    ssh-keygen
    ```
    默认生成RSA密钥对，保存在 `~/.ssh/id_rsa`（私钥）和 `~/.ssh/id_rsa.pub`（公钥）。

![](img/f327e59fd8f406304e640e40d5755093_47.png)

2.  **将公钥部署到服务端**：使用 `ssh-copy-id` 命令将客户端的公钥复制到服务端相应用户的 `~/.ssh/authorized_keys` 文件中。
    ```bash
    ssh-copy-id user@serverb
    ```
    此命令会提示输入服务端用户密码，完成后即部署成功。

![](img/f327e59fd8f406304e640e40d5755093_49.png)

3.  **测试免密登录**：从客户端再次SSH到服务端，将不再需要输入密码。
    ```bash
    ssh user@serverb
    ```

![](img/f327e59fd8f406304e640e40d5755093_51.png)

**关键概念澄清**：在整个SSH连接中，实际上涉及两套密钥。一套是服务端本身的**主机密钥对**（用于加密通道），另一套是客户端生成的**用户密钥对**（用于身份验证）。`ssh-copy-id` 部署的是后者中的公钥。

![](img/f327e59fd8f406304e640e40d5755093_53.png)

![](img/f327e59fd8f406304e640e40d5755093_55.png)

## 高级主题：SSH-Agent与安全配置建议

为了进一步提升安全性和使用体验，我们可以了解以下高级配置。

### SSH-Agent：管理密钥密码

![](img/f327e59fd8f406304e640e40d5755093_57.png)

如果生成密钥对时设置了密码（`ssh-keygen` 时提示的`passphrase`），那么每次使用该密钥连接时都需要输入这个密码。`ssh-agent` 可以解决这个问题，它将解密的私钥临时保存在内存中。

![](img/f327e59fd8f406304e640e40d5755093_59.png)

使用方法如下：

![](img/f327e59fd8f406304e640e40d5755093_61.png)

```bash
# 启动ssh-agent并设置环境变量
eval $(ssh-agent)
# 将私钥添加到agent，此时会提示输入密钥密码
ssh-add ~/.ssh/id_rsa
# 之后使用该密钥连接时，将不再提示输入密钥密码
ssh user@servera
```

![](img/f327e59fd8f406304e640e40d5755093_63.png)

### SSH安全配置建议

![](img/f327e59fd8f406304e640e40d5755093_65.png)

通过修改服务端配置文件 `/etc/ssh/sshd_config` 可以增强安全性。

以下是两项重要的安全建议：

![](img/f327e59fd8f406304e640e40d5755093_67.png)

1.  **禁止root用户直接SSH登录**：将 `PermitRootLogin` 参数改为 `no`。这样，即使攻击者获取了root密码，也无法远程登录，必须通过普通用户登录后再切换（`su`或`sudo`），增加了安全层。
2.  **禁用密码验证，强制使用密钥验证**：将 `PasswordAuthentication` 参数改为 `no`。这彻底杜绝了密码被暴力破解的风险，只允许拥有对应私钥的用户连接。

![](img/f327e59fd8f406304e640e40d5755093_69.png)

修改配置后，需要重启SSH服务使更改生效：

![](img/f327e59fd8f406304e640e40d5755093_71.png)

```bash
systemctl reload sshd.service
```

![](img/f327e59fd8f406304e640e40d5755093_73.png)
