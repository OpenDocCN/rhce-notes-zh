## 概述

![](img/a0db9b19eec65b0544b267bfb3ffe249_14.png)

SELinux 是 Linux 内核中一个强大的安全模块，它通过为系统中的进程和文件对象（客体）定义安全上下文，提供了比传统用户-组-权限模型更精细的访问控制。当进程的安全上下文与文件的安全上下文不匹配时，即使传统的文件权限允许，访问也会被拒绝。本节我们将通过一个 Web 服务器的实例来理解并解决这类问题。

## SELinux 核心概念

![](img/a0db9b19eec65b0544b267bfb3ffe249_24.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_26.png)

上一节我们介绍了传统的文件权限模型，本节中我们来看看 SELinux 如何增强系统安全。

在传统权限模型中，访问控制的主体是**用户**，客体是**文件**。判断依据是用户对文件是否拥有读、写、执行权限。

而在 SELinux 的增强安全模型中：
*   **主体** 变成了 **进程**。
*   **客体** 仍然是 **文件**。
*   访问控制的核心规则是：只有当客体的安全上下文符合主体进程的安全上下文设定时，访问才被允许。

![](img/a0db9b19eec65b0544b267bfb3ffe249_28.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_30.png)

我们可以用一个简单的**公式**来描述这个思想：
**进程访问文件** 的条件是 `进程安全上下文` 匹配 `文件安全上下文`。

![](img/a0db9b19eec65b0544b267bfb3ffe249_32.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_34.png)

## 实验准备与问题复现

![](img/a0db9b19eec65b0544b267bfb3ffe249_38.png)

为了理解 SELinux 的影响，我们首先配置一个 Apache HTTPD 服务并复现一个典型问题。

以下是实验前提准备步骤：
1.  确保 `httpd` 服务已安装。可使用 `yum install httpd` 命令安装。
2.  启动 `httpd` 服务并设置为开机自启。在 RHEL 8 中，命令为 `systemctl enable --now httpd`。
3.  在默认的网站根目录 `/var/www/html/` 下创建首页文件 `index.html`，并写入简单内容，例如 `hello linux`。
4.  使用 `curl localhost` 命令测试，可以正常访问到页面内容。

现在，我们来模拟一个常见问题：将网站文件移动到另一个自定义目录。

![](img/a0db9b19eec65b0544b267bfb3ffe249_40.png)

1.  创建一个新目录，例如 `/usr/local/www/`。
2.  将 `index.html` 文件移动到新目录。
3.  修改 `/etc/httpd/conf/httpd.conf` 配置文件中的 `DocumentRoot` 参数，将其指向新目录 `/usr/local/www`。
4.  重启 `httpd` 服务后，再次使用 `curl` 访问，会发现返回 **403 Forbidden** 错误。

为什么移动了目录就无法访问了？从传统权限看，Apache 进程用户 `apache` 对新目录和文件拥有读权限。问题根源在于 SELinux。

![](img/a0db9b19eec65b0544b267bfb3ffe249_42.png)

## 分析问题：安全上下文不匹配

上一节我们复现了问题，本节中我们来分析其根本原因。

![](img/a0db9b19eec65b0544b267bfb3ffe249_44.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_46.png)

我们分别查看进程和文件的安全上下文。

![](img/a0db9b19eec65b0544b267bfb3ffe249_48.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_50.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_52.png)

*   **查看 HTTPD 进程的安全上下文**：
    ```bash
    ps -eZ | grep httpd
    ```
    输出中会包含类似 `system_u:system_r:httpd_t:s0` 的信息，其中 `httpd_t` 是进程的**类型**。

![](img/a0db9b19eec65b0544b267bfb3ffe249_54.png)

*   **查看自定义目录的安全上下文**：
    ```bash
    ls -ldZ /usr/local/www/
    ```
    输出可能类似 `unconfined_u:object_r:usr_t:s0`，其中 `usr_t` 是目录的**类型**。

![](img/a0db9b19eec65b0544b267bfb3ffe249_56.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_58.png)

关键问题在于：进程的类型 (`httpd_t`) 与文件的类型 (`usr_t`) 不匹配。SELinux 策略规定，`httpd_t` 类型的进程默认只能访问类型为 `httpd_sys_content_t` 的文件内容。因此访问被拒绝。

![](img/a0db9b19eec65b0544b267bfb3ffe249_62.png)

## SELinux 工作模式

![](img/a0db9b19eec65b0544b267bfb3ffe249_69.png)

在学习如何解决问题前，我们需要了解 SELinux 的三种运行模式。

![](img/a0db9b19eec65b0544b267bfb3ffe249_71.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_73.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_75.png)

SELinux 有三种工作模式：
*   **Enforcing**：强制模式。违反策略的行为将被阻止并记录到日志。
*   **Permissive**：宽容模式。违反策略的行为仅被记录，不会被阻止。常用于调试。
*   **Disabled**：关闭模式。SELinux 完全不起作用。

![](img/a0db9b19eec65b0544b267bfb3ffe249_77.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_79.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_80.png)

以下是管理 SELinux 模式的相关操作：

![](img/a0db9b19eec65b0544b267bfb3ffe249_82.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_84.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_86.png)

**查看当前模式**
```bash
getenforce
```

![](img/a0db9b19eec65b0544b267bfb3ffe249_87.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_88.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_90.png)

**临时切换模式**（在 Enforcing 和 Permissive 之间切换无需重启）
*   设置为 Enforcing：`setenforce 1`
*   设置为 Permissive：`setenforce 0`
*   *注意：无法使用 `setenforce` 命令切换到 Disabled 模式。*

![](img/a0db9b19eec65b0544b267bfb3ffe249_91.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_92.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_93.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_94.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_95.png)

**永久修改模式**
需要编辑配置文件 `/etc/selinux/config`（或它的软链接 `/etc/sysconfig/selinux`）。
找到 `SELINUX=` 这一行，将其值修改为 `enforcing`、`permissive` 或 `disabled`。
*   **重要**：从 `disabled` 切换到 `enforcing` 或 `permissive`，**必须重启系统**，并且系统会为整个文件系统重新打上 SELinux 标签，此过程较慢。

## 解决问题：管理安全上下文

![](img/a0db9b19eec65b0544b267bfb3ffe249_108.png)

上一节我们了解了 SELinux 的模式，本节中我们来看看如何修正安全上下文以解决访问问题。

我们的目标是修改文件或目录的安全上下文类型，使其符合进程的访问要求。主要有两种方法：修改默认规则和临时修改。

### 方法一：修改默认安全上下文并还原

此方法会为指定路径定义默认的上下文规则，之后创建的文件会自动继承，并可用命令将现有文件还原到此默认上下文。

1.  **为目录设定默认上下文类型**：
    以下命令将为 `/usr/local/www` 目录及其未来所有子内容设置默认类型为 `httpd_sys_content_t`。
    ```bash
    semanage fcontext -a -t httpd_sys_content_t “/usr/local/www(/.*)?”
    ```
    *   `semanage fcontext`: 管理文件上下文规则。
    *   `-a`: 添加规则。
    *   `-t httpd_sys_content_t`: 设置类型。
    *   `“/usr/local/www(/.*)?”`: 路径规则，`(/.*)?` 表示匹配该目录及其下所有文件和子目录。

2.  **应用新的默认上下文**：
    使用 `restorecon` 命令将上面定义的默认规则应用到实际文件系统上。
    ```bash
    restorecon -Rv /usr/local/www/
    ```
    *   `-R`: 递归处理子目录。
    *   `-v`: 显示详细过程。
    *   执行后，该目录及其内容的上下文类型会变为 `httpd_sys_content_t`。

### 方法二：临时修改安全上下文

![](img/a0db9b19eec65b0544b267bfb3ffe249_110.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_112.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_114.png)

使用 `chcon` 命令可以直接更改文件或目录的上下文，但此更改不会被默认规则记录。如果系统执行了默认上下文还原（如 `restorecon`），更改可能会被覆盖。

![](img/a0db9b19eec65b0544b267bfb3ffe249_116.png)

![](img/a0db9b19eec65b0544b267bfb3ffe249_118.png)

**直接修改类型**：
```bash
chcon -t httpd_sys_content_t /usr/local/www/index.html
```

**参考另一个文件的上下文进行修改**：
```bash
chcon --reference=/var/www/html /usr/local/www
```
这条命令会将 `/usr/local/www` 的上下文设置成与 `/var/www/html` 完全相同。

## 总结

本节课中我们一起学习了 SELinux 的核心安全模型、三种工作模式以及管理文件安全上下文的实用技巧。

![](img/a0db9b19eec65b0544b267bfb3ffe249_120.png)

关键点总结：
1.  SELinux 通过**安全上下文**（尤其是**类型**）在进程和文件之间实施强制访问控制。
2.  当 Web 服务器等进程无法访问文件时，在检查传统权限后，应排查 SELinux 上下文是否匹配。
3.  使用 `semanage fcontext` 和 `restorecon` 组合可以永久性地修正并应用正确的上下文规则。
4.  使用 `chcon` 命令可以快速进行临时调整。
5.  熟练掌握 `getenforce`、`setenforce` 和配置文件修改，以管理 SELinux 的运行模式。

通过正确配置 SELinux 安全上下文，可以在不降低系统安全性的前提下，确保各种服务正常运行。
