### 静态清单定义

![](img/dba3e96d8989484282e83dfb265cacf1_15.png)

![](img/dba3e96d8989484282e83dfb265cacf1_16.png)

在自定义清单文件中，可以按以下方式定义主机：

![](img/dba3e96d8989484282e83dfb265cacf1_18.png)

![](img/dba3e96d8989484282e83dfb265cacf1_20.png)

*   **直接指定**：可以直接写入主机的 IP 地址或域名（注意，IP 和域名被视为不同的主机）。
*   **使用通配符**：`*` 和 `all` 的含义完全相同，都代表所有主机。例如，`*.example.com` 可以匹配该域下的所有主机。
*   **使用列表**：可以使用逗号分隔多个主机或组名。
*   **使用逻辑运算符**：
    *   `:` 代表并集（或）。例如，`group1:group2` 表示属于 group1 **或** group2 的主机。
    *   `:&` 代表交集（与）。例如，`group1:&group2` 表示既属于 group1 **也**属于 group2 的主机。
    *   `:!` 代表排除（非）。例如，`group1:!host1` 表示属于 group1，但排除 host1 这台主机。

![](img/dba3e96d8989484282e83dfb265cacf1_22.png)

### 动态清单与多清单

除了静态文件，Ansible 还支持动态清单。

![](img/dba3e96d8989484282e83dfb265cacf1_30.png)

![](img/dba3e96d8989484282e83dfb265cacf1_32.png)

![](img/dba3e96d8989484282e83dfb265cacf1_34.png)

*   **动态清单**：可以使用如 `ansible-inventory` 命令动态生成清单，并通过 `-i` 选项指定，或导出为 JSON 格式文件后再使用。
    ```bash
    # 查看动态清单（JSON格式）
    ansible-inventory --list
    # 指定动态清单脚本
    ansible all -i dynamic_inventory.py -m ping
    ```
*   **多清单**：可以将 `inventory` 参数指向一个**目录**，该目录下可以存放多个清单文件，Ansible 会自动合并它们。如果动态清单中定义的组是静态清单中某个组的子组，在静态清单中需要为该子组**留空占位**。

## 管理并行执行：分叉与滚动更新 ⚙️

![](img/dba3e96d8989484282e83dfb265cacf1_45.png)

![](img/dba3e96d8989484282e83dfb265cacf1_47.png)

![](img/dba3e96d8989484282e83dfb265cacf1_48.png)

默认情况下，Ansible 按顺序执行 Playbook 中的每个任务，并在所有主机上完成一个任务后，再开始下一个任务。虽然它能同时连接所有主机，但对于数百或数千台主机，我们需要控制并发数量。

![](img/dba3e96d8989484282e83dfb265cacf1_50.png)

### 使用分叉控制并发

![](img/dba3e96d8989484282e83dfb265cacf1_52.png)

**分叉** 用于控制 Ansible 每次能与多少台主机并行建立连接并执行任务。

![](img/dba3e96d8989484282e83dfb265cacf1_54.png)

![](img/dba3e96d8989484282e83dfb265cacf1_55.png)

![](img/dba3e96d8989484282e83dfb265cacf1_57.png)

*   **默认值**：分叉数默认为 5。
*   **设置方法**：
    *   在命令行中，使用 `-f` 或 `--forks` 参数。
        ```bash
        # 每次只在一台主机上执行任务
        ansible all -m ping -f 1
        ```
    *   在 `ansible.cfg` 配置文件的 `[defaults]` 部分设置 `forks` 参数。
    *   执行 Playbook 时也可使用 `-f` 参数。
        ```bash
        ansible-playbook site.yml -f 10
        ```

![](img/dba3e96d8989484282e83dfb265cacf1_59.png)

![](img/dba3e96d8989484282e83dfb265cacf1_61.png)

![](img/dba3e96d8989484282e83dfb265cacf1_63.png)

当主机数量超过分叉数时，超出的主机会排队等待。设置分叉数为 1 时，Ansible 会在一台主机上完成所有任务后，再处理下一台主机。

![](img/dba3e96d8989484282e83dfb265cacf1_65.png)

![](img/dba3e96d8989484282e83dfb265cacf1_67.png)

![](img/dba3e96d8989484282e83dfb265cacf1_68.png)

![](img/dba3e96d8989484282e83dfb265cacf1_69.png)

### 使用滚动更新控制批次

![](img/dba3e96d8989484282e83dfb265cacf1_71.png)

![](img/dba3e96d8989484282e83dfb265cacf1_73.png)

![](img/dba3e96d8989484282e83dfb265cacf1_75.png)

**滚动更新** 用于控制执行 Play 时，每批处理多少台主机。这在需要按顺序更新服务（例如先停止一台，再停止下一台）以避免服务全部中断的场景中非常有用。

*   **设置方法**：在 Play 中使用 `serial` 关键字。
    ```yaml
    - name: 滚动更新 Web 服务器
      hosts: webservers
      serial: 1  # 每次只在一台主机上执行整个 Play
      tasks:
        - name: 停止服务
          service:
            name: nginx
            state: stopped
        - name: 更新应用
          copy:
            src: /tmp/app.war
            dest: /opt/tomcat/webapps/
        - name: 启动服务
          service:
            name: nginx
            state: started
    ```

**分叉与滚动更新的区别**：
*   `forks` 控制**单个任务**在所有主机上的并发数。
*   `serial` 控制**整个 Play** 按批次在所有主机上执行的顺序。`serial` 定义的每批主机数不能超过 `forks` 的值，否则会以 `forks` 值为准。

## 组织内容：包含与导入 📦

当 Playbook 变得非常庞大时，将所有任务写在一个文件中会难以维护。Ansible 允许你将内容模块化，通过**包含**或**导入**的方式组合多个文件，实现代码复用和易于管理。

![](img/dba3e96d8989484282e83dfb265cacf1_77.png)

### 导入操作

![](img/dba3e96d8989484282e83dfb265cacf1_79.png)

![](img/dba3e96d8989484282e83dfb265cacf1_80.png)

![](img/dba3e96d8989484282e83dfb265cacf1_82.png)

![](img/dba3e96d8989484282e83dfb265cacf1_84.png)

**导入**是静态操作。Ansible 在**最初解析 Playbook 时**就会处理导入语句，将被导入的内容完整地插入到指定位置。

![](img/dba3e96d8989484282e83dfb265cacf1_86.png)

![](img/dba3e96d8989484282e83dfb265cacf1_87.png)

![](img/dba3e96d8989484282e83dfb265cacf1_89.png)

*   **导入整个 Playbook**：使用 `import_playbook` 指令。这只能在 Playbook 的顶层使用，不能用在某个任务内部。
    ```yaml
    # main.yml
    - import_playbook: webserver.yml
    - import_playbook: dbserver.yml
    ```
*   **导入任务文件**：使用 `import_tasks` 指令。任务文件是一个只包含任务列表的“平面文件”，没有 `hosts`、`name` 等 Playbook 头部信息。
    ```yaml
    # tasks/main.yml
    - name: 安装 HTTPD
      yum:
        name: httpd
        state: present

    - name: 导入通用配置任务
      import_tasks: common_tasks.yml
    ```

![](img/dba3e96d8989484282e83dfb265cacf1_91.png)

![](img/dba3e96d8989484282e83dfb265cacf1_93.png)

### 包含操作

![](img/dba3e96d8989484282e83dfb265cacf1_97.png)

![](img/dba3e96d8989484282e83dfb265cacf1_98.png)

![](img/dba3e96d8989484282e83dfb265cacf1_99.png)

**包含**是动态操作。Ansible 在**运行期间执行到包含语句时**才会处理被包含的内容。

![](img/dba3e96d8989484282e83dfb265cacf1_101.png)

![](img/dba3e96d8989484282e83dfb265cacf1_102.png)

![](img/dba3e96d8989484282e83dfb265cacf1_104.png)

![](img/dba3e96d8989484282e83dfb265cacf1_105.png)

*   **包含任务文件**：使用 `include_tasks` 指令。
    ```yaml
    - name: 包含动态决策的任务
      include_tasks: setup.yml
    ```

![](img/dba3e96d8989484282e83dfb265cacf1_106.png)

![](img/dba3e96d8989484282e83dfb265cacf1_108.png)

![](img/dba3e96d8989484282e83dfb265cacf1_110.png)

**包含与导入的重要区别**：
1.  **处理时机**：导入是初期静态合并；包含是运行时动态加载。
2.  由于包含是动态的，因此**不能**在包含的任务文件中使用 `static` 指令、`tags` 指令，也**不能**在包含的任务中直接使用 `notify` 触发在父 Playbook 中定义的 Handler。
3.  **变量使用**：在导入/包含时，尽量使用变量，这有助于在不同文件间传递和覆盖参数，使组合更加灵活。

![](img/dba3e96d8989484282e83dfb265cacf1_112.png)

![](img/dba3e96d8989484282e83dfb265cacf1_113.png)

![](img/dba3e96d8989484282e83dfb265cacf1_114.png)

> **提示**：在学习了更强大的 **角色** 功能后，许多包含/导入的场景可以被角色替代，使项目结构更清晰。

本节课中我们一起学习了管理大型 Ansible 项目的核心技巧：

![](img/dba3e96d8989484282e83dfb265cacf1_129.png)

1.  **灵活引用清单**：掌握了使用通配符、列表和逻辑运算符来精确选择目标主机，并了解了动态清单与多清单的用法。
2.  **控制任务执行**：
    *   通过 **`forks`** 参数控制任务执行的并发主机数量。
    *   通过 **`serial`** 关键字实现滚动更新，控制 Play 的执行批次，这对于需要顺序操作的场景至关重要。
3.  **模块化组织 Playbook**：通过 **`import_playbook`**、**`import_tasks`** 和 **`include_tasks`** 指令，可以将大型 Playbook 拆分为可重用的模块，提高了代码的可维护性和复用性。理解了“导入”与“包含”在静态/动态处理上的关键区别。
