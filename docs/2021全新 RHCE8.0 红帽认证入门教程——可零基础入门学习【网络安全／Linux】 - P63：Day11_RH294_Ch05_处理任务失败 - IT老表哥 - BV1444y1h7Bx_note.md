# Ansible RHCE 8.0 教程：5：处理任务失败 🛠️

在本节课中，我们将学习如何在 Ansible Playbook 中处理任务失败。我们将探讨 `notify` 和 `handler` 的联动机制，以及如何通过 `ignore_errors`、`force_handlers` 和 `failed_when` 等指令来控制任务执行流程，确保自动化剧本的健壮性。

## 概述 📋

Ansible 默认情况下，当一个任务执行失败时，后续的任务将不会被执行，整个 Playbook 会退出。但在实际运维中，我们可能希望忽略某些非关键错误，或者在任务状态改变时触发特定操作（如重启服务）。本节课将介绍几种处理任务失败和状态变化的常用方法。

上一节我们介绍了 Ansible 的基础任务编写，本节中我们来看看如何让任务执行更智能、更健壮。

## Notify 与 Handler 机制 🔄

![](img/326661d72dc775e09c006c17faa22452_1.png)

`notify` 和 `handler` 是 Ansible 中用于处理“状态改变”的一对组合。当一个任务（例如修改配置文件）的执行结果发生了**改变**（`changed`），它可以 `notify`（通知）一个对应的 `handler`。`handler` 本质上也是一个任务，它只会在被通知时执行一次，通常用于执行重启服务、重新加载配置等操作。

![](img/326661d72dc775e09c006c17faa22452_3.png)

![](img/326661d72dc775e09c006c17faa22452_5.png)

其核心关系可以理解为：
```
任务（状态 changed） --[notify]--> 对应的 Handler（执行特定操作）
```

以下是部署 HTTPD 服务并确保其配置正确的 Playbook 示例。我们将任务分解为几个步骤，并在修改配置后触发服务重启。

![](img/326661d72dc775e09c006c17faa22452_25.png)

![](img/326661d72dc775e09c006c17faa22452_26.png)

![](img/326661d72dc775e09c006c17faa22452_27.png)

![](img/326661d72dc775e09c006c17faa22452_28.png)

![](img/326661d72dc775e09c006c17faa22452_30.png)

**Playbook 结构说明：**
1.  安装 HTTPD 软件包。
2.  创建测试页面。
3.  修改 HTTPD 监听端口配置，并在配置改变时 `notify` 一个 handler。
4.  启动 HTTPD 服务。
5.  配置防火墙规则。
6.  定义用于重启 HTTPD 的 `handler`。

![](img/326661d72dc775e09c006c17faa22452_32.png)

![](img/326661d72dc775e09c006c17faa22452_33.png)

![](img/326661d72dc775e09c006c17faa22452_35.png)

![](img/326661d72dc775e09c006c17faa22452_36.png)

```yaml
---
- name: Deploy and Configure HTTPD Service
  hosts: web
  tasks:
    # 1. 安装 HTTPD 软件包
    - name: Install HTTPD
      yum:
        name: httpd
        state: present

    # 2. 创建测试页面
    - name: Create test page
      copy:
        content: “This is a test page.\n”
        dest: /var/www/html/index.html
        owner: apache
        group: apache
        mode: ‘0644’

    # 3. 修改配置文件，如果端口被改动，则通知 handler
    - name: Configure HTTPD listen port
      lineinfile:
        path: /etc/httpd/conf/httpd.conf
        regexp: ‘^Listen‘
        line: ‘Listen 80‘
        state: present
      notify: restart httpd service # 关键：状态改变时发出通知

    # 4. 启动 HTTPD 服务
    - name: Start HTTPD service
      service:
        name: httpd
        state: started
        enabled: yes

    # 5. 配置防火墙
    - name: Configure firewall for HTTPD
      firewalld:
        port: 80/tcp
        permanent: yes
        immediate: yes
        state: enabled

  # 6. 定义 handlers
  handlers:
    - name: restart httpd service
      service:
        name: httpd
        state: restarted
```
**关键点解释：**
*   在“Configure HTTPD listen port”任务中，我们使用了 `lineinfile` 模块来确保配置文件中 `Listen` 行是 `Listen 80`。
*   该任务末尾的 `notify: restart httpd service` 表示，**只有当这个任务实际修改了文件（状态为 `changed`）**，才会触发名为 `“restart httpd service”` 的 handler。
*   在 `handlers` 部分，我们定义了与之同名的 handler，其操作是重启 httpd 服务。
*   如果 Playbook 再次运行，且配置文件已是 `Listen 80`，则该任务状态为 `ok`（未改变），不会触发 handler，服务也不会被不必要的重启。

### 忽略错误：`ignore_errors`

![](img/326661d72dc775e09c006c17faa22452_55.png)

![](img/326661d72dc775e09c006c17faa22452_57.png)

![](img/326661d72dc775e09c006c17faa22452_59.png)

![](img/326661d72dc775e09c006c17faa22452_61.png)

使用 `ignore_errors: yes` 可以让 Ansible 忽略当前任务的错误，继续执行后续任务。这适用于一些非关键或可预期的失败场景。

![](img/326661d72dc775e09c006c17faa22452_63.png)

![](img/326661d72dc775e09c006c17faa22452_64.png)

![](img/326661d72dc775e09c006c17faa22452_66.png)

![](img/326661d72dc775e09c006c17faa22452_68.png)

```yaml
- name: This task might fail
  command: /some/failing/command
  ignore_errors: yes
```

![](img/326661d72dc775e09c006c17faa22452_70.png)

![](img/326661d72dc775e09c006c17faa22452_72.png)

### 强制处理 Handlers：`force_handlers`

![](img/326661d72dc775e09c006c17faa22452_73.png)

![](img/326661d72dc775e09c006c17faa22452_75.png)

![](img/326661d72dc775e09c006c17faa22452_77.png)

![](img/326661d72dc775e09c006c17faa22452_79.png)

默认情况下，如果 Playbook 中某个任务失败导致中途退出，即使前面有任务触发了 `notify`，对应的 `handler` 也不会执行。使用 `force_handlers: yes` 可以强制让已通知的 handler 在 Playbook 退出前执行。

![](img/326661d72dc775e09c006c17faa22452_81.png)

![](img/326661d72dc775e09c006c17faa22452_82.png)

![](img/326661d72dc775e09c006c17faa22452_83.png)

![](img/326661d72dc775e09c006c17faa22452_85.png)

```yaml
- hosts: all
  force_handlers: yes
  tasks:
    - name: A task that notifies a handler
      debug:
        msg: “This will notify”
      changed_when: yes # 手动标记为“已改变”，以触发 notify
      notify: my handler

    - name: A task that fails
      command: /bin/false

  handlers:
    - name: my handler
      debug:
        msg: “This handler will run even if a task fails later.”
```

### 自定义失败条件：`failed_when`

![](img/326661d72dc775e09c006c17faa22452_94.png)

![](img/326661d72dc775e09c006c17faa22452_96.png)

![](img/326661d72dc775e09c006c17faa22452_97.png)

`failed_when` 允许你根据任务的输出（`result`）来自定义失败的条件。这非常灵活，例如，你可以命令执行返回特定字符串时才视为失败。

![](img/326661d72dc775e09c006c17faa22452_99.png)

![](img/326661d72dc775e09c006c17faa22452_100.png)

![](img/326661d72dc775e09c006c17faa22452_102.png)

![](img/326661d72dc775e09c006c17faa22452_104.png)

```yaml
- name: Check if a process is running
  shell: ps aux | grep -v grep | grep myapp
  register: app_status
  failed_when: “‘running‘ not in app_status.stdout“ # 当输出中没有‘running‘字样时才失败
  ignore_errors: yes # 可以结合使用
```

![](img/326661d72dc775e09c006c17faa22452_106.png)

![](img/326661d72dc775e09c006c17faa22452_107.png)

![](img/326661d72dc775e09c006c17faa22452_109.png)

一个常见的用法是反转默认逻辑（默认返回码 `rc!=0` 为失败）：
```yaml
- name: This command is expected to fail
  command: /usr/bin/false
  register: command_result
  failed_when: command_result.rc == 0 # 仅当返回码为0（成功）时，才标记此任务为失败
```

![](img/326661d72dc775e09c006c17faa22452_111.png)

### 任务状态改变触发器：`changed_when`

与 `failed_when` 类似，`changed_when` 用于手动定义任务何时算作“已改变”。这对于那些总是返回 `changed` 状态的命令或脚本非常有用，你可以根据其真实输出决定是否触发后续的 `notify`。

![](img/326661d72dc775e09c006c17faa22452_113.png)

![](img/326661d72dc775e09c006c17faa22452_115.png)

```yaml
- name: Create a directory if it doesn‘t exist
  command: ls -ld /tmp/mydir
  register: dir_check
  changed_when: dir_check.rc != 0 # 当命令执行失败（即目录不存在）时，标记为 changed
  notify: create the dir

![](img/326661d72dc775e09c006c17faa22452_117.png)

![](img/326661d72dc775e09c006c17faa22452_118.png)

handlers:
  - name: create the dir
    file:
      path: /tmp/mydir
      state: directory
```

## 任务块：`block`，`rescue`，`always`

![](img/326661d72dc775e09c006c17faa22452_120.png)

![](img/326661d72dc775e09c006c17faa22452_122.png)

![](img/326661d72dc775e09c006c17faa22452_123.png)

Ansible 提供了类似于编程语言中 `try-catch-finally` 的结构，即 `block`。

![](img/326661d72dc775e09c006c17faa22452_125.png)

![](img/326661d72dc775e09c006c17faa22452_126.png)

*   **`block`**: 定义要执行的主要任务序列。
*   **`rescue`**: 如果 `block` 中的**任何**任务失败，则执行这里的任务，用于错误恢复。
*   **`always`**: 无论 `block` 中的任务成功还是失败，**之后**都会执行这里的任务，常用于清理工作。

![](img/326661d72dc775e09c006c17faa22452_128.png)

![](img/326661d72dc775e09c006c17faa22452_129.png)

```yaml
- name: Handle errors with block
  block:
    - name: Attempt a risky operation
      debug:
        msg: “This is the main task that might fail.“

    - name: Another task in the block
      debug:
        msg: “This runs only if the first task succeeds.“

  rescue:
    - name: Run only if the block failed
      debug:
        msg: “Something went wrong in the block. Executing rescue steps.“

  always:
    - name: This always runs
      debug:
        msg: “This cleanup step runs regardless of success or failure.“
```

本节课中我们一起学习了 Ansible 中处理任务失败和状态变化的核心技巧。

我们首先深入了解了 `notify` 和 `handler` 的联动机制，这是实现“幂等性”和“状态驱动”自动化的关键。接着，我们探讨了多种控制任务流程的方法：
*   使用 `ignore_errors` 忽略非关键错误。
*   使用 `force_handlers` 确保关键 handler 总能执行。
*   使用 `failed_when` 和 `changed_when` 精细定义任务的成功、失败和改变状态。
*   使用 `block`， `rescue`， `always` 来构建更健壮的错误处理逻辑。
