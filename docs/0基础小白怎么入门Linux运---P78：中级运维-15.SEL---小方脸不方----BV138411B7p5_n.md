## 授权访问的实践 🔐

上一节我们介绍了用户权限管理的基本命令。本节中我们来看看授权访问在实际环境中的应用和注意事项。

授权访问功能非常有用，后续的许多实验都会频繁使用它。虽然相关命令很简单，但却是数据库管理的核心操作之一。

![](img/f75bd00d0b2341e6aa21636c05f4892f_11.png)

关于防火墙策略，需要特别注意：`iptables` 和 `firewalld` 是两种不同的防火墙管理工具，它们的默认策略完全相反。`iptables` 默认放行所有端口，而 `firewalld` 默认禁止所有端口。为了避免规则冲突，建议在系统中只启用其中一种工具。

## 存储引擎详解 🗃️

接下来，我们将深入探讨MySQL的存储引擎。在第一节课介绍MySQL特点时，我们就曾提到过这个概念。

![](img/f75bd00d0b2341e6aa21636c05f4892f_15.png)

存储引擎决定了数据库表如何存储、索引和查询数据。选择一个合适的存储引擎至关重要，不过MySQL通常已经为我们选好了默认的最佳选项。

### 常见存储引擎对比

![](img/f75bd00d0b2341e6aa21636c05f4892f_17.png)

目前，最常用的存储引擎是 **InnoDB**，它功能完善、性能强大，也是当前版本的默认引擎。另一种常见的引擎是 **MyISAM**，它是InnoDB的前身。

我们可以通过查看数据库文件来区分它们：
*   InnoDB 引擎的表通常对应 `.frm`（表结构）和 `.ibd`（表数据和索引）文件。
*   MyISAM 引擎的表则对应 `.frm`（表结构）、`.MYD`（表数据）和 `.MYI`（表索引）文件。

以下是两种引擎的主要特性对比：

*   **事务支持**：InnoDB 支持事务（ACID特性），而 MyISAM 不支持。事务是保证数据操作原子性、一致性、隔离性和持久性的关键机制。
*   **外键约束**：InnoDB 支持外键约束，用于维护表间数据的引用完整性；MyISAM 不支持。
*   **并发性能**：在高并发读写场景下，InnoDB 的性能通常优于 MyISAM。
*   **全文索引**：在较早版本中，MyISAM 支持全文索引而 InnoDB 不支持，但新版本的 InnoDB 也已支持该功能。
*   **查询速度**：对于只读或读多写少的场景，MyISAM 的查询速度可能更快，因为它存储了表的行数等元数据。

总体而言，**InnoDB** 因其对事务、崩溃恢复和外键的支持，已成为绝大多数应用场景的首选。**MyISAM** 则适用于主要进行读取操作、且不需要事务支持的特定情况。

### 其他存储引擎简介

除了上述两种，MySQL还支持其他一些存储引擎，各有其特殊用途：

![](img/f75bd00d0b2341e6aa21636c05f4892f_23.png)

*   **MEMORY**：将数据存储在内存中，速度极快，但服务器重启后数据会丢失。适用于缓存或临时数据。
*   **BLACKHOLE**（黑洞）：接收数据但不实际存储，任何写入操作都会“消失”。常用于数据复制和中转场景。
*   **CSV**：将数据以逗号分隔值格式存储，便于与电子表格等外部程序交换数据。
*   **ARCHIVE**：适用于存储大量历史归档数据，支持插入和查询，但不支持更新和删除。

### 修改存储引擎的方法

![](img/f75bd00d0b2341e6aa21636c05f4892f_43.png)

![](img/f75bd00d0b2341e6aa21636c05f4892f_45.png)

![](img/f75bd00d0b2341e6aa21636c05f4892f_47.png)

主要有三种方式修改存储引擎：

1.  **修改配置文件（全局默认）**
    编辑MySQL配置文件（如 `/etc/my.cnf` 或 `/etc/mysql/my.cnf`），在 `[mysqld]` 部分添加或修改以下行：
    ```ini
    default-storage-engine=MyISAM
    ```
    此方法会改变之后**新创建表**的默认引擎。修改后需要重启MySQL服务生效。由于InnoDB是主流选择，通常不建议轻易修改全局默认引擎。

2.  **建表时指定（最常用）**
    在创建表时，使用 `ENGINE` 子句直接指定存储引擎。这是最灵活、最推荐的方式。
    ```sql
    CREATE TABLE my_table (
        id INT AUTO_INCREMENT PRIMARY KEY,
        name VARCHAR(100)
    ) ENGINE=MyISAM DEFAULT CHARSET=utf8mb4;
    ```

![](img/f75bd00d0b2341e6aa21636c05f4892f_49.png)

![](img/f75bd00d0b2341e6aa21636c05f4892f_51.png)

3.  **修改已有表的引擎**
    使用 `ALTER TABLE` 语句可以修改已有表的存储引擎。
    ```sql
    ALTER TABLE my_table ENGINE = InnoDB;
    ```
    **注意**：此操作可能存在风险，特别是当两种引擎支持的特性不同时（如从MyISAM转为InnoDB，原有的全文索引可能受影响）。务必在操作前确认兼容性并备份数据。

![](img/f75bd00d0b2341e6aa21636c05f4892f_53.png)

### 查看存储引擎

可以使用以下命令查看表的存储引擎信息：
```sql
-- 查看建表语句，其中包含引擎信息
SHOW CREATE TABLE table_name;

-- 查看表的详细状态信息，包括引擎
SHOW TABLE STATUS LIKE 'table_name';
```

## “黑洞”引擎演示 🕳️

最后，我们来演示一下特殊的 **BLACKHOLE** 存储引擎。

创建一个使用黑洞引擎的表：
```sql
CREATE TABLE blackhole_demo (
    id INT,
    data VARCHAR(10)
) ENGINE=BLACKHOLE;
```
这个表看起来和普通表没有区别。

向其中插入数据：
```sql
INSERT INTO blackhole_demo VALUES (1, 'test');
```
命令执行成功，提示影响了一行。

然而，查询该表：
```sql
SELECT * FROM blackhole_demo;
```
结果集是空的。插入的数据如同被“黑洞”吞噬，没有真正存储下来。

**黑洞引擎的作用**：它并不存储数据，但会正常记录二进制日志（Binary Log）。这在主从复制架构中非常有用，例如，可以配置一个不存储数据、仅用于转发日志的中间节点，以节省磁盘空间并简化架构。我们将在后续关于主从复制的课程中具体应用它。
