# 华为云PaaS微服务治理技术 - P116：08.学成在线项目部署-portalview-接口变更自动升级

在本节课中，我们将学习如何修复学成在线项目中 `portalview` 微服务的接口问题，并通过重新构建和上传镜像、更新工作负载的方式，实现服务的自动升级。我们将重点关注因依赖管理疏忽导致的接口错误，以及完整的升级流程。

## 问题定位与原因分析

![](img/fa009e4991270bacd3b8a010c4bed8c2_1.png)

上一节我们介绍了服务部署的基本流程，本节中我们来看看如何解决接口调用错误的问题。

![](img/fa009e4991270bacd3b8a010c4bed8c2_3.png)

![](img/fa009e4991270bacd3b8a010c4bed8c2_5.png)

首先，检查 `portalview` 服务的源代码。发现其接口调用路径中似乎包含了一个多余的前缀。但进一步检查代码，发现该前缀实际上是一个空字符串，理论上不应导致问题。

![](img/fa009e4991270bacd3b8a010c4bed8c2_6.png)

![](img/fa009e4991270bacd3b8a010c4bed8c2_8.png)

本地程序运行正常，但打包成镜像并部署到云平台后，接口调用就出现了错误。原因在于构建镜像的脚本存在疏漏。

以下是构建脚本的关键部分，它负责将依赖的模块安装到本地Maven仓库：

![](img/fa009e4991270bacd3b8a010c4bed8c2_10.png)

```bash
# 构建脚本示例片段
mvn clean install -DskipTests
```

![](img/fa009e4991270bacd3b8a010c4bed8c2_12.png)

![](img/fa009e4991270bacd3b8a010c4bed8c2_13.png)

问题在于，该脚本遗漏了对 `xc-service-api` 模块的打包和安装。因此，当构建 `portalview` 服务时，Maven从本地仓库拉取的是旧版本的 `xc-service-api` 依赖包。旧版本API中的接口路径定义存在错误，导致了最终部署的服务接口异常。

![](img/fa009e4991270bacd3b8a010c4bed8c2_15.png)

## 修复与重新构建镜像

![](img/fa009e4991270bacd3b8a010c4bed8c2_17.png)

找到问题根源后，我们需要修复构建流程并生成新的镜像。

![](img/fa009e4991270bacd3b8a010c4bed8c2_19.png)

1.  **修改构建脚本**：在构建脚本中，添加对 `xc-service-api` 模块的打包和安装命令，确保本地仓库中的依赖是最新版本。
2.  **提交代码**：将修改后的构建脚本提交到Git仓库。
3.  **拉取与执行**：在部署服务器上，从Git仓库拉取最新的构建脚本。
4.  **重新构建镜像**：执行构建命令。**关键点在于，必须为本次构建的镜像指定一个新的版本号**（例如，从 `1.0` 升级到 `1.0.1`）。这样云平台的镜像仓库才能区分新旧版本。

执行上述步骤后，云平台镜像仓库中将出现两个版本的 `portalview` 镜像。

![](img/fa009e4991270bacd3b8a010c4bed8c2_21.png)

## 服务升级操作

![](img/fa009e4991270bacd3b8a010c4bed8c2_23.png)

![](img/fa009e4991270bacd3b8a010c4bed8c2_25.png)

新版本镜像上传成功后，即可对线上运行的服务进行升级。

![](img/fa009e4991270bacd3b8a010c4bed8c2_26.png)

![](img/fa009e4991270bacd3b8a010c4bed8c2_28.png)

![](img/fa009e4991270bacd3b8a010c4bed8c2_30.png)

以下是升级 `portalview` 工作负载的步骤：

![](img/fa009e4991270bacd3b8a010c4bed8c2_32.png)

1.  在云平台控制台，进入“工作负载”管理页面。
2.  找到 `portalview` 对应的工作负载，点击进入详情页。
3.  在操作选项中，选择“更换镜像”。
4.  在弹出的界面中，选择我们刚刚上传的新版本镜像（如 `v1.0.1`）。
5.  点击“提交”或“确定”，开始升级。

升级过程中，云平台会先启动一个新版本的Pod实例，待其运行就绪后，再逐步终止旧版本的Pod，从而实现服务的无缝更新。

## 升级后验证与问题处理

服务升级完成后，需要进行验证。由于我们变更了接口，需要特别注意一个关键操作：**在云平台的服务目录中，手动删除旧的 `portalview` 服务注册信息**。这是因为微服务名未变但接口已变更，直接注册可能会产生冲突。

删除旧服务注册信息后，重启工作负载。稍等片刻，新的服务实例会重新注册到服务目录中。

此时，再次进入服务目录，查看 `portalview` 的服务契约，确认接口路径已修正为正确格式。随后，通过API测试工具调用接口，验证功能是否恢复正常。

## 进阶：配置自动升级触发器

我们刚才演示的是手动升级流程。云平台通常提供更高效的自动升级机制。

例如，可以在镜像仓库中配置一个“触发器”。触发条件可以设置为“当有新版本的 `portalview` 镜像推送至仓库时”。触发动作则配置为“自动更新指定集群、命名空间下的 `portalview` 工作负载”。

![](img/fa009e4991270bacd3b8a010c4bed8c2_34.png)

配置成功后，未来我们只需推送新镜像，系统就会自动完成后续的部署升级工作，实现持续交付。

![](img/fa009e4991270bacd3b8a010c4bed8c2_35.png)

## 课程总结

本节课中我们一起学习了微服务接口变更后的完整升级流程。

我们首先分析了因构建脚本遗漏依赖导致接口错误的根本原因。接着，通过修改脚本、指定新版本号重新构建并上传镜像。然后，在云平台工作负载中手动执行了更换镜像的升级操作。升级后，我们通过删除旧服务注册信息并重启服务完成了验证。最后，我们了解了如何通过配置镜像仓库触发器来实现自动升级，提升运维效率。

![](img/fa009e4991270bacd3b8a010c4bed8c2_37.png)

核心要点总结：
*   **接口变更后**，需删除服务目录中的旧注册信息。
*   **重新构建镜像时**，务必提升镜像版本号（如 `v1.0.1`）。
*   **升级流程**：构建新镜像 -> 上传至仓库 -> 在工作负载中更换镜像 -> 验证。
*   **自动升级**可通过配置镜像仓库的**触发器**来实现。
