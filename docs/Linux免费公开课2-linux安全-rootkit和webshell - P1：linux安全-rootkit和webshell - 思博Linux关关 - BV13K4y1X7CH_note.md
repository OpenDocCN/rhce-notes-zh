# Linux安全：提权问题与检查 🔒

![](img/728df35f7218d0b065d59cd63b0bebd4_1.png)

在本节课中，我们将学习Linux系统中的两大安全威胁：提权（Privilege Escalation）和Rootkit/Webshell。我们将了解它们的基本概念、常见攻击手法，以及如何进行检查和防御。课程内容将结合具体实例，力求简单直白，让初学者也能理解。

## 课程概述 📋

本节课将分为三个主要部分：
1.  **提权问题介绍与检查**：了解攻击者如何将普通用户权限提升为管理员（root）权限。
2.  **Webshell介绍与检查**：认识通过Web服务植入的后门木马及其危害。
3.  **日志审计**：学习如何通过日志记录和分析来追踪入侵痕迹。

## 第一部分：提权问题介绍与检查 ⚠️

提权通常发生在攻击者已经渗透进入一台Linux系统后。他们需要将获得的普通用户权限提升为管理员（root）权限，从而完全控制系统。

### 1. SUDO配置漏洞提权

在Linux中，`/etc/sudoers` 文件用于配置哪些用户能以何种身份执行哪些命令。配置不当可能导致严重漏洞。

**示例：危险的sudoers配置**
假设 `/etc/sudoers` 文件中有如下配置：
```
%wheel ALL=(ALL:ALL) ALL
```
这表示 `wheel` 组中的用户可以切换成任何用户执行任何命令。

但如果配置存在限制，例如：
```
test ALL=(ALL,!root) ALL
```
这表示 `test` 用户可以切换成除 `root` 外的任何用户。然而，利用 `sudo` 命令的特定参数，仍可能绕过限制。

**攻击演示：**
1.  切换到 `test` 用户：`su - test`
2.  正常情况下，`test` 用户无法修改 `root` 密码：`passwd root` （提示权限不足）
3.  使用 `sudo` 的 `-u#-1` 参数尝试提权：
    ```bash
    sudo -u#-1 passwd root
    ```
    输入 `test` 用户的密码后，竟然可以设置 `root` 的新密码。
4.  验证当前用户身份：
    ```bash
    sudo -u#-1 id
    ```
    输出显示当前用户为 `root`。

**漏洞原因与修复：**
此漏洞源于 `sudo` 软件包版本过低（低于 1.8.28）。攻击者可能利用此漏洞提权。
*   **检查版本**：`rpm -q sudo`
*   **解决方案**：将 `sudo` 升级到 1.8.28 或更高版本。

### 2. 通配符劫持（Wildcard Injection）提权

某些命令（如 `tar`）在使用通配符（`*`）时，如果目录中存在特殊命名的文件，可能导致命令执行非预期的操作。

![](img/728df35f7218d0b065d59cd63b0bebd4_3.png)

![](img/728df35f7218d0b065d59cd63b0bebd4_4.png)

**基础知识：通配符劫持**
例如，当前目录有一个名为 `--help` 的文件。执行 `ls *` 时，本应列出所有文件，但实际效果变成了 `ls --help`，将文件内容作为参数传递给了 `ls` 命令。

![](img/728df35f7218d0b065d59cd63b0bebd4_6.png)

**攻击演示：利用 `tar` 命令**
1.  假设目录中有以下文件：
    *   `data/` （正常目录）
    *   `--checkpoint=1`
    *   `--checkpoint-action=exec=sh shell.sh`
    *   `shell.sh` （一个脚本，内容为 `echo “恶意代码执行！”`）
2.  管理员意图打包 `data` 目录：`tar czf backup.tar.gz *`
3.  由于通配符 `*` 展开，实际执行的命令类似于：
    ```bash
    tar czf backup.tar.gz --checkpoint=1 --checkpoint-action=exec=sh shell.sh data/
    ```
4.  这会导致 `tar` 命令在执行过程中，运行 `shell.sh` 脚本。如果该脚本是反弹shell等恶意代码，系统就会被控制。

![](img/728df35f7218d0b065d59cd63b0bebd4_8.png)

![](img/728df35f7218d0b065d59cd63b0bebd4_10.png)

**防御建议：**
*   避免在敏感命令中使用通配符 `*`，尽量指定具体的文件或目录名。
*   对用户上传的文件名进行严格过滤。

![](img/728df35f7218d0b065d59cd63b0bebd4_11.png)

![](img/728df35f7218d0b065d59cd63b0bebd4_13.png)

![](img/728df35f7218d0b065d59cd63b0bebd4_15.png)

## 第二部分：Webshell介绍与检查 🕸️

![](img/728df35f7218d0b065d59cd63b0bebd4_21.png)

Webshell是一种通过Web服务（如Apache+PHP）植入的后门脚本。攻击者可以通过浏览器远程执行系统命令，管理服务器。

![](img/728df35f7218d0b065d59cd63b0bebd4_23.png)

### 1. Webshell示例

![](img/728df35f7218d0b065d59cd63b0bebd4_25.png)

假设系统已安装Apache和PHP模块。攻击者上传了一个简单的PHP Webshell文件（例如 `cmd.php`）到网站目录。

![](img/728df35f7218d0b065d59cd63b0bebd4_26.png)

![](img/728df35f7218d0b065d59cd63b0bebd4_27.png)

**访问Webshell：**
通过浏览器访问：`http://<目标IP>/cmd.php?cmd=id`
页面会显示执行 `id` 命令的结果，通常是Web服务进程的运行用户（如 `apache`）。

**危害：**
攻击者可以通过此Webshell执行更多命令，例如：
*   `cat /etc/passwd`：查看系统用户。
*   `ifconfig`：查看网络配置。
*   写入文件、添加用户等，进一步控制服务器。

![](img/728df35f7218d0b065d59cd63b0bebd4_29.png)

### 2. 复杂Webshell：C99 Shell

![](img/728df35f7218d0b065d59cd63b0bebd4_31.png)

`C99.php` 等是功能更强大的Webshell，提供图形化界面，可以轻松进行文件管理、命令执行、数据库操作等。

**检查与清除：**
*   **手动检查**：定期扫描Web目录，查找可疑的PHP文件。
*   **使用扫描工具**：例如使用专门的PHP Webshell扫描脚本。这些工具通常会检查文件中是否包含 `eval(`、`system(` 等危险函数。
*   **发现后处理**：一旦确认是非法的Webshell，应立即删除，并检查系统是否已被进一步入侵。

## 第三部分：Rootkit介绍与检查 🛡️

Rootkit是攻击者在获得root权限后植入的一套工具包，用于维持访问权限、隐藏自身痕迹、收集信息等。

![](img/728df35f7218d0b065d59cd63b0bebd4_37.png)

![](img/728df35f7218d0b065d59cd63b0bebd4_38.png)

### 1. Rootkit示例：SSH后门

攻击者可能替换系统的 `sshd` 服务程序，植入一个带有后门的版本。

![](img/728df35f7218d0b065d59cd63b0bebd4_40.png)

**后门功能演示：**
1.  **正常密码**：`123`
2.  **后门密码**：`123456` （无论root密码如何修改，此密码始终有效）
3.  **键盘记录**：所有通过SSH登录的账号密码都会被记录到隐藏文件（如 `/tmp/.ilog`）中。

攻击者使用后门密码可以随时登录，而管理员修改正常密码的行为会被记录，密码反而泄露。

![](img/728df35f7218d0b065d59cd63b0bebd4_42.png)

### 2. Rootkit检查工具

Rootkit擅长隐藏，常规命令（如 `ls`、`ps`）可能被其篡改而无法显示其进程或文件。

**推荐检查方法：文件完整性校验（AIDE）**
1.  **原理**：在系统干净（刚安装好、未上线时）时，使用AIDE工具为关键系统文件（如 `/usr/sbin/sshd`， `/etc/ssh/sshd_config`）生成“指纹”数据库。
2.  **定期检查**：系统运行后，定期使用AIDE对比当前文件与初始数据库的“指纹”。
3.  **报警**：如果发现关键文件被修改（MD5值变化、大小变化等），而管理员并未进行合法操作，则极可能被植入了Rootkit。
    ```bash
    aide --check
    ```

![](img/728df35f7218d0b065d59cd63b0bebd4_44.png)

**其他检查方式：**
*   **使用静态编译的工具**：从其他干净系统拷贝 `ls`、`ps` 等命令来检查，避免使用可能被篡改的系统命令。
*   **查看网络连接**：使用 `netstat -antp` 检查可疑的外联。
*   **检查系统日志**：查看 `/var/log/secure`、`/var/log/messages` 等，寻找异常登录记录。

## 第四部分：日志审计 📊

完善的日志记录和分析是事后追溯、定位安全事件的关键。

### 1. 增强历史命令记录

默认的 `history` 命令只记录命令本身。我们可以优化格式，记录执行时间和用户。
```bash
export HISTTIMEFORMAT="%F %T `whoami` "
```
将此命令加入 `/etc/profile` 可使配置永久生效。优化后，`history` 输出如下：
```
174  2020-02-19 23:31:30 root sudo -u#-1 passwd root
```
这清晰地显示了**时间、用户、命令**，便于审计。

### 2. 日志集中管理与分析

*   **集中存储**：使用 `rsync` 或日志收集代理（如 `Filebeat`）将各服务器的日志统一发送到一台日志服务器。
*   **统一分析**：使用 `ELK`（Elasticsearch, Logstash, Kibana）堆栈或 `Splunk` 等工具对海量日志进行检索、分析和可视化展示。
*   **两种分析模式**：
    *   **实时分析**：对安全级别高的操作进行实时监控和报警。
    *   **存量（历史）分析**：对长期积累的日志进行深度分析，挖掘潜在的攻击模式和规律。
