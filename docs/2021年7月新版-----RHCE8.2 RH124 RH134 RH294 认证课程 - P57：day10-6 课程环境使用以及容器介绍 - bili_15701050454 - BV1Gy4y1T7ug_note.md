使用 `rht-vmctl` 命令可以切换课程环境。**重要提示**：切换前需要确保所有虚拟机已关机。

切换课程环境的命令格式为：
```bash
rht-vmctl set course <课程代码>
```
例如，切换到RH134课程：
```bash
rht-vmctl set course RH134
```

### 环境重置操作

![](img/f2934d60591e8e0879a076bb40019b83_74.png)

![](img/f2934d60591e8e0879a076bb40019b83_75.png)

![](img/f2934d60591e8e0879a076bb40019b83_77.png)

为了保证练习环境的一致性，提供了两种重置虚拟机的命令。

![](img/f2934d60591e8e0879a076bb40019b83_79.png)

![](img/f2934d60591e8e0879a076bb40019b83_80.png)

![](img/f2934d60591e8e0879a076bb40019b83_82.png)

![](img/f2934d60591e8e0879a076bb40019b83_84.png)

**1. `reset` 命令**
此命令将虚拟机恢复到初始快照状态，速度较快。适用于日常练习后的清理。
```bash
rht-vmctl reset <虚拟机名称>
```

![](img/f2934d60591e8e0879a076bb40019b83_86.png)

![](img/f2934d60591e8e0879a076bb40019b83_88.png)

**2. `fullreset` 命令**
此命令会进行更彻底的重置（如重新加载虚拟硬盘），耗时较长。仅在环境出现底层问题或切换课程后建议使用。
```bash
rht-vmctl fullreset <虚拟机名称>
```

![](img/f2934d60591e8e0879a076bb40019b83_90.png)

**操作建议**：
*   非切换课程或环境无异常时，建议使用 `reset` 命令。
*   切换课程后，或环境出现异常时，使用 `fullreset` 命令。
*   重置时，建议按顺序操作：先 `classroom` 和 `bastion`，再 `workstation` 和 `utility`，最后是各个server节点。

![](img/f2934d60591e8e0879a076bb40019b83_92.png)

![](img/f2934d60591e8e0879a076bb40019b83_94.png)

所有虚拟机都配置了主机名解析，例如，可以直接通过 `ssh root@workstation` 命令连接到工作站。

### 什么是容器？

![](img/f2934d60591e8e0879a076bb40019b83_108.png)

![](img/f2934d60591e8e0879a076bb40019b83_110.png)

容器（Container）的灵感来源于生活中用于收纳、存储和运输物品的箱子、柜子等工具。在IT领域，容器技术将一个应用程序及其所有依赖项（库、配置文件、环境变量等）打包在一起，形成一个标准化、轻量级、可移植的运行环境。

可以这样理解：宿主机操作系统（内核）好比一艘货轮，每个容器就是一个标准化的集装箱。集装箱内封装了独立的应用程序，它们共享货轮（内核）的动力和基础设施，但彼此隔离，互不影响。

### 容器标准化与OCI

容器是一个通用概念，而 `Docker`、`Podman` 等是具体的实现工具。为了确保不同工具创建的容器能够互相操作和移植，业界成立了 **开放容器计划（OCI）**。

OCI发布了两大核心规范：
*   **运行时规范（Runtime Specification）**： 定义了容器的运行状态和生命周期。
*   **镜像规范（Image Specification）**： 定义了容器镜像的格式、构建和分发标准。

任何符合OCI规范的容器，都可以被不同的容器引擎（如Docker, Podman）运行。

### 容器运行时（Container Runtime）

容器运行时是真正负责运行容器的组件，它与操作系统内核紧密协作，为容器提供运行环境。主流的容器运行时包括：
*   **runc**： 目前最主流的OCI运行时，Docker和Podman默认使用它。
*   **lxc**： 早期的Linux容器运行时。
*   **rkt**： CoreOS项目开发的运行时。

### Podman 简介

`Podman`（Pod Manager）是一个开源的、无需守护进程的容器引擎。它的名字来源于Kubernetes中的最小调度单元 **Pod**。

与Docker相比，Podman的主要特点包括：
*   **无守护进程**： 直接通过`runc`启动容器，更轻量、更安全。
*   **兼容Docker CLI**： 大部分命令与Docker相同，学习成本低。
*   **Rootless容器**： 允许普通用户无需root权限即可运行容器，安全性更高。
*   **集成systemd**： 可以方便地将容器作为系统服务管理。

因此，Podman常被视为更现代、更安全的Docker替代品，特别是在与Kubernetes相关的生态中。

### 容器 vs. 传统虚拟机

为了更直观地理解容器的优势，我们将其与传统虚拟机进行对比：

| 特性 | 传统虚拟机 (VM) | 容器 (Container) |
| :--- | :--- | :--- |
| **虚拟化级别** | 硬件虚拟化 | 操作系统（内核）虚拟化 |
| **启动速度** | 慢（分钟级） | 快（秒级） |
| **性能损耗** | 较高（5%-20%） | 极低（近原生） |
| **磁盘占用** | 大（GB级） | 小（MB级） |
| **隔离性** | 强（完全隔离的OS） | 一般（共享内核，隔离进程、文件系统等） |
| **单机部署密度** | 低（通常十几个） | 高（可达数百个） |
| **镜像构建** | 复杂、笨重 | 简单、高效 |
| **与CI/CD集成** | 一般 | 优秀，易于实现自动化 |

**核心区别**：
*   **虚拟机** 虚拟出一套完整的硬件，并在其上运行一个完整的客户操作系统。
*   **容器** 直接共享宿主机的内核，只是通过命名空间和控制组（cgroups）等技术对进程、文件系统、网络等进行隔离和资源限制，因此更加轻量化。
