在红帽8系统中，存在多种启动目标，用于不同的运行状态。以下是常见的四种启动目标：
*   **图形界面**：提供完整的桌面环境。
*   **文本界面**：也称为多用户模式，仅提供命令行界面。
*   **救援模式**：用于修复系统问题，仅挂载根文件系统。
*   **紧急模式**：用于更严重的故障修复，仅挂载根文件系统为只读。

![](img/fd2afdcd7c525c1ab8725422a3728ba5_20.png)

### 查看与切换启动目标
我们可以使用 `systemctl` 命令来管理启动目标。

![](img/fd2afdcd7c525c1ab8725422a3728ba5_22.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_24.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_26.png)

*   **查看当前默认启动目标**：
    ```bash
    systemctl get-default
    ```
*   **临时切换到文本界面**：
    ```bash
    systemctl isolate multi-user.target
    ```
*   **设置默认启动目标为文本界面**：
    ```bash
    systemctl set-default multi-user.target
    ```

![](img/fd2afdcd7c525c1ab8725422a3728ba5_28.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_29.png)

**注意**：不要随意将默认目标设置为紧急模式，否则系统可能无法正常启动。

上一节我们介绍了如何切换启动目标，本节中我们来看看如果在救援或紧急模式下，如何将系统恢复至正常状态。

![](img/fd2afdcd7c525c1ab8725422a3728ba5_47.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_49.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_51.png)

如果不慎进入紧急模式，系统会提示输入root密码，并且根文件系统通常以**只读**方式挂载。此时无法直接修改系统配置。

![](img/fd2afdcd7c525c1ab8725422a3728ba5_55.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_57.png)

以下是恢复步骤：
1.  将根文件系统重新挂载为**读写**模式：
    ```bash
    mount -o remount,rw /
    ```
2.  此时，可以执行命令切换默认启动目标，例如切换回文本界面：
    ```bash
    systemctl set-default multi-user.target
    ```
3.  执行 `exit` 命令退出并重启系统。

![](img/fd2afdcd7c525c1ab8725422a3728ba5_59.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_61.png)

**核心概念**：在紧急模式下，必须先将根文件系统挂载为可写，才能进行有效的修复操作。

## 章节 10.3：重置root密码 🔑

上一节我们学习了在特殊模式下如何操作，本节中我们来看一个非常实用的场景——当忘记root密码时，如何重置它。

重置root密码需要进入系统的紧急救援模式（单用户模式）进行操作。以下是具体步骤：

1.  在系统启动的GRUB菜单界面，按 **`e`** 键编辑启动参数。
2.  找到以 `linux` 开头的行（在RHEL 7中可能是 `linux16`）。
3.  将光标移动到该行，找到 `ro`（只读）参数，将其修改为 `rw`（读写）。
4.  在同一行末尾，添加参数 `rd.break`。
5.  按 **`Ctrl+x`** 使用修改后的参数启动，系统将进入一个临时的紧急救援环境。
6.  执行以下命令切换到真实的根文件系统：
    ```bash
    chroot /sysroot
    ```
7.  现在可以重置root密码了。有两种方法：
    *   使用 `passwd` 命令交互式修改。
    *   使用 `echo` 命令非交互式修改（例如：`echo “redhat” | passwd --stdin root`）。
8.  **关键步骤**：由于在单用户模式下SELinux上下文可能不一致，必须重新打上文件系统标签：
    ```bash
    touch /.autorelabel
    ```
9.  执行两次 `exit` 命令退出chroot环境并重启系统。

**重要提示**：步骤8中的 `/.autorelabel` 命令至关重要，如果遗漏，系统可能因SELinux策略问题而无法启动。在考试或实验环境中，如果操作失误导致系统无法启动，应果断重置虚拟机并重新开始。

## 章节 10.4：修复启动故障 🔧

上一节我们解决了密码丢失的问题，本节中我们来看看如何诊断和修复其他常见的启动故障。

![](img/fd2afdcd7c525c1ab8725422a3728ba5_69.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_71.png)

![](img/fd2afdcd7c525c1ab8725422a3728ba5_72.png)

系统启动失败通常与文件系统配置有关。常见的故障原因包括：
*   `/etc/fstab` 文件配置错误。
*   挂载点不存在或UUID错误。
*   文件系统损坏。

![](img/fd2afdcd7c525c1ab8725422a3728ba5_74.png)

### 故障排查方法
1.  **查看启动日志**：使用 `journalctl` 命令查看从本次启动开始的日志，这是定位问题的首要步骤。
    ```bash
    journalctl -xb
    ```
2.  **检查文件系统挂载**：在救援模式下，使用 `mount -a` 命令测试 `/etc/fstab` 中的配置是否能正确挂载。这可以帮助发现错误的配置行。
3.  **临时注释错误配置**：如果发现 `/etc/fstab` 中有导致启动失败的配置行（例如一个无法访问的网络存储），可以在救援模式下将其注释掉（在行首添加 `#`），让系统先正常启动，然后再处理该挂载问题。

![](img/fd2afdcd7c525c1ab8725422a3728ba5_76.png)

**核心思路**：修复启动问题的关键是利用日志定位故障点，并在救援模式下进行最小化的修改，使系统能够先进入可操作状态。
