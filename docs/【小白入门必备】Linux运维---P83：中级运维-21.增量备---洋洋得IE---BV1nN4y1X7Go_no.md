# MySQL主从复制与高级架构：P83：中级运维-21. 增量备份，主从复制-下 🚀

在本节课中，我们将深入学习MySQL主从复制的高级架构——主从从（级联复制）结构。我们将探讨为何需要引入中继库，以及如何使用黑洞引擎来构建一个高效、低负载的复制链路。

## 概述

上一节我们介绍了基础的MySQL主从复制原理与配置。本节中，我们来看看当从库数量增多时，主库面临的读写压力问题，并学习如何通过引入中继库（使用黑洞引擎）来构建主从从架构，以分担主库的负载，提升整个复制链路的稳定性。

## 一主多从架构的压力问题

![](img/2337e2ca8f7cda3d8958c6602ee2639b_1.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_3.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_4.png)

在标准的一主多从架构中，主库每次提交数据修改，都需要向每一个从库发送同步数据。从库越多，主库的I/O和网络压力就越大。服务器压力过大（如CPU、内存、硬盘资源耗尽）是导致宕机或故障的常见原因之一。

![](img/2337e2ca8f7cda3d8958c6602ee2639b_6.png)

为了避免主库因压力过大而出问题，我们需要引入一个中继层。

## 中继库的作用与黑洞引擎

![](img/2337e2ca8f7cda3d8958c6602ee2639b_8.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_10.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_12.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_14.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_16.png)

中继库的核心作用是作为数据中转站，减轻主库的直接压力。对于主库而言，它只连接中继库这一个“从库”。中继库再将数据同步给后端的多个从库。

![](img/2337e2ca8f7cda3d8958c6602ee2639b_18.png)

以下是中继库配置的关键点，特别是使用黑洞引擎的原因：

1.  **减轻负载**：中继库如果使用常规存储引擎（如InnoDB），它自身在写入数据时也会产生较大的I/O和CPU压力，这违背了我们引入中继库分担压力的初衷。
2.  **黑洞引擎的优势**：黑洞引擎（`BLACKHOLE`）的特点是**接收数据但不实际存储**。当中继库使用黑洞引擎时，它只负责传递二进制日志，自身不进行实际的数据写入操作，这极大地降低了其硬盘I/O和资源消耗。
3.  **保证链路稳定**：一个低负载的中继库更不容易宕机，从而保证了整个主从复制链路的稳定性。

**核心配置公式**：
在中继库的MySQL配置文件（如 `my.cnf`）中，需要关键的三项设置：
```ini
server-id = 2               # 服务器唯一ID，不能与主从其他节点重复
log-bin = mysql-bin         # 必须开启二进制日志
relay-log = mysql-relay-bin # 开启中继日志
log-slave-updates = ON      # 允许将从主库接收的更新写入自己的二进制日志
default-storage-engine = BLACKHOLE # 将默认存储引擎设置为黑洞
```

## 主从从架构配置步骤

接下来，我们以三台服务器为例（IP：131-主库，129-中继库，128-从库），演示配置过程。

### 第一步：修改配置文件

**主库 (131)**：只需确保 `server-id` 唯一并开启 `log-bin`（基础主从配置已包含）。

**中继库 (129)**：配置文件需添加上述核心配置项，并重启MySQL服务。
```bash
systemctl restart mysqld
```

**从库 (128)**：只需配置唯一的 `server-id` 并重启服务。

### 第二步：数据初始化

![](img/2337e2ca8f7cda3d8958c6602ee2639b_20.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_22.png)

在启动复制前，确保主库、中继库、从库的**数据状态一致**。虽然复制进程可以正常启动，但旧数据不会自动同步。因此，需要手动或通过备份工具先将主库的数据同步到中继库和从库。

![](img/2337e2ca8f7cda3d8958c6602ee2639b_24.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_26.png)

### 第三步：建立复制链路

![](img/2337e2ca8f7cda3d8958c6602ee2639b_28.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_30.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_32.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_34.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_36.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_38.png)

1.  **主库 -> 中继库**：
    *   在主库为从库（129）授权复制权限。
    ```sql
    GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'192.168.0.129' IDENTIFIED BY 'password';
    ```
    *   在中继库上执行 `CHANGE MASTER TO` 命令，指向主库（131）的日志文件和位置，然后启动复制。
    ```sql
    START SLAVE;
    ```

![](img/2337e2ca8f7cda3d8958c6602ee2639b_40.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_42.png)

2.  **中继库 -> 从库**：
    *   在中继库为从库（128）授权复制权限。
    ```sql
    -- 简化版授权，授予所有权限（包含复制权限）
    GRANT ALL ON *.* TO 'root'@'192.168.0.%' IDENTIFIED BY 'password';
    ```
    *   在从库上执行 `CHANGE MASTER TO` 命令，指向中继库（129）的日志文件和位置，然后启动复制。
    ```sql
    START SLAVE;
    ```

![](img/2337e2ca8f7cda3d8958c6602ee2639b_44.png)

### 第四步：验证与测试

![](img/2337e2ca8f7cda3d8958c6602ee2639b_46.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_48.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_50.png)

*   使用 `SHOW SLAVE STATUS\G` 命令在所有从库节点检查复制状态，确保 `Slave_IO_Running` 和 `Slave_SQL_Running` 均为 `Yes`。
*   在主库插入测试数据，观察数据是否最终能同步到末端的从库（128）。
*   验证中继库（129）的黑洞特性：数据表结构存在，但表内无实际数据；然而，其二进制日志中已记录所有同步事件。

## 重要注意事项

![](img/2337e2ca8f7cda3d8958c6602ee2639b_52.png)

![](img/2337e2ca8f7cda3d8958c6602ee2639b_54.png)

*   **从库只读**：配置主从复制后，所有从库（包括中继库）**不应再直接写入业务数据**，否则可能导致数据冲突或同步混乱。
*   **日志传递链**：理解数据流向：`主库二进制日志` -> `中继库的中继日志` -> `中继库的二进制日志` -> `末端从库的中继日志`。
*   **中继库的特殊性**：中继库既是主库的从库，又是后端从库的主库。它自身写入的数据（虽然黑洞引擎不存储）会进入自己的二进制日志，并同步给后端从库，但这部分数据与主库无关，极易造成数据不一致，因此必须避免。

![](img/2337e2ca8f7cda3d8958c6602ee2639b_56.png)

## 拓展：双主复制简介

![](img/2337e2ca8f7cda3d8958c6602ee2639b_58.png)

双主复制（Master-Master）即两个数据库互为主从，均可读写。其配置本质是**相互执行一次主从复制配置**。

![](img/2337e2ca8f7cda3d8958c6602ee2639b_60.png)

**优点**：双方都可提供读写服务，可用性高。
**缺点**：数据冲突风险增加，配置和维护更复杂，性能压力较大。在实际生产中，其应用场景远少于读写分离架构下的主从复制。
