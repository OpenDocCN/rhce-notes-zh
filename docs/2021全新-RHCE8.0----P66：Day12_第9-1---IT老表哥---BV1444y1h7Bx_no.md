# Ansible 自动化运维：第12章：第9-11天知识回顾 📚

在本节课中，我们将回顾第9天到第11天学习的关键知识，涵盖Ansible的基础架构、配置、模块、剧本编写、变量与事实以及任务控制。通过本次回顾和练习，旨在巩固前五章的核心概念，为后续学习打下坚实基础。

## 概述：Ansible基础架构回顾 🏗️

首先，我们通过一个架构图来回顾Ansible的核心工作模式。控制节点作为“指挥部”，与受管主机建立连接后，通过剧本或角色下发指令，实现批量管理和部署的目标状态，这体现了Ansible的**幂等性**。

核心概念包括：
*   **控制节点**：执行Ansible命令的主机。
*   **受管主机**：被控制节点管理的主机。
*   **资产清单**：受管主机的列表。
*   **剧本**：定义要执行的任务列表。
*   **角色**：组织剧本和文件的一种方式。

## 第一部分：安装、初始化与配置 ⚙️

上一节我们介绍了Ansible的基础架构，本节中我们来看看如何部署和配置Ansible环境。

### Ansible部署步骤

以下是Ansible部署的基本步骤：

1.  **安装Ansible**：在有EPEL源和互联网连接的环境下，使用 `yum install ansible` 命令安装。
2.  **配置资产清单**：定义受管主机。资产清单文件支持多种格式：
    *   单台主机：直接写IP或主机名。
    *   主机分组：将多台主机归入一个组。
    *   嵌套分组：使用 `children` 关键字组合多个组。
    *   主机范围：使用范围（如 `node[1:3]`）简化定义。
    *   **注意**：IP地址和域名被视为两条不同的记录。
3.  **配置免密认证**：在控制节点和受管主机之间建立SSH密钥认证。
    *   步骤一：在控制节点生成密钥对：`ssh-keygen`。
    *   步骤二：将公钥复制到受管主机：`ssh-copy-id user@hostname`。
    *   **最佳实践**：在实验环境中，可能需要配置 `/etc/hosts` 文件进行主机名解析。
4.  **验证连接**：使用 `ansible all --list-hosts` 命令验证资产清单中的主机是否可被识别。

### Ansible配置文件

Ansible配置文件的优先级从高到低如下：

1.  环境变量 `ANSIBLE_CONFIG` 指定的文件。
2.  当前工作目录下的 `ansible.cfg` 文件。
3.  用户家目录下的 `.ansible.cfg` 文件。
4.  系统默认的 `/etc/ansible/ansible.cfg` 文件。

一个典型的 `ansible.cfg` 文件内容示例如下，它配置了使用普通用户连接并启用权限提升：
```ini
[defaults]
inventory = ./inventory
remote_user = student
private_key_file = /home/student/.ssh/id_rsa

![](img/e9e148d20df93849171bc620a53b2ff2_1.png)

![](img/e9e148d20df93849171bc620a53b2ff2_2.png)

[privilege_escalation]
become = True
become_method = sudo
become_user = root
become_ask_pass = False
```
**注意**：在受管主机上，需要对 `student` 用户配置 `sudo` 权限。

在完成环境配置后，我们可以开始使用Ansible执行任务。最直接的方式是使用临时命令。

### 临时命令格式

![](img/e9e148d20df93849171bc620a53b2ff2_8.png)

![](img/e9e148d20df93849171bc620a53b2ff2_10.png)

临时命令的基本格式为：
```bash
ansible <pattern> -m <module_name> -a “<module_arguments>” -i <inventory_file>
```
*   `pattern`：匹配资产清单中的主机或组。
*   `-m`：指定要使用的模块。
*   `-a`：传递给模块的参数。
*   `-i`：指定资产清单文件（如果未在配置文件中指定）。

![](img/e9e148d20df93849171bc620a53b2ff2_12.png)

![](img/e9e148d20df93849171bc620a53b2ff2_14.png)

### 常用模块介绍

![](img/e9e148d20df93849171bc620a53b2ff2_16.png)

以下是之前课程中介绍的一些核心模块：

![](img/e9e148d20df93849171bc620a53b2ff2_18.png)

*   **文档查询**：
    *   `ansible-doc <module_name>`：查看模块详细说明。
    *   `ansible-doc -s <module_name>`：查看模块使用示例。
    *   官方文档网站：`docs.ansible.com`。

*   **文件模块**：
    *   `copy`：从控制节点复制文件到受管主机，或直接在受管主机上生成文件内容。
    *   `file`：在受管主机上创建、删除目录、文件或链接，设置属性。
    *   `fetch`：从受管主机拉取文件到控制节点。
    *   `lineinfile`：确保文件中存在某行内容（增、改）。
    *   `synchronize`：使用 `rsync` 协议同步文件。

*   **软件包模块**：
    *   `yum_repository`：配置Yum仓库。
    *   `yum` 或 `dnf`：管理软件包（安装、删除、更新）。

*   **系统模块**：
    *   `user`/`group`：管理用户和组。
    *   `service`：管理系统服务（启动、停止、启用）。
    *   `firewalld`：管理防火墙规则。
    *   `command`/`shell`：在受管主机上执行命令。
        *   `command`：直接执行命令，不通过shell解释，因此无法使用管道 `|`、重定向 `>`、环境变量（如 `$HOME`）等shell特性。
        *   `shell`：通过shell执行命令，可以使用所有shell特性，但需注意潜在的安全风险（命令注入）。

*   **其他模块**：
    *   `uri`：与Web服务交互。

## 第三部分：剧本编写基础 📜

临时命令适合执行简单任务，复杂或重复的任务则需要编写剧本。

### 剧本结构

一个基本的剧本（Playbook）是一个YAML格式文件，结构如下：
```yaml
---
- name: Play名称
  hosts: target_hosts
  tasks:
    - name: 任务一名称
      module_name:
        module_parameter: value
    - name: 任务二名称
      module_name:
        module_parameter: value
```
*   `---`：YAML文件起始标记。
*   `name`：Play或任务的描述。
*   `hosts`：指定在哪些受管主机上执行。
*   `tasks`：定义要执行的任务列表。

### 剧本编写与执行要点

1.  **语法严谨**：YAML对缩进（通常为两个空格）非常敏感，错误的缩进会导致执行失败。
2.  **列表项**：使用短横线 `-` 加空格开头。
3.  **键值对**：冒号 `:` 后面必须有一个空格。
4.  **执行流程**：
    *   语法检查：`ansible-playbook --syntax-check playbook.yml`
    *   空运行（模拟执行）：`ansible-playbook -C playbook.yml`
    *   实际执行：`ansible-playbook playbook.yml`
    *   查看详细输出：`ansible-playbook -v playbook.yml` （`-vvv` 可显示更详细日志）

## 第四部分：变量与事实 🔢

剧本的灵活性很大程度上依赖于变量。变量允许我们参数化配置和行为。

![](img/e9e148d20df93849171bc620a53b2ff2_20.png)

### 变量定义与引用

![](img/e9e148d20df93849171bc620a53b2ff2_22.png)

*   **命名规则**：以字母开头，可包含字母、数字和下划线。
*   **定义位置与优先级（从高到低）**：
    1.  通过 `-e` 或 `--extra-vars` 在命令行直接传递的变量。
    2.  在Playbook中通过 `vars` 定义的变量。
    3.  在独立的变量文件（通过 `vars_files` 引入）中定义。
    4.  在资产清单文件（`host_vars/`, `group_vars/`）中为特定主机或组定义的变量。
*   **变量引用**：使用双大括号包裹变量名，如 `{{ variable_name }}`。为避免歧义，建议在剧本中始终使用引号包裹整个字符串。
*   **变量类型**：
    *   简单变量：`var_name: value`
    *   列表/数组：`list_var: [‘item1‘， ‘item2‘]`
    *   字典：`dict_var: {‘key1‘: ‘value1‘， ‘key2‘: ‘value2‘}`
*   **注册变量**：使用 `register` 关键字将一个任务的结果保存到变量中，供后续任务使用。
    ```yaml
    - name: 获取系统信息
      command: uname -r
      register: kernel_version
    - name: 打印内核版本
      debug:
        msg: “内核版本是 {{ kernel_version.stdout }}“
    ```
*   **加密变量**：使用 `ansible-vault` 加密敏感信息（如密码）。
    *   创建加密文件：`ansible-vault create secret.yml`
    *   编辑加密文件：`ansible-vault edit secret.yml`
    *   查看加密文件：`ansible-vault view secret.yml`
    *   运行使用加密变量的剧本：`ansible-playbook --ask-vault-pass playbook.yml`

### 事实

![](img/e9e148d20df93849171bc620a53b2ff2_24.png)

事实是Ansible从受管主机自动收集的系统信息。

*   **收集事实**：默认情况下，每个Play都会先执行 `gather_facts` 任务。可以设置 `gather_facts: no` 来禁用。
*   **使用事实**：事实本身就是变量。例如，主机IP地址可以通过 `{{ ansible_default_ipv4.address }}` 引用。
*   **引用方式**：
    *   点号表示法：`{{ ansible_facts[‘default_ipv4‘][‘address‘] }}`
    *   方括号表示法：`{{ ansible_facts[‘default_ipv4‘][‘address‘] }}` （如果键名包含点，必须使用此方式）
*   **常用事实**：`ansible_distribution`（系统发行版）， `ansible_memtotal_mb`（总内存）， `ansible_devices`（磁盘设备）等。
*   **自定义事实**：可以在受管主机的 `/etc/ansible/facts.d/` 目录下创建 `.fact` 文件来定义自定义事实。
*   **魔法变量**：由Ansible内部提供的特殊变量。
    *   `hostvars`：包含所有主机事实的字典。
    *   `groups`：包含资产清单中所有组的字典。
    *   `group_names`：当前主机所属的组列表。
    *   `inventory_hostname`：资产清单中定义的主机名。

## 第五部分：任务控制 ⚡

为了编写更智能、健壮的剧本，我们需要控制任务的执行流程。

### 循环

循环用于对一组数据项重复执行一个任务。

*   **简单列表循环**：
    ```yaml
    - name: 创建多个用户
      user:
        name: “{{ item }}“
        state: present
      loop:
        - alice
        - bob
        - charlie
    ```
    在循环中，当前项通过 `{{ item }}` 引用。

*   **字典列表循环**：
    ```yaml
    - name: 添加多个用户并指定UID
      user:
        name: “{{ item.name }}“
        uid: “{{ item.uid }}“
      loop:
        - { name: ‘alice‘， uid: 1001 }
        - { name: ‘bob‘， uid: 1002 }
    ```
    可以通过 `{{ item.name }}`， `{{ item.uid }}` 访问字典的键。

*   **对字典变量循环**：如果变量本身是一个字典，需要使用 `dict2items` 过滤器将其转换为适合循环的列表。
    ```yaml
    vars:
      users:
        alice: 1001
        bob: 1002
    tasks:
      - name: 使用字典变量创建用户
        user:
          name: “{{ item.key }}“
          uid: “{{ item.value }}“
        loop: “{{ users | dict2items }}“
    ```

### 条件判断

![](img/e9e148d20df93849171bc620a53b2ff2_26.png)

使用 `when` 语句根据条件决定是否执行任务。
```yaml
- name: 仅在CentOS系统上安装软件包
  yum:
    name: httpd
    state: present
  when: ansible_facts[‘distribution‘] == “CentOS“
```

![](img/e9e148d20df93849171bc620a53b2ff2_28.png)

条件判断常与 `register` 结合，根据前一个任务的结果来决定后续操作。
```yaml
- name: 检查服务状态
  command: systemctl is-active httpd
  register: httpd_status
  ignore_errors: yes # 忽略命令执行失败的错误

![](img/e9e148d20df93849171bc620a53b2ff2_30.png)

![](img/e9e148d20df93849171bc620a53b2ff2_32.png)

- name: 如果服务未运行则启动
  service:
    name: httpd
    state: started
  when: httpd_status.rc != 0 # 返回码不为0表示服务未运行
```

### 触发器与处理程序

![](img/e9e148d20df93849171bc620a53b2ff2_35.png)

![](img/e9e148d20df93849171bc620a53b2ff2_36.png)

![](img/e9e148d20df93849171bc620a53b2ff2_37.png)

处理程序是一种特殊的任务，只在被通知时执行一次，通常用于在配置文件变更后重启服务。

![](img/e9e148d20df93849171bc620a53b2ff2_39.png)

![](img/e9e148d20df93849171bc620a53b2ff2_41.png)

*   **定义处理程序**：
    ```yaml
    handlers:
      - name: restart nginx
        service:
          name: nginx
          state: restarted
    ```
*   **通知处理程序**：在任务中使用 `notify` 关键字。
    ```yaml
    tasks:
      - name: 更新nginx配置
        template:
          src: nginx.conf.j2
          dest: /etc/nginx/nginx.conf
        notify: restart nginx # 只有当模板文件内容真正改变时，才会触发通知
    ```
    **关键点**：`notify` 仅在任务状态为 **changed** 时触发。如果任务未做任何更改（幂等性），则不会通知。

![](img/e9e148d20df93849171bc620a53b2ff2_43.png)

![](img/e9e148d20df93849171bc620a53b2ff2_45.png)

### 错误处理

为了控制任务失败时的剧本行为，Ansible提供了多种机制。

以下流程图清晰地概括了Playbook的错误处理流程与相关参数的作用：

```
[开始执行Playbook]
        |
        v
[顺序执行任务] ---失败且未忽略---> [任务终止，Playbook失败]
        |                               ^
        | 失败但ignore_errors: yes      |
        v                               |
[忽略错误，继续执行下一个任务]          |
        |                               |
        v                               |
[任务成功或状态为changed]                |
        |                               |
        v                               |
[检查notify条件] ---状态changed---> [执行对应的handler]
        |           |                    |
        |   状态未changed                |
        v           |                    |
[继续下一个任务]    |                    |
        |           |                    |
        v           |                    |
[block/rescue/always流程]               |
        |                               |
        v                               |
[结束]                                 |
```

相关参数详解：

1.  **忽略错误**：`ignore_errors: yes`
    *   在任务或Play级别设置。
    *   即使任务失败，剧本也会继续执行后续任务。

2.  **强制处理程序执行**：`force_handlers: yes`
    *   在Play级别设置。
    *   即使Play中途失败，也会执行所有已被通知的handler。

![](img/e9e148d20df93849171bc620a53b2ff2_47.png)

3.  **自定义失败条件**：
    *   `failed_when`: 自定义任务失败的条件。例如，命令返回码为0时反而标记为失败。
    *   `changed_when`: 自定义任务状态是否为“changed”。可以用于精细控制 `notify` 的触发。

![](img/e9e148d20df93849171bc620a53b2ff2_49.png)

4.  **块错误处理**：`block`， `rescue`， `always`
    ```yaml
    - block:
        - name: 尝试执行某些操作
          debug:
            msg: “正在执行主要任务“
        - name: 一个会失败的任务
          command: /bin/false
      rescue:
        - name: 当block中任何任务失败时执行
          debug:
            msg: “捕获到错误，执行救援任务“
      always:
        - name: 无论block成功还是失败都执行
          debug:
            msg: “总是执行清理任务“
    ```
    *   `block`：定义要执行的主要任务块。
    *   `rescue`：当 `block` 中的任务失败时执行。
    *   `always`：无论 `block` 成功还是失败，最后都会执行。

3.  配置一个模板任务，更新 `mariadb` 的配置文件。当配置文件发生更改时，使用 `notify` 触发一个处理程序来重启 `mariadb` 服务。
