## 公钥与私钥的安全概念 🔐

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_11.png)

上一节我们介绍了 SSH 服务的基本连接。本节中，我们来看看 SSH 中更安全的身份验证方式——密钥对。

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_12.png)

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_14.png)

公钥的权限设置通常较为宽松，允许所有人读取。从权限角度看，这本身是安全的。然而，私钥必须严格保密。任何形式的暴露（例如被他人用手机拍摄到）都意味着私钥泄露。因此，私钥绝不能以任何形式展示给他人，包括在演示环境中。本次演示仅为实验目的，在生产环境中必须避免此类暴露。

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_16.png)

如果服务器系统重装，且未提前备份私钥，那么原有的私钥将永久丢失。密钥对是一次性的，系统重装后无法生成完全相同的密钥。

**核心概念**：SSH 使用非对称加密。客户端首次连接服务器时，会获取服务器的公钥用于后续验证。通信时，客户端使用服务器的公钥加密数据，服务器则使用自己的私钥解密。反之，服务器使用客户端的公钥加密，客户端用自己的私钥解密。公式可以简化为：
*   **加密**：`密文 = 加密函数(明文, 接收方公钥)`
*   **解密**：`明文 = 解密函数(密文, 接收方私钥)`

由于私钥不对外公开，因此第三方无法解密通信内容，从而实现了安全的加密传输。我们在 SSH 会话中执行的所有命令都经过此种加密。

## 配置基于密钥的身份验证 🛠️

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_20.png)

理解了密钥对的理论后，我们来进行一个有趣的实验：配置 SSH 服务器，使其验证客户端时不再使用密码，而是验证客户端的密钥。

以下是配置基于密钥身份验证的步骤：

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_22.png)

1.  **在客户端生成密钥对**
    在客户端执行命令 `ssh-keygen`。程序会提示密钥的保存路径（默认为用户家目录下的 `.ssh/id_rsa`），直接按回车接受默认值即可。当询问是否为私钥设置密码短语时，连续按两次回车表示不设置。命令执行成功后，会在 `~/.ssh/` 目录下生成两个文件：`id_rsa`（私钥）和 `id_rsa.pub`（公钥）。

    ```bash
    ssh-keygen
    ```

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_24.png)

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_25.png)

2.  **查看生成的密钥**
    可以使用 `cat` 命令查看密钥内容。私钥内容很长，以 `-----BEGIN RSA PRIVATE KEY-----` 开头。公钥内容较短，包含了算法（如 RSA）和密钥长度（如 2048 位）。较长的密钥长度使其比普通密码更难被暴力破解。

    ```bash
    cat ~/.ssh/id_rsa
    cat ~/.ssh/id_rsa.pub
    ```

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_27.png)

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_28.png)

3.  **将客户端公钥传输到服务器**
    为了让服务器识别客户端，需要将客户端的公钥复制到服务器对应用户的家目录中。使用 `ssh-copy-id` 命令可以方便地完成此操作。

    ```bash
    ssh-copy-id root@server_b
    ```
    执行此命令时，仍然需要输入一次服务器用户的密码以完成身份验证和传输。

4.  **验证密钥登录**
    公钥传输完成后，再次尝试登录服务器。你会发现，系统不再要求输入密码，而是直接登录成功。这是因为 SSH 默认优先尝试基于密钥的身份验证，这种方式比密码验证更安全。

    ```bash
    ssh root@server_b
    ```

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_30.png)

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_31.png)

**重要提示**：客户端的私钥文件（`~/.ssh/id_rsa`）是登录服务器的凭证，必须妥善保管。只有拥有它，才能解密服务器用对应公钥加密后传回的信息。

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_33.png)

## 增强服务器安全配置 🚀

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_35.png)

成功配置密钥登录后，我们可以进一步关闭服务器的密码验证功能，以大幅提升安全性。

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_37.png)

1.  **禁用密码验证**
    编辑 SSH 服务器的配置文件 `/etc/ssh/sshd_config`，找到 `PasswordAuthentication` 配置项，将其值从 `yes` 改为 `no`。

    ```bash
    vi /etc/ssh/sshd_config
    # 找到并修改
    PasswordAuthentication no
    ```

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_38.png)

2.  **重启 SSH 服务**
    修改配置后，需要重启 SSH 守护进程（`sshd`）使新配置生效。服务名后的 `d` 通常代表 `daemon`（守护进程）。

    ```bash
    systemctl restart sshd.service
    ```

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_40.png)

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_41.png)

3.  **验证配置效果**
    配置生效后，任何尝试使用密码登录服务器的行为都将被拒绝。例如，即使用户“张三”拥有正确的密码，也无法登录。这有效防御了针对密码的暴力破解攻击。只有拥有对应私钥的用户才能成功登录。

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_43.png)

## 管理用户登录权限 👥

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_47.png)

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_49.png)

除了禁用密码，我们还可以通过配置限制特定用户的登录，例如禁止 root 用户直接登录。

1.  **恢复密码验证（临时）**
    为了演示，我们先将 `PasswordAuthentication` 改回 `yes`，以便其他用户（如“张三”）暂时能用密码登录。

2.  **禁止 root 用户直接登录**
    在同一配置文件 `/etc/ssh/sshd_config` 中，找到 `PermitRootLogin` 配置项，将其值改为 `no`。

    ```bash
    PermitRootLogin no
    ```

3.  **重启服务并测试**
    重启 SSH 服务后，尝试用 root 账户登录，即使输入正确密码也会被拒绝。这增加了系统安全性，因为攻击者无法直接以最高权限账户进行尝试。

    ```bash
    systemctl restart sshd.service
    ssh root@server_b # 将被拒绝
    ```

4.  **使用普通用户登录并切换**
    此时，普通用户“张三”仍可通过密码或密钥（如果已配置）登录。登录后，如果需要执行管理员操作，可以使用 `su -` 或 `sudo` 命令切换到 root 用户。这种方式比直接允许 root 登录更安全。

    ```bash
    ssh zhangsan@server_b # 可以登录
    su - # 或使用 sudo
    ```

## 总结
本节课我们一起学习了 SSH 服务的密钥认证与高级配置。我们首先理解了公钥和私钥在 SSH 加密通信中的核心作用。接着，我们实践了生成密钥对、配置基于密钥的身份验证，并成功实现了无密码登录。为了进一步提升安全性，我们学习了如何禁用服务器的密码验证功能，以及如何限制 root 用户的直接登录。

![](img/996a4eb07fdfbbd48af10a7612e7d9b1_51.png)

通过这些操作，我们不仅掌握了 SSH 服务的一项关键安全特性，也熟悉了 Linux 服务配置的通用模式：**修改配置文件 -> 重启服务使生效**。这种模式将在后续学习其他服务时反复出现。密钥认证机制极大地增强了远程管理的安全性，是系统管理员必须掌握的核心技能之一。
