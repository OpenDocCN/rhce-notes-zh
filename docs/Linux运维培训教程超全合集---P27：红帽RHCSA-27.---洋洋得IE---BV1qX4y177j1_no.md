## 概述
在本节课中，我们将学习逻辑卷的扩容操作，并初步了解RAID磁盘阵列的基本概念。逻辑卷管理（LVM）的核心优势在于其动态扩展能力，而RAID技术则用于提升数据的安全性和存储性能。我们将从实际操作出发，理解这些技术在企业环境中的应用。

![](img/97187198618a0925dd053326887ff88b_15.png)

![](img/97187198618a0925dd053326887ff88b_17.png)

### 扩容前的准备
在对逻辑卷进行扩容前，需要先确认卷组中是否有足够的剩余空间可供分配。

以下是检查卷组和逻辑卷状态的命令：
```bash
vgs
lvs
```
执行上述命令后，可以查看卷组的剩余空间和逻辑卷的当前大小。

### 扩展逻辑卷空间
确认卷组有足够空间后，即可对逻辑卷进行扩容。扩容命令使用 `lvresize` 或 `lvextend`。

扩展逻辑卷的命令格式如下：
```bash
lvextend -L +[要增加的大小] [逻辑卷的完整路径]
```
例如，为逻辑卷 `/dev/system_vg/my_lv` 增加 19GB 空间：
```bash
lvextend -L +19G /dev/system_vg/my_lv
```
命令执行后，逻辑卷的物理空间会立即增大。

### 扩展文件系统
逻辑卷空间扩展后，其内部的文件系统并不会自动增大。必须手动扩展文件系统，操作系统才能识别并使用新增的空间。

以下是扩展不同文件系统的命令：
*   对于 XFS 文件系统，使用 `xfs_growfs` 命令。
*   对于 ext2/3/4 文件系统，使用 `resize2fs` 命令。

例如，扩展挂载在 `/web_back` 目录下的 XFS 文件系统：
```bash
xfs_growfs /web_back
```
执行此命令后，使用 `df -h` 查看，会发现逻辑卷的可用空间已真正变大。

![](img/97187198618a0925dd053326887ff88b_19.png)

![](img/97187198618a0925dd053326887ff88b_21.png)

### 扩容卷组
如果卷组空间不足，需要先对卷组进行扩容。可以将新的物理磁盘或分区加入现有卷组。

![](img/97187198618a0925dd053326887ff88b_23.png)

扩展卷组的命令格式如下：
```bash
vgextend [卷组名] [新设备路径]
```
例如，将新分区 `/dev/sdb5` 和 `/dev/sdb6` 加入 `system_vg` 卷组：
```bash
vgextend system_vg /dev/sdb5 /dev/sdb6
```
卷组扩容后，便可以继续为逻辑卷分配更多空间。

## 逻辑卷的缩减与删除

![](img/97187198618a0925dd053326887ff88b_29.png)

### 缩减逻辑卷
逻辑卷支持缩减，但操作风险较高，且**不支持在线操作**。缩减前必须先卸载文件系统，其顺序与扩容相反：先缩小文件系统，再缩小逻辑卷空间。企业环境中极少进行缩容操作。

### 删除逻辑卷
当不再需要逻辑卷时，可以按顺序删除。删除同样**不支持在线操作**。

![](img/97187198618a0925dd053326887ff88b_31.png)

![](img/97187198618a0925dd053326887ff88b_33.png)

![](img/97187198618a0925dd053326887ff88b_35.png)

![](img/97187198618a0925dd053326887ff88b_37.png)

以下是删除逻辑卷的步骤：
1.  卸载逻辑卷的挂载点。
2.  删除逻辑卷。
3.  删除卷组（如果需要）。
4.  如需彻底清除，可以删除物理卷属性或直接格式化设备。

![](img/97187198618a0925dd053326887ff88b_39.png)

删除逻辑卷的命令示例：
```bash
# 1. 卸载
umount /web_back

# 2. 删除逻辑卷
lvremove /dev/system_vg/my_lv

# 3. 删除卷组
vgremove system_vg

![](img/97187198618a0925dd053326887ff88b_41.png)

# 4. 将物理设备恢复为普通分区（例如格式化）
mkfs.xfs -f /dev/sdb2
```
**注意：删除操作会清除该逻辑卷上的所有数据，且不可恢复。**

## RAID磁盘阵列简介

了解了逻辑卷的动态扩展能力后，我们还需要关注数据存储的可靠性。本节中，我们来看看RAID技术如何提升数据安全性和性能。

RAID（Redundant Array of Independent Disks），即独立磁盘冗余阵列，其核心思想是将多块独立的物理磁盘通过特定方式组合起来，形成一个统一的逻辑磁盘。

### RAID的主要优势
*   **提升性能**：通过并行读写，提高数据访问速度。
*   **增加容量**：将多块小容量磁盘组合成大容量存储池。
*   **增强可靠性**：通过数据冗余，在部分磁盘损坏时保障数据不丢失。

![](img/97187198618a0925dd053326887ff88b_49.png)

![](img/97187198618a0925dd053326887ff88b_51.png)

### RAID常见级别
以下是几种常见的RAID级别及其简要特点：

![](img/97187198618a0925dd053326887ff88b_53.png)

![](img/97187198618a0925dd053326887ff88b_55.png)

![](img/97187198618a0925dd053326887ff88b_57.png)

*   **RAID 0（条带化）**
    *   **原理**：将数据分块并交替写入多块磁盘。
    *   **优点**：读写速度最快，容量利用率100%（总容量为所有磁盘之和）。
    *   **缺点**：无冗余，任何一块磁盘损坏将导致所有数据丢失。
    *   **应用场景**：对性能要求极高、但对数据安全性要求不高的场景，如视频缓存。

![](img/97187198618a0925dd053326887ff88b_58.png)

![](img/97187198618a0925dd053326887ff88b_60.png)

*   **RAID 1（镜像）**
    *   **原理**：将相同的数据同时写入两块或多块磁盘。
    *   **优点**：数据安全性高，读取速度有所提升。
    *   **缺点**：容量利用率低（仅50%），写入速度无提升。
    *   **应用场景**：对数据安全性要求极高的场景，如操作系统盘、数据库日志盘。

![](img/97187198618a0925dd053326887ff88b_62.png)

![](img/97187198618a0925dd053326887ff88b_64.png)

![](img/97187198618a0925dd053326887ff88b_66.png)

*   **RAID 5（带奇偶校验的条带化）**
    *   **原理**：将数据与校验信息交叉存储在所有磁盘上。
    *   **优点**：兼顾性能、容量和安全性。允许损坏一块磁盘而不丢失数据。
    *   **缺点**：写入速度受校验计算影响。重建数据时系统压力大。
    *   **应用场景**：通用型存储，平衡了性能、安全与成本。

*   **RAID 10（RAID 1+0，先镜像再条带）**
    *   **原理**：先两两组成RAID 1镜像对，再将多个镜像对组成RAID 0。
    *   **优点**：兼具RAID 1的高安全性和RAID 0的高性能。
    *   **缺点**：成本最高，容量利用率仅50%。
    *   **应用场景**：对性能和安全性都有极高要求的场景，如核心数据库、金融交易系统。

### RAID的实现方式
*   **硬件RAID**：通过专用的RAID控制卡实现，性能好，不占用主机资源，但成本高。
*   **软件RAID**：通过操作系统内核或软件实现，成本低，配置灵活，但会消耗一定的CPU和内存资源。在Linux中，可以使用 `mdadm` 工具创建和管理软件RAID。

在企业实际应用中，常将RAID与LVM结合使用：先使用RAID保障底层物理磁盘的可靠性与性能，再在RAID虚拟出的“大硬盘”上创建逻辑卷，以获得动态扩展的灵活性。

## 总结
本节课中我们一起学习了两个核心内容：
1.  **逻辑卷扩容**：掌握了如何检查空间、扩展逻辑卷容量以及扩展文件系统的完整流程。理解了LVM动态扩展的优势在于无需预估初始空间，可根据需求灵活扩容。
2.  **RAID磁盘阵列**：了解了RAID技术的基本概念、主要优势、常见级别（如RAID 0, 1, 5, 10）及其应用场景。认识到RAID通过冗余和并行操作来提升数据安全性和存储性能，是构建可靠存储系统的重要基础。

![](img/97187198618a0925dd053326887ff88b_68.png)

逻辑卷解决了空间灵活管理的问题，而RAID则解决了底层存储的可靠性与性能问题。在实际生产环境中，二者常常协同工作，共同构建高效、可靠、易管理的存储架构。
