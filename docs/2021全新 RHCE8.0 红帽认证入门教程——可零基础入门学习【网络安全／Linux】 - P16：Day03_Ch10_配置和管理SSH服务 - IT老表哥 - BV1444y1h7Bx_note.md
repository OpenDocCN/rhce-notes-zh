# RHCE8.0 红帽认证入门教程：Day03：配置和管理SSH服务 🔐

在本节课中，我们将要学习如何配置和管理SSH服务。SSH是远程登录和管理Linux系统的重要工具，比Telnet等明文协议更加安全。我们将重点探讨SSH的加密原理、如何生成密钥对、配置免密认证，以及一些常见的安全配置。

## 概述：SSH与加密原理

![](img/3d0952a3ae84bb86c46b11a00b277f3d_1.png)

上一节我们介绍了进程管理和系统服务。本节中我们来看看如何安全地进行远程连接。

SSH是一种加密的远程登录协议。与Telnet的明文传输不同，SSH通过加密来保证通信安全。加密方式主要有两种：对称加密和非对称加密。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_3.png)

*   **对称加密**：加密和解密使用**同一个密钥**。特点是速度快，但安全性相对较低。常见的算法有DES、AES。
*   **非对称加密**：使用一对密钥，即**公钥**和**私钥**。公钥可以公开，用于加密和认证；私钥必须自己秘密保存，用于解密。这种方式更安全，但速度较慢。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_5.png)

SSH连接建立的过程，就是基于非对称加密的“握手”过程。当客户端A连接服务器B时：
1.  B将自己的公钥发送给A。
2.  A使用B的公钥加密一个随机值（或密码）并发送给B。
3.  B使用自己的私钥解密该信息。
4.  验证成功后，双方建立加密连接。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_7.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_9.png)

首次连接成功后，客户端会将服务器的公钥信息保存在本地 `~/.ssh/known_hosts` 文件中。下次连接同一台服务器时，即可跳过公钥交换的步骤，直接进入身份验证。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_10.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_11.png)

## 生成SSH密钥对

理解了加密原理后，我们来看看如何创建自己的密钥对。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_13.png)

在Linux系统中，可以使用 `ssh-keygen` 命令生成密钥对。以下是生成一对RSA密钥的示例：

```bash
ssh-keygen -t rsa
```

![](img/3d0952a3ae84bb86c46b11a00b277f3d_15.png)

执行命令后，系统会提示你选择密钥的保存路径（默认为 `~/.ssh/id_rsa`）和设置密钥的密码短语（可直接回车留空）。生成后，你会在 `~/.ssh/` 目录下得到两个文件：
*   `id_rsa`：**私钥文件**，必须严格保密。
*   `id_rsa.pub`：**公钥文件**，可以分发给需要连接的服务器。

## 配置基于密钥的免密认证

现在，我们来实践如何配置免密登录，这是自动化运维和Ansible等工具的基础。

假设我们有两台机器：`serverA`（客户端）和 `serverB`（服务器）。目标是让 `serverA` 的root用户能免密SSH登录到 `serverB` 的root用户。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_17.png)

以下是实现步骤：

![](img/3d0952a3ae84bb86c46b11a00b277f3d_19.png)

1.  **在 `serverA` 上生成密钥对**（如果尚未生成）：
    ```bash
    ssh-keygen -t rsa
    ```

![](img/3d0952a3ae84bb86c46b11a00b277f3d_21.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_23.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_25.png)

2.  **将 `serverA` 的公钥复制到 `serverB`**：
    使用 `ssh-copy-id` 命令可以方便地完成这个操作。
    ```bash
    ssh-copy-id root@serverB_IP地址
    ```
    执行后，需要输入 `serverB` 上root用户的密码。该命令会将 `serverA` 的公钥内容自动追加到 `serverB` 上对应用户（这里是root）的 `~/.ssh/authorized_keys` 文件中。

3.  **测试免密登录**：
    完成以上步骤后，从 `serverA` 再次SSH连接 `serverB`，就应该不需要输入密码了。
    ```bash
    ssh root@serverB_IP地址
    ```

![](img/3d0952a3ae84bb86c46b11a00b277f3d_27.png)

**核心概念**：免密认证的本质是服务器 `serverB` 信任了来自 `serverA` 的特定公钥。只要 `serverA` 能提供对应的私钥进行签名验证，`serverB` 就允许其登录，无需密码。这种单向信任关系常用于堡垒机、自动化运维等场景。

若要实现双向免密或为其他用户配置，需要对每个方向、每个用户重复上述步骤。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_29.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_36.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_37.png)

*   **远程执行单条命令**：在命令末尾直接跟上要在远程服务器上执行的命令，执行完毕后会立刻返回本地终端。
    ```bash
    ssh user@hostname “要执行的命令”
    ```
*   **指定密钥文件连接**：使用 `-i` 选项可以指定使用非默认路径的私钥文件进行连接。
    ```bash
    ssh -i /path/to/private_key user@hostname
    ```

## 配置SSH服务端以增强安全

![](img/3d0952a3ae84bb86c46b11a00b277f3d_41.png)

作为系统管理员，对SSH服务端进行适当配置可以显著提升安全性。主要配置文件是 `/etc/ssh/sshd_config`。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_43.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_44.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_45.png)

以下是两个常见的安全配置项：

![](img/3d0952a3ae84bb86c46b11a00b277f3d_47.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_48.png)

1.  **禁止root用户直接登录**：
    编辑 `/etc/ssh/sshd_config` 文件，找到 `PermitRootLogin` 配置项，将其值改为 `no`。
    ```
    PermitRootLogin no
    ```
    修改后，建议先使用普通用户登录，再通过 `su` 或 `sudo` 切换为root用户。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_50.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_51.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_52.png)

2.  **修改默认SSH端口**：
    同样在 `/etc/ssh/sshd_config` 文件中，找到 `Port` 配置项，去掉行首的注释符 `#`，并将端口号 `22` 改为一个未被系统其他服务占用的高端口号（如 `2222`）。
    ```
    Port 2222
    ```
    **注意**：修改端口后，客户端连接时必须使用 `-p` 选项指定新端口，例如 `ssh -p 2222 user@hostname`。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_54.png)

![](img/3d0952a3ae84bb86c46b11a00b277f3d_56.png)

**重要提示**：每次修改 `sshd_config` 文件后，必须重启SSH服务才能使配置生效。
```bash
systemctl restart sshd
```
在重启前，建议保持一个当前的SSH连接会话，并在新窗口中测试新配置是否生效，以防配置错误导致无法远程连接。

![](img/3d0952a3ae84bb86c46b11a00b277f3d_58.png)
