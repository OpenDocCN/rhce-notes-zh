### 🔧 `sefcontext` 模块：设置安全上下文

![](img/82e0710f04b5a9536365d4ebb66ff83f_29.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_30.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_31.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_33.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_35.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_36.png)

`sefcontext`模块用于设置文件或目录的SELinux安全上下文。虽然我们可以在`copy`或`file`模块中通过`seuser`、`segroup`等参数设置上下文，但那是临时生效的。要设置永久的安全上下文策略，需要使用`sefcontext`模块。

![](img/82e0710f04b5a9536365d4ebb66ff83f_38.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_39.png)

**核心概念公式：**
```
模块名: sefcontext
关键参数:
  - target: 目标文件路径
  - setype: 要设置的安全上下文类型
```

![](img/82e0710f04b5a9536365d4ebb66ff83f_41.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_42.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_43.png)

例如，将一个网页文件的安全上下文设置为Web服务器默认类型：
```yaml
- name: 设置index.html的SELinux安全上下文
  sefcontext:
    target: /var/www/html/index.html
    setype: httpd_sys_content_t
```
运行此任务后，文件的默认安全上下文将被永久设置为`httpd_sys_content_t`，使其能够被Web服务器正常访问。

![](img/82e0710f04b5a9536365d4ebb66ff83f_44.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_45.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_46.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_48.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_49.png)

> **注意**：如果被管理节点是最小化安装且启用了SELinux，可能需要先安装`python3-libsemanage`包，`sefcontext`模块才能正常工作。

### 📝 `blockinfile` 模块：以文本块形式管理文件

![](img/82e0710f04b5a9536365d4ebb66ff83f_61.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_62.png)

我们之前学过`lineinfile`模块可以按行修改文件。而`blockinfile`模块则以**文本块**的形式来插入、修改或标记文件中的内容。它非常适合用于添加多行配置片段。

![](img/82e0710f04b5a9536365d4ebb66ff83f_64.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_65.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_67.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_68.png)

以下是`blockinfile`模块的基本用法：
```yaml
- name: 在配置文件中插入一个文本块
  blockinfile:
    path: /etc/ssh/sshd_config  # 目标文件路径
    block: |  # “|”符号表示后面是一个多行文本块
      # 这是一个由Ansible管理的配置块
      PermitRootLogin no
      PasswordAuthentication yes
    marker: "# {mark} ANSIBLE MANAGED BLOCK"  # 可选：自定义标记
```
执行后，它会在指定文件的末尾（或指定位置）插入这个文本块，并在块的首尾自动添加类似`# BEGIN ANSIBLE MANAGED BLOCK`和`# END ANSIBLE MANAGED BLOCK`的注释标记（默认），这些标记有助于Ansible后续识别和管理这个块。

![](img/82e0710f04b5a9536365d4ebb66ff83f_71.png)

### 📊 `stat` 模块：检索文件状态信息

![](img/82e0710f04b5a9536365d4ebb66ff83f_76.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_78.png)

`stat`模块用于检索文件的详细状态信息（类似Linux的`stat`命令）。它不会修改文件，而是返回一个包含文件各种属性的字典（哈希），我们可以将这些信息注册为变量，并在后续任务中引用。

**核心概念代码：**
```yaml
- name: 获取/etc/passwd文件的MD5校验和
  stat:
    path: /etc/passwd
    checksum_algorithm: md5
  register: file_stat  # 将返回信息注册到变量file_stat

![](img/82e0710f04b5a9536365d4ebb66ff83f_80.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_82.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_84.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_85.png)

- name: 输出文件的MD5校验和
  debug:
    msg: "文件的MD5校验和是: {{ file_stat.stat.checksum }}"
```
`stat`模块返回的信息非常丰富，包括文件大小(`size`)、修改时间(`mtime`)、权限(`mode`)、所属用户/组(`pw_name`, `gr_name`)等。通过`debug`模块，我们可以查看所有信息：
```yaml
- name: 显示文件的所有状态信息
  debug:
    var: file_stat.stat
```

Jinja2模板基于Python，使用特定的分隔符来嵌入变量和逻辑。

![](img/82e0710f04b5a9536365d4ebb66ff83f_119.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_121.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_123.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_125.png)

以下是核心的分隔符：
*   **`{{ ... }}`**：用于**输出表达式或变量的结果**。在呈现模板时，这部分会被替换为实际的变量值。
    *   例如：`Listen {{ ansible_default_ipv4.address }}`
*   **`{% ... %}`**：用于**执行控制语句**，如循环和条件判断，不直接输出内容。
    *   例如：`{% for user in users %} ... {% endfor %}`
*   **`{# ... #}`**：用于添加**注释**，注释内容不会出现在最终生成的文件中。

让我们通过一个实际例子来理解模板的工作流程：为不同的Web服务器（节点）部署Apache配置文件，并确保每台服务器监听其自身的IP地址。

![](img/82e0710f04b5a9536365d4ebb66ff83f_146.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_148.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_150.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_152.png)

**步骤概述：**
1.  **准备模板**：从一个干净的Apache配置文件中提取有效配置项，移除所有注释。
2.  **修改模板**：在模板中，将需要动态化的部分（如监听地址）替换为Jinja2变量表达式。
3.  **获取配置**：从目标服务器上拉取当前的配置文件（如果需要基于现有文件修改）。
4.  **部署模板**：使用`template`模块将模板文件复制到目标服务器，Ansible会自动将变量替换为对应主机的实际值。

![](img/82e0710f04b5a9536365d4ebb66ff83f_154.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_156.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_158.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_160.png)

**具体剧本分析 (`template-demo.yml`):**
```yaml
---
- name: 部署自定义HTTPD配置模板
  hosts: node2  # 目标主机
  tasks:
    # 任务1：从目标节点获取原始配置文件到控制机
    - name: 从node2获取httpd.conf
      fetch:
        src: /etc/httpd/conf/httpd.conf
        dest: /tmp/httpd.conf
        flat: yes

    # 任务2：清理原始文件，创建模板（移除注释行）
    - name: 创建干净的配置模板
      lineinfile:
        path: /tmp/httpd.conf.j2  # 模板文件，通常以.j2结尾
        line: "{{ item }}"
      loop: "{{ lookup('file', '/tmp/httpd.conf').split('\n') | select('match', '^[^#]') | list }}"
      # 使用lookup读取文件，过滤掉以#开头的行，循环写入新文件

    # 任务3：在模板中动态化监听地址
    - name: 修改模板中的监听地址
      lineinfile:
        path: /tmp/httpd.conf.j2
        regexp: '^Listen'
        line: "Listen {{ ansible_default_ipv4.address }}:80"
        # 将Listen行替换为包含主机IP地址的Jinja2变量

    # 任务4：使用template模块部署模板
    - name: 应用模板到目标主机
      template:
        src: /tmp/httpd.conf.j2
        dest: /etc/httpd/conf/httpd.conf
      notify: restart httpd  # 配置文件改变后，触发重启Apache的处理程序

  handlers:
    - name: restart httpd
      service:
        name: httpd
        state: restarted
```
在这个例子中，关键的一步是使用`template`模块。它会读取`/tmp/httpd.conf.j2`模板文件，将里面的`{{ ansible_default_ipv4.address }}`替换为`node2`主机实际的IPv4地址，然后将最终内容写入`node2`的`/etc/httpd/conf/httpd.conf`。

**语法格式：**
```
{% for item in list %}
    内容 {{ item }}
{% endfor %}
```

![](img/82e0710f04b5a9536365d4ebb66ff83f_204.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_205.png)

**示例：批量输出用户列表**
```yaml
# 剧本变量
vars:
  users:
    - user1
    - user2

![](img/82e0710f04b5a9536365d4ebb66ff83f_207.png)

![](img/82e0710f04b5a9536365d4ebb66ff83f_208.png)

# 模板内容 (users.j2)
{% for user in users %}
用户名: {{ user }}
{% endfor %}
```
使用`template`模块部署后，生成的文件内容将是：
```
用户名: user1
用户名: user2
```

**更实用的示例：动态生成`/etc/hosts`片段**
假设我们有一个主机组`servers`，包含`node1`和`node2`。我们可以用循环为组内所有主机生成hosts记录。
```jinja2
# 模板内容 (hosts.j2)
{% for host in groups['servers'] %}
{{ hostvars[host].ansible_default_ipv4.address }} {{ hostvars[host].ansible_fqdn }} {{ host }}
{% endfor %}
```
这个模板会循环`servers`组中的每个主机名(`host`)，并通过`hostvars`字典获取该主机的IP地址和完全限定域名，输出成一行。

```jinja2
AdminNode yes
{% endif %}
```
只有当主机的IP地址是`192.168.1.100`时，`AdminNode yes`这行配置才会被添加到生成的文件中。

**目标**：生成一个模板，为`servers`主机组中的所有主机创建一个`/etc/myhosts`文件。
**要求**：
1.  文件格式与`/etc/hosts`类似。
2.  文件内容应为每台被管理主机添加一行记录，包含其**IP地址**、**完整主机名(FQDN)** 和**短主机名**。
3.  主机记录的显示顺序不重要。

**提示**：
*   你需要使用`groups[‘servers’]`来循环主机组。
*   使用`hostvars[host]`来获取特定主机的变量（如`ansible_default_ipv4.address`和`ansible_fqdn`）。
*   在模板中使用`{% for ... %}`循环。

**参考模板 (`myhosts.j2`)**:
```jinja2
# 由Ansible管理的hosts文件
{% for host in groups['servers'] %}
{{ hostvars[host].ansible_default_ipv4.address }} {{ hostvars[host].ansible_fqdn }} {{ host }}
{% endfor %}
```

## ✅ 本章总结

在本章中，我们一起学习了：
1.  三个高级文件管理模块：
    *   `sefcontext`：永久设置SELinux安全上下文。
    *   `blockinfile`：以文本块的形式插入或修改文件内容。
    *   `stat`：检索文件的详细状态信息并注册为变量。
2.  **Jinja2模板引擎**的核心用法，这是Ansible自动化配置的关键技术：
    *   理解了`{{ }}`（输出）、`{% %}`（逻辑控制）、`{# #}`（注释）三种分隔符。
    *   掌握了使用`template`模块部署模板，实现配置文件的动态生成。
    *   学会了在模板中使用`{% for %}`循环和`{% if %}`条件判断来构建复杂的配置文件。

通过掌握Jinja2模板，你可以极大地提升Ansible playbook的灵活性和可维护性，轻松应对为不同环境、不同主机生成个性化配置的需求。
