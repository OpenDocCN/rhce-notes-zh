# MySQL数据库管理：P77：事务，锁，备份

![](img/829f8d9bda8a36932a439be41e119c7b_1.png)

在本节课中，我们将要学习MySQL中的事务控制语言，这是SQL语句的最后一种类型。我们将了解事务的概念、作用以及如何使用它来确保数据操作的安全性和一致性。同时，我们也会初步接触与事务密切相关的“锁”的概念。

## 回顾SQL语句类型

上一节我们介绍了SQL语句的前三种类型，本节中我们来看看第四种。首先，让我们回顾一下之前学过的内容。

SQL语句主要分为四种类型：
*   **数据定义语言**：用于管理数据库和表格结构，包括创建、修改和删除。
*   **数据操作语言**：针对表格中的数据，进行增、删、改、查操作。
*   **数据控制语言**：用于管理用户权限，主要包括授予和撤销权限。
*   **事务控制语言**：这是我们本节课要学习的内容，用于控制一组数据库操作。

## 什么是事务？

事务是数据库操作的一个逻辑单元，它包含一组SQL语句。事务的主要作用是确保这组操作要么全部成功，要么全部失败，从而维护数据的完整性。

在没有使用事务的情况下，我们执行如`INSERT`、`UPDATE`、`DELETE`等命令时，修改会直接应用到数据库文件中。

而启用事务后，操作流程会发生变化：
1.  修改的数据首先会存入事务日志。
2.  只有在事务被明确提交后，修改才会最终写入数据库文件。
3.  如果操作过程中出现问题，可以回滚整个事务，使数据恢复到事务开始前的状态。

![](img/829f8d9bda8a36932a439be41e119c7b_3.png)

这个过程非常类似于为虚拟机创建快照。你可以在执行一系列有风险的操作前“拍一个快照”（开启事务）。如果操作成功，就“保存快照”（提交事务）；如果操作出错，就“恢复到快照”（回滚事务）。这为数据操作提供了一个安全网。

事务控制主要针对**数据操作语言**中的`INSERT`、`UPDATE`、`DELETE`命令，因为它们会修改数据。`SELECT`查询命令不会修改数据，因此不受事务控制的核心影响。

## 事务的基本命令

事务的控制主要依赖于三个命令：
*   `BEGIN;` 或 `START TRANSACTION;`：用于显式地开启一个事务。
*   `COMMIT;`：提交事务，使所有修改永久生效。
*   `ROLLBACK;`：回滚事务，撤销当前事务内的所有修改。

![](img/829f8d9bda8a36932a439be41e119c7b_5.png)

![](img/829f8d9bda8a36932a439be41e119c7b_7.png)

默认情况下，MySQL是**自动提交**模式，即每一条SQL语句都被视为一个独立的事务并立即提交。要使用手动控制的事务，需要先关闭自动提交。

以下是关闭自动提交和手动控制事务的示例：
```sql
-- 首先，关闭自动提交
SET AUTOCOMMIT = 0;

![](img/829f8d9bda8a36932a439be41e119c7b_9.png)

![](img/829f8d9bda8a36932a439be41e119c7b_11.png)

![](img/829f8d9bda8a36932a439be41e119c7b_12.png)

-- 开启一个事务
BEGIN;

![](img/829f8d9bda8a36932a439be41e119c7b_14.png)

![](img/829f8d9bda8a36932a439be41e119c7b_16.png)

-- 执行数据操作，例如删除一条记录
DELETE FROM your_table WHERE id = 5;

-- 此时可以查看，记录似乎被删除了
SELECT * FROM your_table;

-- 如果发现删错了，可以回滚
ROLLBACK;

![](img/829f8d9bda8a36932a439be41e119c7b_18.png)

-- 再次查看，记录恢复
SELECT * FROM your_table;

-- 确认操作无误后，提交事务
-- BEGIN;
-- DELETE FROM your_table WHERE id = 5;
-- COMMIT;
```
**关键点**：在手动事务中，只有执行了`COMMIT`后，修改才会对其他数据库连接可见并持久化。执行`ROLLBACK`或连接中断时未提交，所有修改都会被丢弃。

## 事务与二进制日志

![](img/829f8d9bda8a36932a439be41e119c7b_20.png)

MySQL的二进制日志记录了所有对数据库进行更改的操作，用于数据复制和恢复。事务操作也会被记录在二进制日志中。

![](img/829f8d9bda8a36932a439be41e119c7b_22.png)

![](img/829f8d9bda8a36932a439be41e119c7b_24.png)

使用以下命令可以查看二进制日志内容：
```bash
mysqlbinlog /var/lib/mysql/binlog.000001
```
在日志中，一个完整的事务会以`BEGIN`开始，以`COMMIT`结束。如果事务被回滚，则相关的操作不会被记录到二进制日志中，因为数据没有发生实际变化。

![](img/829f8d9bda8a36932a439be41e119c7b_26.png)

**注意**：二进制日志功能默认可能未开启，需要在MySQL配置文件（如`/etc/my.cnf`）中添加以下配置来启用：
```ini
[mysqld]
log-bin=/var/lib/mysql/binlog  # 二进制日志文件路径和前缀
server-id=1                     # 服务器ID，主从复制时必需
```

## 查看事务状态

![](img/829f8d9bda8a36932a439be41e119c7b_33.png)

![](img/829f8d9bda8a36932a439be41e119c7b_35.png)

![](img/829f8d9bda8a36932a439be41e119c7b_37.png)

当一个事务正在进行时，可以从其他会话查看其状态。

在另一个MySQL客户端中，可以查询信息模式数据库来查看当前运行的事务：
```sql
-- 查看当前正在运行的事务
SELECT * FROM information_schema.INNODB_TRX\G
```
这条命令会显示事务的详细信息，如事务ID、状态、开始时间等。你会注意到，在第一个会话中未提交的修改，在第二个会话中是查询不到的，这体现了事务的**隔离性**。

## 什么是锁？

介绍完事务后，我们自然会遇到锁的问题。锁是一种机制，用于在多个事务同时访问数据库时，管理对数据的并发访问，防止数据不一致。

主要的锁类型有两种：
*   **共享锁**：也称为读锁。多个事务可以同时持有同一数据的共享锁，用于读取操作。`SELECT`语句通常隐式使用共享锁。它允许多个读操作并发进行。
*   **排他锁**：也称为写锁。一个事务对数据加上排他锁后，其他事务就不能再给该数据加任何锁（包括共享锁和排他锁）。`INSERT`、`UPDATE`、`DELETE`操作会隐式使用排他锁。

![](img/829f8d9bda8a36932a439be41e119c7b_39.png)

锁机制确保了数据的安全性。例如，当一个事务正在修改某行数据时（持有排他锁），其他事务的修改或读取操作可能会被阻塞，直到该事务提交或回滚释放锁。

## 总结

本节课中我们一起学习了MySQL的事务控制语言和锁的基本概念。
*   **事务** 将一组SQL操作绑定，提供“全部成功”或“全部失败”的保证，通过`BEGIN`、`COMMIT`、`ROLLBACK`命令控制，是数据安全性的重要机制。
*   使用事务前，通常需要`SET AUTOCOMMIT=0`关闭自动提交。
*   事务的操作被记录在**二进制日志**中，用于数据恢复和复制。
*   **锁** 是管理并发访问的机制，分为共享锁（读锁）和排他锁（写锁），确保在多个事务同时操作时数据的一致性。

![](img/829f8d9bda8a36932a439be41e119c7b_41.png)
