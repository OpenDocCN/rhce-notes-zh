# 华为认证ICT学院HCIA/HCIP-Datacom教程：第3册-第3章-2：IP ACL中的匹配顺序 🔍

![](img/c191b66ed6839fb85b3b9cc434ef39ca_1.png)

在本节课程中，我们将学习IP访问控制列表（ACL）中规则的匹配顺序。理解数据包如何与多条ACL规则进行匹配，以及匹配顺序对网络流量控制的影响，是正确配置ACL的关键。

## 概述 📋

IP ACL用于控制网络数据包的转发或丢弃。当ACL中包含多条规则时，数据包会按照特定的顺序与这些规则进行匹配。本节将详细讲解这个匹配顺序的逻辑、规则编号的机制，以及ACL在设备入方向和出方向上的应用流程。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_3.png)

## ACL规则的匹配顺序

![](img/c191b66ed6839fb85b3b9cc434ef39ca_5.png)

上一节我们介绍了ACL的基本概念，本节中我们来看看当存在多条规则时，设备如何处理。

数据包在通过配置了ACL的设备时，会按照ACL中规则的**编号从小到大**的顺序，逐一进行匹配。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_7.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_1.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_9.png)

ACL中的每条规则都有一个对应的编号。管理员在配置时可以手动指定编号（例如1， 2， 3）。如果未手动指定，系统会采用默认的自动编号机制：第一条规则编号为5，第二条为10，后续规则以步长5递增（15， 20， 25...）。因此，先配置的规则编号小，位置靠上。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_11.png)

设备收到数据包后，即按照此编号顺序进行匹配。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_13.png)

## 设计ACL顺序的注意事项

在规划和配置ACL条目的顺序时，管理员需要牢记以下几点：

以下是设计ACL顺序时需要重点考虑的几个原则：

1.  **顺序匹配**：设备严格按照规则编号**从小到大**（即配置中从上到下）的顺序匹配数据包。必须将需要优先处理的规则放在前面。
2.  **首次匹配即停止**：一旦数据包与某条规则匹配成功，设备将立即执行该规则定义的动作（**permit** 允许 或 **deny** 拒绝），并**停止**后续规则的匹配。因此，规则的排列顺序至关重要。
3.  **最终处理方式**：一个ACL对数据包的最终处理结果分为三种情况：
    *   **允许**：数据包匹配了某条 **`permit`** 规则。
    *   **拒绝**：数据包匹配了某条 **`deny`** 规则。
    *   **未命中**：数据包与ACL中所有显式配置的规则都不匹配。在华为设备上，**缺省情况下，未命中的数据包将被允许通过**。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_15.png)

## ACL的应用方向：入向与出向

![](img/c191b66ed6839fb85b3b9cc434ef39ca_17.png)

ACL可以应用在路由器的接口上，并分为两个方向：**入方向（inbound）** 和 **出方向（outbound）**。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_19.png)

路由器在转发数据包时，从一个接口接收，从另一个接口发送。因此，一个数据包可能会在入站接口和出站接口分别接受ACL的检查，类似于乘坐火车时进站和出站都可能需要安检。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_15.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_21.png)

方向是相对于数据流而言的。对于从左向右的数据流，左侧接口是入方向，右侧接口是出方向。反向数据流则相反。实际部署时，可根据安全需求选择在入方向、出方向或两者同时应用ACL。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_23.png)

### 入向IP ACL

当数据包进入路由器的接口时，会受到应用在该接口入方向的ACL的过滤。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_25.png)

处理结果有两种：
1.  **转发**：数据包被允许（匹配`permit`规则或未命中），交给路由转发进程进行后续处理（如查路由表）。
2.  **丢弃**：数据包被拒绝（匹配`deny`规则）。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_27.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_27.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_29.png)

以下是入向ACL过滤数据包的流程图：

![](img/c191b66ed6839fb85b3b9cc434ef39ca_29.png)
*流程描述*：接口收到数据包 → 检查是否应用入向ACL → 若无，直接转发 → 若有，则从第一条规则开始顺序匹配 → 若匹配`permit`则转发 → 若匹配`deny`则丢弃 → 若所有规则均未匹配（未命中），则**允许**转发。

### 出向IP ACL

![](img/c191b66ed6839fb85b3b9cc434ef39ca_31.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_33.png)

当路由器决定将数据包从某个接口转发出去时，在发出之前，会受到应用在该接口出方向的ACL的过滤。

处理结果同样有两种：
1.  **转发**：数据包被允许（匹配`permit`规则或未命中），从接口发送出去。
2.  **丢弃**：数据包被拒绝（匹配`deny`规则）。

![](img/c191b66ed6839fb85b3b9cc434ef39ca_35.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_37.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_37.png)

以下是出向ACL过滤数据包的流程图：

![](img/c191b66ed6839fb85b3b9cc434ef39ca_39.png)

![](img/c191b66ed6839fb85b3b9cc434ef39ca_39.png)
*流程描述*：数据包需从接口发出 → 检查是否应用出向ACL → 若无，直接转发 → 若有，则从第一条规则开始顺序匹配 → 若匹配`permit`则转发 → 若匹配`deny`则丢弃 → 若所有规则均未匹配（未命中），则**允许**转发。

## 总结 🎯

![](img/c191b66ed6839fb85b3b9cc434ef39ca_41.png)

本节课中我们一起学习了IP ACL的核心机制——匹配顺序。我们了解到ACL规则按照编号从小到大顺序匹配，且遵循“首次匹配即停止”的原则。同时，我们明确了ACL可以应用在接口的入方向和出方向，并分别分析了数据包在这两个方向上的过滤流程。记住**缺省未命中的流量是允许通过的**，这对于编写正确、高效的ACL至关重要。掌握这些原理，是进行精确流量控制和管理的基础。
