### SSH连接回顾
SSH连接的步骤包括密钥交换、获取服务器公钥，并通过密钥连接到服务器。

以下是两种登录方式：
*   **基于用户密码登录**：在交换并得到服务器公钥后，使用密码进行验证。
*   **基于客户端密钥对登录**：首先需要在客户端创建密钥对。

![](img/4b76ebc154d132489b3caf3c73503e89_8.png)

![](img/4b76ebc154d132489b3caf3c73503e89_10.png)

创建客户端密钥对的命令是 `ssh-keygen`。可以使用 `-b` 选项指定密钥大小，`-t` 选项指定密钥类型。默认情况下，密钥对创建在用户家目录下的 `.ssh` 目录中，文件名为 `id_rsa`（私钥）和 `id_rsa.pub`（公钥）。

![](img/4b76ebc154d132489b3caf3c73503e89_12.png)

![](img/4b76ebc154d132489b3caf3c73503e89_14.png)

![](img/4b76ebc154d132489b3caf3c73503e89_15.png)

如果指定了非默认的密钥对存放路径，在连接时需要使用 `-i` 选项来指定私钥文件。例如：
```bash
ssh -i /path/to/private_key user@host
```
发送客户端公钥到服务器的命令是 `ssh-copy-id`。如果使用了自定义的密钥文件，同样需要使用 `-i` 选项指定公钥文件。例如：
```bash
ssh-copy-id -i /path/to/public_key user@host
```

### 时间同步服务
系统时间主要通过 `chronyd` 服务进行同步。这是一个C/S架构的服务，但客户端与服务端使用同一个配置文件 `/etc/chrony.conf`。

作为客户端，需要在配置文件中添加NTP服务器地址。更改配置后，需要重启 `chronyd` 服务，并使用 `chronyc sources` 命令查看同步状态。

作为服务端，可以在配置文件中使用 `allow` 指令添加允许同步的远程IP或网段，或使用 `deny` 指令拒绝。

### 时间设置工具
系统提供了 `timedatectl` 工具来管理时间和时区。

以下是常用命令：
*   `timedatectl`：打印当前时间、时区等信息。
*   `timedatectl list-timezones`：列出所有可用时区。
*   `timedatectl set-timezone <时区>`：修改系统时区。
*   `timedatectl set-time “YYYY-MM-DD HH:MM:SS”`：修改系统时间（格式：年月日 时分秒）。
*   `timedatectl set-ntp true/false`：启用或禁用NTP时间同步服务。

公共NTP服务器的时间源最终来自于原子钟，例如铯原子钟，它提供了高精度的时间基准。

## 日志管理基础
日志是记录系统中事件（时间、地点、人物、事件）的文件，对于运维和排错至关重要。

### 日志服务简介
当前系统的日志主要由两个服务协同工作：
1.  **systemd-journald**：这是一个系统服务，它以二进制格式将系统从开机到当前的所有事件记录在内存中。这些日志在系统重启后会被清空。
2.  **rsyslog**：这是一个更传统的日志服务。它的主要工作之一，就是读取 `systemd-journald` 生成的二进制日志，并将其转换为文本格式，持久化保存到 `/var/log` 目录下的各个日志文件中。

许多系统服务（如SSH、Cron）的日志会通过 `rsyslog` 收集。一些第三方应用也可能集成到 `rsyslog`，或者拥有自己独立的日志文件。

由 `rsyslog` 自动生成和管理的常见系统日志文件包括：
*   `/var/log/messages`：通用系统消息日志。
*   `/var/log/secure`：安全和身份验证相关日志。
*   `/var/log/maillog`：邮件服务日志。
*   `/var/log/cron`：计划任务日志。
*   `/var/log/boot.log`：系统启动日志。

![](img/4b76ebc154d132489b3caf3c73503e89_24.png)

### 认识rsyslog服务
`rsyslog` 是从早期 `syslog` 发展而来的高性能日志服务，具有模块化设计。它能够从各种来源收集日志，进行处理（过滤、转换），并输出到不同的目的地（如本地文件、远程服务器、数据库）。

在企业环境中，通常会搭建集中的日志收集系统，例如 **ELK Stack** (Elasticsearch, Logstash, Kibana)，而 `rsyslog` 常作为日志收集和传输的代理。

![](img/4b76ebc154d132489b3caf3c73503e89_28.png)

### 核心概念：设施与优先级
`rsyslog` 通过两个关键概念来分类日志：**设施 (Facility)** 和 **优先级 (Priority)**。

![](img/4b76ebc154d132489b3caf3c73503e89_30.png)

**设施** 定义了日志消息的来源或类型。
以下是常见的设施：
*   `auth`, `authpriv`：身份验证和安全相关消息。
*   `cron`：计划任务相关消息。
*   `kern`：内核消息。
*   `mail`：邮件系统消息。
*   `user`：用户进程消息。
*   `local0` 到 `local7`：保留给本地自定义使用的设施。

![](img/4b76ebc154d132489b3caf3c73503e89_32.png)

**优先级** 定义了日志消息的严重程度，从低到高分为8级：
*   `debug`：调试信息。
*   `info`：一般性信息。
*   `notice`：正常但值得注意的信息。
*   `warning` 或 `warn`：警告信息，可能有问题但不影响运行。
*   `err` 或 `error`：错误信息，功能受损。
*   `crit`：严重情况。
*   `alert`：需要立即采取行动。
*   `emerg` 或 `panic`：系统不可用，最高级别。

![](img/4b76ebc154d132489b3caf3c73503e89_34.png)

![](img/4b76ebc154d132489b3caf3c73503e89_36.png)

通常，`debug` 到 `notice` 级别的日志用于记录常规操作，`warning` 及以上级别的日志需要关注，`error` 及以上级别的日志通常意味着需要干预的问题。

![](img/4b76ebc154d132489b3caf3c73503e89_38.png)

![](img/4b76ebc154d132489b3caf3c73503e89_40.png)

### 理解rsyslog配置文件
`rsyslog` 的主配置文件是 `/etc/rsyslog.conf`。它主要由三部分组成：
1.  **Modules (`$ModLoad`)**：加载所需的模块。
2.  **Global Directives**：全局配置指令。
3.  **Rules**：规则部分，这是配置日志如何路由的核心。

规则部分的语法格式为：
```
设施.优先级        动作
```
*   `设施` 和 `优先级` 使用点号 `.` 连接。
*   可以使用 `*` 通配符代表“所有”。例如 `*.*` 代表所有设施的所有优先级。
*   多个设施可以用分号 `;` 隔开。
*   优先级指定了该级别及**更高级别**的消息。例如 `auth.err` 会捕获 `auth` 设施的 `err`, `crit`, `alert`, `emerg` 级别消息。
*   `none` 作为一个优先级，表示不记录任何信息。
*   `动作` 定义了日志的去向，最常见的是保存到文件（如 `/var/log/messages`）。

![](img/4b76ebc154d132489b3caf3c73503e89_42.png)

![](img/4b76ebc154d132489b3caf3c73503e89_44.png)

![](img/4b76ebc154d132489b3caf3c73503e89_46.png)

![](img/4b76ebc154d132489b3caf3c73503e89_47.png)

在规则行中，文件名前的 `-` 符号表示异步写入，可以提高性能，但可能在系统崩溃时丢失少量最新日志。

![](img/4b76ebc154d132489b3caf3c73503e89_49.png)

![](img/4b76ebc154d132489b3caf3c73503e89_51.png)

![](img/4b76ebc154d132489b3caf3c73503e89_53.png)

一个配置示例如下，它将所有 `authpriv` 设施的消息（所有级别）记录到 `/var/log/secure` 文件：
```
authpriv.*                /var/log/secure
```

![](img/4b76ebc154d132489b3caf3c73503e89_54.png)

### 配置自定义日志
我们可以修改 `/etc/rsyslog.conf` 文件来定义自己的日志规则。

例如，如果我们希望将 SSH 的认证日志（默认使用 `authpriv` 设施）单独存放，可以添加规则。首先，需要确认 SSH 服务端配置 `/etc/ssh/sshd_config` 中 `SyslogFacility` 的设置（默认是 `AUTHPRIV`）。

![](img/4b76ebc154d132489b3caf3c73503e89_56.png)

![](img/4b76ebc154d132489b3caf3c73503e89_58.png)

![](img/4b76ebc154d132489b3caf3c73503e89_60.png)

![](img/4b76ebc154d132489b3caf3c73503e89_62.png)

然后，在 `rsyslog.conf` 的规则部分添加：
```
# 将 authpriv 设施的所有消息记录到自定义文件
authpriv.*                /var/log/my_ssh.log
```
修改配置后，需要重启 `rsyslog` 服务使配置生效：
```bash
systemctl restart rsyslog
```
之后，SSH 的登录、退出等日志就会记录到 `/var/log/my_ssh.log` 文件中。

![](img/4b76ebc154d132489b3caf3c73503e89_66.png)

![](img/4b76ebc154d132489b3caf3c73503e89_68.png)

### 使用自定义设施
`local0` 到 `local7` 这8个自定义设施，非常适合用于将特定应用程序的日志归类。

![](img/4b76ebc154d132489b3caf3c73503e89_70.png)

例如，我们可以配置一个应用程序（如 Nginx）将其日志发送到 `local6` 设施。首先，在应用程序配置中指定设施（具体方法因程序而异）。

![](img/4b76ebc154d132489b3caf3c73503e89_72.png)

![](img/4b76ebc154d132489b3caf3c73503e89_74.png)

然后，在 `rsyslog.conf` 中添加规则，将 `local6` 的日志记录到指定文件：
```
local6.*                /var/log/myapp.log
```
这样，所有使用 `local6` 设施的应用日志都会集中到 `myapp.log` 文件中，便于管理。

![](img/4b76ebc154d132489b3caf3c73503e89_76.png)

![](img/4b76ebc154d132489b3caf3c73503e89_78.png)

![](img/4b76ebc154d132489b3caf3c73503e89_80.png)

### 手动发送日志：logger命令
`logger` 命令是一个 shell 接口，可以手动向系统日志发送消息。这在脚本调试或测试日志配置时非常有用。

![](img/4b76ebc154d132489b3caf3c73503e89_82.png)

![](img/4b76ebc154d132489b3caf3c73503e89_84.png)

![](img/4b76ebc154d132489b3caf3c73503e89_86.png)

命令基本用法：
```bash
logger “This is a test log message”
```
默认情况下，消息会以 `user.notice` 的设施和优先级记录。

![](img/4b76ebc154d132489b3caf3c73503e89_88.png)

可以使用 `-p` 选项指定设施和优先级：
```bash
logger -p local0.err “This is an error message from my app”
```
一个重要的测试是发送 `emerg` 级别消息，根据配置，它可能会广播到所有已登录用户的终端：
```bash
logger -p emerg “EMERGENCY: System test alert!”
```
